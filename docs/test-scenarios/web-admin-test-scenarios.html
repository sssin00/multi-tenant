Web Admin Test Scenarios

web-admin 테스트 시나리오

관리자 콘솔의 로그인, 세션 초기화, 권한 기반 메뉴, 대시보드, 고객사 관리, 사용자 추가, 감사 로그, 오류 처리를 시나리오 기반으로 검증합니다. 1차 검증은 자동 E2E 코드가 아니라 화면 상태, API 응답, requestId, 감사 로그 evidence를 남기는 방식으로 수행합니다.

공통 실행 기준

대상 앱apps/web-admin, Docker 로컬과 Vite 단독 개발 서버 기준 URL http://localhost:5173/
API 진입점gateway http://localhost:3000 경유 /api/auth/**, /api/admin/**
Seed 관리자슈퍼 관리자 super@admin.local과 테넌트 관리자 admin@demo.local은 web-admin 로그인 화면에서 tenant ID 없이 email/password만 입력합니다. 서버가 tenant 범위를 판별하며, 비밀번호는 로컬 seed 기준값을 사용하고 evidence에는 남기지 않습니다.
공통 HeadersAuthorization, optional X-Tenant-Id, X-Request-Id, 변경성 요청은 Idempotency-Key. 로그인 요청은 X-Tenant-Id를 보내지 않고, 로그인 이후 tenant 관리자의 요청은 JWT에서 resolved tenantId가 전파됩니다.
성공 Evidence화면 캡처 또는 화면 상태 설명, API response JSON 요약, requestId, 변경 후 재조회 결과, 필요 시 audit log row
실패 Evidence실제 결과, 기대 결과, 재현 단계, error.code, HTTP status, requestId

검증 묶음

Auth Shell로그인, 세션 초기화, 메뉴 권한 Dashboard관리 요약과 API evidence Visual/UX대시보드 스타일 통일 검증 Tenant Admin생성, 상태, 모듈, 도메인 User Management사용자 목록, 필터, 추가 Audit/Risk감사 로그와 위험 작업 안내 Negative인증 실패, 권한 실패, tenant mismatch

실행 순서

순서단계완료 기준
1로컬 서비스 준비gateway, admin-bff, auth-iam, tenant, audit-log가 ready입니다.
2관리자 로그인슈퍼 관리자와 테넌트 관리자 모두 web-admin에서는 email/password만으로 access token과 refresh token이 발급되고 web-admin 세션이 생성됩니다.
3App Shell 검증GET /api/admin/me 응답 기반으로 한글 메뉴 6개가 표시됩니다.
4조회 화면 검증대시보드, 고객사 관리(시스템 관리자 전용), 사용자 관리, 역할/권한, 감사 로그 화면이 API 또는 샘플 fallback 상태를 명확히 표시합니다.
5변경성 action 검증테넌트 생성/변경, 사용자 추가, 역할 permission 교체가 requestId와 재조회 evidence를 남깁니다.
6오류/권한 검증인증 실패, tenant mismatch, 필수 헤더 누락이 표준 error envelope로 실패합니다.
