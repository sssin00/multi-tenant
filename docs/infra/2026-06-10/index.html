Infra · CDK Review

CDK 인프라 코드 분석

infra/cdk의 AWS CDK 코드를 파일별·기능별로 정리한 문서입니다. 현재 스택은 gateway-service와 auth-iam-service를 하나의 CloudFormation 스택으로 함께 배포하며, 두 서비스를 ECS Cloud Map 내부 DNS로 연결합니다.

이 문서는 CDK 코드 기준의 아키텍처 현황입니다. 현재 AWS 계정에 리소스가 실행 중이라는 의미는 아니며, 테스트 배포 리소스는 비용 절감을 위해 삭제되었을 수 있습니다.

검토 기준일: 2026-06-10

전체 구조

현재 CDK는 단일 스택만 정의하며, 스택 이름은 {project}-{env}-gateway-service-stack입니다.

루트에서 pnpm infra:synth, infra:diff, infra:deploy로 호출합니다.

파일별 설명

bin/app.ts — CDK 앱 진입점

CDK App을 만들고 GatewayServiceStack 하나를 인스턴스화합니다. cdk.json 또는 CLI -c key=value로 context를 주입합니다.

gateway context

gatewayImageTag, gatewayDesiredCount, gatewayUseNatGateway, gatewayRedisNodeType, ACM/도메인/CORS/JWT secret

auth context

authImageTag, authDesiredCount, authCorsAllowedOrigins, authJwtAlgorithm, DB/JWT/internal secret ARN

기본값

project=multi-tenant, envName=dev, defaultRegion=ap-northeast-2, desired count는 모두 0

lib/gateway-service-stack.ts — 핵심 인프라 스택

gateway-service와 auth-iam-service 배포에 필요한 AWS 리소스를 한 스택에 모두 정의합니다. 네이밍은 {project}-{env} 접두사를 사용합니다.

기능별 리소스

gateway-service-stack.ts가 생성하는 AWS 리소스를 기능 단위로 정리합니다.

기능gateway-serviceauth-iam-service
ECR repositoryOO
ECS Fargate task/serviceOO
ALB 노출O (public)X (내부 전용)
Cloud Map 등록-O
ElastiCache Redis 접근rate limitcache/session
Secrets ManagerJWT_SECRETDB URL, JWT, internal auth
CloudWatch LogsOO
RDS/PostgreSQL-CDK 미포함 (외부 secret 참조)
네트워크 (VPC)

2 AZ. gatewayUseNatGateway=true이면 private subnet + NAT Gateway 1개, false이면 public subnet만 사용합니다.

ECS 클러스터

Fargate 클러스터에 Container Insights V2와 Cloud Map private DNS namespace {env}.multi-tenant.local을 구성합니다.

Security Group (4개)

ALB SG, gateway SG, auth SG, redis SG. auth는 gateway SG에서 오는 3000만 허용하고 ALB에 직접 노출하지 않습니다.

ElastiCache Redis

1노드(기본 cache.t4g.micro). gateway rate limit과 auth cache/session에서 공유합니다.

ALB

인터넷 facing. ACM ARN이 있으면 HTTPS 443 listener를 만들고 HTTP 80은 HTTPS로 리다이렉트합니다. gateway만 target group에 연결합니다.

Cloud Map 내부 연결

gateway는 http://auth-iam-service.{env}.multi-tenant.local:3000/api/auth으로 auth를 호출합니다.

주요 CfnOutput: GatewayServiceRepositoryUri, GatewayLoadBalancerDns, GatewayRedisEndpoint, AuthIamServiceRepositoryUri, AuthIamServiceInternalUrl 등.

현재 배포 상태 해석

코드 기준

CDK synth 기준으로 gateway-service와 auth-iam-service 배포 리소스가 모두 정의되어 있습니다.

AWS 실행 상태

이 문서만으로 실제 AWS 리소스 실행 여부를 판단하지 않습니다. 콘솔 또는 AWS CLI에서 CloudFormation stack, ECS service, ALB, ECR을 확인해야 합니다.

비용 상태

테스트 리소스는 비용 절감을 위해 삭제하거나 desired count를 0으로 낮출 수 있습니다. NAT Gateway, ALB, ElastiCache는 task가 0이어도 비용이 발생할 수 있습니다.

Auth/IAM 실행 전 필수 조건

authDesiredCount=1로 올리기 전에 이미지와 runtime secret이 준비되어 있어야 합니다.

항목CDK context용도
Auth/IAM Docker imageauthImageTagECR에 push된 auth-iam-service image tag
PostgreSQL URL secretauthDatabaseUrlSecretArnECS task의 DATABASE_URL secret
JWT secretauthJwtSecretArnHS256 access token 서명 secret
JWT key pairauthJwtPrivateKeySecretArn, authJwtPublicKeySecretArnRS256 사용 시 private/public key secret
Internal auth secretauthInternalAuthSecretArngateway, BFF, domain service의 내부 호출 인증

secret 없이 task를 실행하면 컨테이너가 시작되어도 /ready의 database, jwt, security check가 실패할 수 있습니다.

Redis 공유 기준

현재 Redis는 gateway-service와 auth-iam-service가 하나의 ElastiCache Redis를 공유합니다.

공유 이유

초기 dev/staging 단계에서 비용과 CDK 복잡도를 낮추기 위한 선택입니다. 서비스별 Redis를 만들면 ElastiCache 노드, subnet group, security group, 설정 값이 함께 늘어납니다.

논리 분리 기준

서비스별 key prefix를 사용해 gateway rate limit key와 auth cache/session key가 섞이지 않도록 관리해야 합니다.

운영 전 검토

prod 전에는 gateway rate-limit Redis와 auth Redis를 분리할지 결정해야 합니다. 장애 격리와 부하 격리가 필요하면 서비스별 또는 용도별 Redis 분리가 적합합니다.

이전 대비 변경점

직전 버전은 gateway-service 단독 스택이었고, 현재는 auth-iam-service가 같은 스택에 추가되었습니다.

영역이전현재

아직 반영되지 않은 부분

GitHub Actions

deploy-gateway-service.yml은 gateway image만 build/push합니다. auth secret과 task definition context는 CDK에 전달할 수 있지만, auth image build/push와 authDesiredCount=1 실행은 별도 Auth/IAM 배포 workflow가 필요합니다.

RDS / Aurora

authDatabaseUrlSecretArn으로 기존 DB secret만 참조합니다. DB 인스턴스와 SG는 CDK에 없습니다.

기타 리소스

Route53, EventBridge, SQS, S3는 아직 CDK로 정의되지 않았습니다.

배포 시 context 예시

dev에서 비용을 줄여 확인할 때는 NAT Gateway를 끄고, 두 서비스 desired count를 0으로 둔 상태에서 ECR과 기본 인프라를 먼저 만듭니다.

# 1단계: 인프라만 생성 (태스크 0, dev 비용 절감용 public subnet 배치)
pnpm --filter @multi-tenant/infra-cdk exec cdk deploy \
  -c gatewayDesiredCount=0 \
  -c authDesiredCount=0 \
  -c gatewayUseNatGateway=false

# 2단계: 이미지 push 후 실행
pnpm --filter @multi-tenant/infra-cdk deploy \
  -c gatewayDesiredCount=1 \
  -c authDesiredCount=1 \
  -c gatewayImageTag=latest \
  -c authImageTag=latest \
  -c gatewayJwtSecretArn=arn:aws:secretsmanager:... \
  -c authDatabaseUrlSecretArn=arn:aws:secretsmanager:... \
  -c authJwtSecretArn=arn:aws:secretsmanager:... \
  -c authInternalAuthSecretArn=arn:aws:secretsmanager:...

gatewayUseNatGateway=true는 private subnet + NAT Gateway 구성이며 dev에서도 NAT Gateway 비용이 발생합니다.
