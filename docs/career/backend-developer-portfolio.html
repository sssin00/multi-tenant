백엔드 아키텍처 흐름

외부 업무 API는 gateway-service 하나로 들어오고, 화면 조합은 BFF, 업무 규칙은 domain service가 담당합니다.

ARCHITECTURE
Client

관리자/WMS 화면은 gateway API만 호출합니다.

Gateway

JWT 검증, tenant context, rate limit, route proxy를 처리합니다.

BFF

admin/user API 조합, tenant readiness, 권한 summary 조회를 담당합니다.

Domain Services

auth, tenant, wms, audit가 각자 소유한 business rule을 처리합니다.

Outbox/Audit

업무 이벤트와 감사 로그를 비동기적으로 전달하고 저장합니다.

아키텍처 설계서

초기 스마트 팩토리 구성도를 MSA와 AWS ECS 운영 구조에 맞게 재설계한 백엔드 설계 요약입니다.

BLUEPRINT
외부 진입점

외부 업무 API는 gateway-service를 단일 진입점으로 사용합니다. 인증, requestId, tenant context, rate limit, route proxy를 gateway에서 처리해 외부 노출 면을 줄였습니다.

화면 조합 계층

Admin/User BFF는 화면에 필요한 여러 내부 API를 조합합니다. tenant status check, permission summary, 화면별 응답 변환은 BFF 계층의 책임입니다.

도메인 서비스 계층

auth-iam, tenant, wms, audit-log는 각자 소유한 업무 규칙과 데이터 접근을 담당합니다. gateway는 domain service를 직접 호출하지 않도록 설계했습니다.

데이터와 테넌트 격리

초기 전략은 shared DB + tenantId입니다. 모든 tenant-scoped 요청, 데이터 접근, 로그, 이벤트에는 tenantId와 requestId가 포함됩니다.

이벤트와 감사 추적

업무 상태 변경과 outbox event 저장을 같은 transaction에 묶고, outbox-relay-service가 SQS/EventBridge/mock publisher로 발행합니다.

AWS 실행 환경

서비스는 ECS Fargate task로 실행하고, CDK로 ECR, ALB, VPC, Cloud Map, Redis, SQS/DLQ, CloudWatch, Security Group을 관리합니다.

DDD 관점의 설계

서비스를 단순 CRUD 단위가 아니라 업무 언어와 책임 경계 기준으로 나누기 위해 DDD 관점을 적용했습니다.

DDD
Bounded Context 분리

Auth/IAM, Tenant, WMS, Audit Log, Outbox Relay를 서로 다른 bounded context로 보고, 각 context가 소유한 모델과 규칙을 다른 서비스가 직접 침범하지 않도록 나눴습니다.

Ubiquitous Language

WMS에서는 warehouse, location, item, inventory, inbound, outbound, packing, shipment 같은 업무 용어를 API, 이벤트, 테스트 시나리오 이름에 일관되게 사용했습니다.

Aggregate 중심 규칙

재고 조정, 입고 확정, 출고 할당, 포장, 출하는 단순 row 변경이 아니라 상태 전이와 정합성 규칙을 가진 업무 행위로 보고 wms-service 내부에서 처리했습니다.

Domain Event

업무에서 이미 발생한 사실을 wms.inventory.adjusted, wms.inbound.confirmed, wms.outbound.shipped 같은 과거형 이벤트로 표현하고 outbox를 통해 발행했습니다.

Application Layer

BFF와 controller는 요청 조합, 인증 컨텍스트, 응답 변환에 집중하고, 중요한 업무 판단은 domain service 계층에 남겨 화면 요구사항이 도메인 규칙을 오염시키지 않도록 했습니다.

Infrastructure 분리

Prisma, Redis, SQS, EventBridge, CDK 같은 기술 세부사항은 repository, client, relay, infra 계층에 배치해 도메인 규칙과 운영 구현을 분리했습니다.

Core Domain 초기 핵심 도메인은 WMS입니다. 재고, 입고, 출고, 포장, 출하의 상태 전이와 정합성 규칙을 가장 중요한 업무 모델로 두었습니다.
Supporting Domain Tenant, Auth/IAM, Audit Log는 WMS 업무를 안전하게 운영하기 위한 지원 도메인입니다. 각 서비스는 독립 모델을 가지며 WMS 내부 데이터에 직접 접근하지 않습니다.
Context Mapping Gateway와 BFF는 upstream/downstream 사이의 anti-corruption layer처럼 동작합니다. 외부 화면 요구사항을 domain service 모델에 직접 밀어 넣지 않고, BFF에서 화면 응답으로 변환합니다.
Consistency Boundary 한 업무 transaction 안에서 반드시 함께 성공해야 하는 변경은 같은 service와 DB transaction 안에서 처리합니다. 다른 context로 전달되는 변화는 outbox event로 비동기 전파합니다.
확장 방향 MES, QMS, FMS, EMS가 추가될 때도 각 도메인을 별도 bounded context로 두고, 공통 모델 공유보다 API/event 계약을 통해 연계하는 방향을 유지합니다.

서비스별 백엔드 구현

각 서비스가 담당하는 백엔드 기능과 설명 포인트입니다.

SERVICES
gateway-service 외부 업무 API 단일 진입점입니다. JWT 서명/만료 검증, tenant mismatch 차단, requestId/tenantId propagation, rate limit, auth/admin/app route proxy를 담당합니다.
auth-iam-service 로그인, refresh token rotation, 사용자/role/permission 관리, permission summary/check API, system_admin tenantless login, tenant 관리자 scope 판별을 담당합니다.
tenant-service tenant, module, domain, status/readiness를 관리합니다. BFF가 tenant 상태를 확인할 수 있도록 내부 API를 제공합니다.
admin-bff-service 관리자 화면용 tenant/user/rbac/audit API를 조합합니다. Gateway 뒤에서 내부 서비스 호출을 중재하고 audit best-effort 기록을 수행합니다.
user-bff-service 사용자 app shell/navigation과 WMS 화면 조회 API를 조합합니다. WMS domain API를 화면에 맞게 재구성하는 역할입니다.
wms-service WMS business rule의 소유 서비스입니다. 창고/로케이션/품목, 재고, 스냅샷, 입고, 출고, 포장, 출하 API와 outbox/audit 이벤트 생성을 담당합니다.
outbox-relay-service 각 source DB의 outbox_events를 polling하고 EventBridge/SQS/mock publisher로 발행합니다. retry, failed 상태, audit queue 전달을 처리합니다.
audit-log-service SQS audit event를 소비하고 append-only 감사 로그로 저장합니다. 관리자 조회 API를 통해 tenant/user/resource/action 기준 검색을 제공합니다.

기술 선택 이유

멀티테넌트 MSA와 AWS 배포 가능성을 기준으로 선택한 기술입니다.

WHY STACK
NestJS module, controller, service, guard, interceptor 구조가 명확해 MSA 서비스별 책임을 나누기 쉽고, 인증/인가/검증/예외 처리 같은 공통 관심사를 일관되게 적용하기 좋습니다.
TypeScript 서비스 간 API DTO, config, event payload, error envelope를 타입으로 관리해 변경 영향을 빨리 발견할 수 있습니다.
Prisma 초기 개발 단계에서 schema와 query를 빠르게 반복하기 좋고, tenantId 조건 누락 같은 데이터 접근 규칙을 repository/service 계층에서 명확히 드러낼 수 있습니다.
PostgreSQL WMS의 재고 원장, 입출고 상태, 감사 로그처럼 정합성이 중요한 업무 데이터를 다루기 적합합니다.
Outbox Pattern 도메인 DB 변경과 이벤트 저장을 같은 transaction에 묶어 이벤트 유실 가능성을 낮춥니다.
SQS / DLQ 감사 로그와 이벤트 소비를 비동기로 처리하고, 실패 메시지를 DLQ로 분리해 운영자가 재처리할 수 있게 만듭니다.
ECS Fargate 컨테이너 기반 MSA를 운영하면서 EC2 관리 부담을 줄이고, 서비스별 task와 desired count를 독립적으로 조정할 수 있습니다.
AWS CDK 애플리케이션과 같은 TypeScript로 인프라를 코드화해 서비스 추가 시 ECR/ECS/ALB/SQS/CloudWatch 구성을 반복 가능한 패턴으로 관리할 수 있습니다.

테스트 기준

1차 단계에서는 자동 E2E보다 시나리오 기반 AI 검증과 evidence 기록을 우선했습니다.

TEST CRITERIA
API 계약

모든 응답은 success, requestId, timestamp, data/error envelope를 따라야 하며, list 응답은 items, page, size, total만 사용합니다.

멀티테넌트 격리

missing tenant, tenant mismatch, inactive tenant, cross-tenant access, tenantId propagation 실패를 검증 대상으로 둡니다.

인증/인가

login/refresh 예외를 제외한 business API는 인증을 요구하고, 권한 확인은 auth-iam-service의 permission check 기준을 따릅니다.

WMS 업무 정합성

재고 조정, 입고 확정, 출고 할당, 포장, 출하에서 재고 부족, 중복 처리, 상태 전이, idempotency 위험을 확인합니다.

Outbox/Audit

업무 변경 후 outbox row 생성, relay publish, retry/failure 상태, audit-log 저장, requestId/tenantId 보존을 evidence로 확인합니다.

성능 기준

p95/p99 latency, error rate, outbox/audit delay, resource usage, data consistency를 확인 대상으로 둡니다.
