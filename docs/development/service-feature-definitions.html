서비스별 기능 정의서

이 문서는 현재 저장소의 서비스와 앱을 기준으로 각 서비스가 제공해야 하는 기능, 호출자, 데이터 책임, 이벤트와 후속 범위를 정리합니다. 구현 상세나 일자별 기록은 서비스 개발과정 문서에 두고, 이 문서는 제품 기능과 서비스 경계를 빠르게 확인하는 기준 문서로 사용합니다.

gateway-service

외부 요청의 공통 진입점입니다. 인증 토큰 확인, tenant context 검증, rate limit, BFF upstream 라우팅을 담당합니다.

1차 구현
주요 호출자web-admin, web-wms, 외부 API client
핵심 책임JWT 검증, X-Tenant-Id와 JWT tenant 일치 확인, requestId 생성/전파, rate limit, upstream proxy
라우팅 범위/api/auth/**는 auth-iam-service, /api/admin/**는 admin-bff-service, /api/app/**는 user-bff-service로 전달합니다.
제외 책임도메인 비즈니스 로직, tenant-service 직접 상태 조회, domain/platform service 직접 라우팅은 맡지 않습니다.
공통 기능GET /health, GET /ready, CORS/security headers, JSON access log, Redis 기반 route별 rate limit
후속 범위운영 환경별 rate limit tuning, gateway route 확장 시 BFF 선행 계약 검증, production CORS origin 확정

상세 기능

gateway-service는 사용자-facing 업무 의미를 해석하지 않고, 요청을 안전하게 분류하고 검증한 뒤 올바른 BFF 또는 인증 서비스로 전달합니다.

기능입력/사용자 행위시스템 처리결과/예외
Health 확인운영자나 로드밸런서가 GET /health를 호출합니다.프로세스 생존 여부만 빠르게 확인합니다.정상 기동 중이면 표준 성공 envelope 또는 health 응답을 반환합니다.
Readiness 확인배포, compose, ECS health check가 GET /ready를 호출합니다.필수 config, upstream URL, Redis rate limit 의존성 등 gateway 준비 상태를 확인합니다.준비되지 않은 항목이 있으면 not_ready 응답으로 배포/트래픽 투입을 막습니다.
인증 API proxy사용자가 로그인, 토큰 갱신, 로그아웃, 내 정보 조회 요청을 보냅니다./api/auth/** 요청을 auth-iam-service upstream으로 전달하고 requestId를 유지합니다.auth-iam-service의 표준 응답을 gateway 응답으로 반환합니다.
관리자 API proxy관리자가 tenant, 사용자, role, audit 조회 요청을 보냅니다.JWT를 검증하고 tenant header 불일치를 차단한 뒤 /api/admin/**를 admin-bff-service로 전달합니다.인증 실패는 401, tenant mismatch는 403 TENANT_MISMATCH, upstream 장애는 표준 gateway 오류로 반환합니다.
사용자 앱 API proxyWMS 사용자가 app shell, navigation, WMS 화면 데이터를 요청합니다./api/app/**를 user-bff-service로 전달하고 사용자/tenant/request context를 보존합니다.User BFF의 업무 응답을 그대로 envelope 규칙에 맞춰 전달합니다.
Rate limit특정 route에 과도한 요청이 들어옵니다.Redis 기반 key로 route별 요청 수를 계산하고 정책치를 초과하면 차단합니다.429 RATE_LIMIT_EXCEEDED로 반환하며 도메인 서비스에는 요청을 전달하지 않습니다.
보안 header와 CORS브라우저가 preflight 또는 실제 API 요청을 보냅니다.허용 origin/method/header를 확인하고 보안 header를 추가합니다.허용되지 않은 origin은 브라우저 접근을 차단하고, production wildcard CORS는 사용하지 않습니다.
개발과정 API 문서

admin-bff-service

관리자 화면의 외부 API surface입니다. 여러 내부 서비스를 조합하되, 도메인 상태 변경은 각 owning service로 위임합니다.

1차 구현
주요 호출자web-admin, gateway-service
핵심 책임관리자 권한 확인, tenant readiness 확인, tenant/user/role/permission/audit 화면 데이터 조합, downstream request context 전파
Tenant 관리tenant 목록, 생성, 상세, 수정, 상태 변경, module 교체, domain 목록/추가/비활성화 API를 tenant-service 내부 API로 위임합니다.
사용자/RBAC 관리tenant 일반 사용자 CRUD, 사용자 상태 변경, role 목록/생성/수정, role permission 교체, user-role 부여/해제를 auth-iam-service로 위임합니다.
화면 조합GET /api/admin/access-control/screen-data에서 사용자, role, permission 목록을 조합해 access-control 화면 초기 데이터를 제공합니다.
Audit 조회GET /api/admin/audit-logs에서 audit-log-service 조회 API를 호출해 관리자 evidence 조회 화면을 지원합니다.
후속 범위GET /api/admin/me, audit-log-service 장애 처리 UX 정책, 운영 위험 작업 확인 흐름

상세 기능

Admin BFF는 관리자 화면의 업무 흐름을 API 단위로 안정화하고, 실제 데이터 변경은 tenant-service와 auth-iam-service 같은 owning service로 위임합니다.

기능사용자 행위시스템 처리권한/결과
Tenant 목록 조회관리자가 tenant 목록 화면에 진입하거나 검색 조건을 입력합니다.tenant-service의 internal admin 목록 API를 호출하고 page/size/상태 조건을 전달합니다.tenant.tenants.read 권한이 필요하며 list envelope로 반환합니다.
Tenant 생성관리자가 tenant 이름, domain, 초기 module 정보를 입력하고 생성을 실행합니다.idempotency key를 확인한 뒤 tenant-service 생성 API로 위임합니다. tenant code 생성과 outbox 저장은 tenant-service가 처리합니다.tenant.tenants.create 권한이 필요합니다. 중복 domain은 TENANT_DOMAIN_CONFLICT로 반환합니다.
Tenant 상태 변경관리자가 tenant를 active/inactive/suspended 등 운영 상태로 변경합니다.관리 권한과 idempotency key를 확인하고 tenant-service 상태 변경 API로 위임합니다.tenant.tenants.updateStatus 권한이 필요하며 변경 후 tenant object를 반환합니다.
Tenant module 관리관리자가 tenant에 사용할 module 목록을 수정합니다.요청한 module set을 tenant-service로 전달하고 enable/disable 차이를 처리하게 합니다.tenant.modules.manage 권한이 필요합니다. 변경 결과와 enabledModules를 반환합니다.
Tenant domain 관리관리자가 tenant domain을 추가하거나 비활성화합니다.domain 형식과 idempotency key를 확인한 뒤 tenant-service domain API로 위임합니다.tenant.domains.manage 권한이 필요합니다.
관리 사용자 목록/상세관리자가 tenant 사용자 목록을 검색하거나 상세 화면을 엽니다.auth-iam-service 사용자 목록/상세 API와 user-role 목록 API를 조합합니다.auth.users.read 권한이 필요하며 다른 tenant 사용자는 반환하지 않습니다.
관리 사용자 생성/수정관리자가 일반 사용자 계정을 생성하거나 이름/email/status를 수정합니다.tenant-scoped 사용자 API로 위임하고 general_user만 생성할 수 있게 제한합니다.auth.users.create 또는 auth.users.update 권한이 필요합니다.
Role/Permission 관리관리자가 role을 만들고 permission 목록을 교체합니다.auth-iam-service role, permission, role-permission API를 호출합니다.auth.roles.*, auth.rolePermissions.manage 계열 권한이 필요합니다.
Access control 화면 초기화관리자가 접근제어 화면에 진입합니다.users, roles, permissions 목록을 병렬 조회하고 화면 초기 데이터 형태로 조합합니다.관련 read 권한을 모두 확인한 뒤 조합 결과를 반환합니다.
Audit log 검색관리자가 기간, action, actor, resource, requestId로 감사 로그를 검색합니다.audit-log-service 내부 조회 API에 query와 context를 전달합니다.audit.logs.read 권한이 필요하며 감사 로그 목록 envelope를 반환합니다.
개발과정 API 문서

user-bff-service

일반 사용자 앱의 BFF입니다. 사용자 app shell, navigation, WMS 화면 데이터를 조합하고 tenant/module/permission 기준으로 화면 노출을 제어합니다.

App shell 구현
주요 호출자web-wms, gateway-service
핵심 책임tenant active 확인, module 사용 가능 여부 확인, permission summary 조회, app shell 데이터 조합, user-facing API envelope 제공
구현 기능GET /api/app/me, GET /api/app/navigation, app context loaded/navigation loaded audit best-effort 기록
WMS 화면 계획창고 목록, 자재 목록, 재고 요약, WMS dashboard, 향후 입고/출고/포장 작업 화면 데이터를 wms-service 내부 API에서 조합합니다.
감사 정책app shell 진입과 navigation 조회는 user-bff-service가 EventBridge 또는 audit-log-service 내부 POST fallback으로 best-effort 기록합니다. 실패해도 원 업무 응답은 차단하지 않습니다.
후속 범위WMS 화면별 screen-data API 구현, 화면 권한별 navigation 세분화, web-wms 첫 화면 연결

상세 기능

User BFF는 현장 사용자가 보는 화면 데이터를 한 번에 읽기 좋게 조합하고, 메뉴/버튼 노출에 필요한 permission summary를 함께 제공합니다.

기능사용자 행위시스템 처리권한/결과
App context 조회사용자가 로그인 후 앱 첫 화면에 진입합니다.사용자 정보, tenant 상태, enabled module, permission summary를 조회해 app shell context로 조합합니다.GET /api/app/me 응답으로 사용자/tenant/module/permission 정보를 반환합니다.
Navigation 조회사용자가 사이드바나 상단 메뉴를 로드합니다.tenant module과 permission summary를 기준으로 표시 가능한 메뉴를 계산합니다.GET /api/app/navigation 응답으로 메뉴 tree와 사용할 수 없는 항목의 제외 결과를 반환합니다.
Tenant readiness guard사용자가 업무 API를 호출합니다.tenant-service status API로 active tenant인지 확인합니다.inactive/suspended tenant는 TENANT_NOT_READY로 차단합니다.
Module guard사용자가 WMS 화면 또는 API에 접근합니다.tenant-service modules API로 wms module이 enabled인지 확인합니다.module이 꺼진 tenant는 WMS 화면 데이터를 반환하지 않습니다.
WMS 창고 화면 데이터사용자가 창고/로케이션 기준정보 화면을 엽니다.wms-service 창고/로케이션 API를 호출해 필터와 paging을 화면 모델로 변환합니다.후속 구현 대상이며 wms.warehouses.manage, wms.locations.manage 또는 읽기 권한 분리가 필요합니다.
WMS 재고 화면 데이터사용자가 재고 검색 조건을 입력하거나 요약을 조회합니다.wms-service inventory/snapshot API를 호출해 현재고, 가용 수량, 과거 스냅샷을 조합합니다.후속 구현 대상이며 wms.inventory.read 권한이 필요합니다.
WMS 작업 화면 데이터사용자가 입고, 출고, 포장 작업 화면을 엽니다.WMS 업무 API 결과를 사용자 화면용 상태명, 액션 가능 여부, 권한 플래그로 변환합니다.후속 구현 대상이며 작업별 권한이 없는 사용자는 액션 버튼을 노출하지 않습니다.
App audit 기록사용자가 app context 또는 navigation을 조회합니다.EventBridge 또는 internal-api fallback으로 userBff.appContext.loaded, userBff.navigation.loaded 기록을 시도합니다.감사 기록 실패는 warning log로 남기고 원 응답은 성공 처리합니다.
개발과정 API 문서

auth-iam-service

인증, 사용자, role, permission의 source of truth입니다. JWT는 최소 claim만 담고 권한 판단은 이 서비스가 수행합니다.

1차 구현
주요 호출자gateway-service, admin-bff-service, user-bff-service, wms-service
인증 기능로그인, access token 발급, refresh token rotation, logout, 현재 사용자 조회를 제공합니다.
사용자 관리tenant-scoped 일반 사용자 생성/조회/수정/비활성화, user type 분리, system_admin과 general_user 경계 검증을 담당합니다.
RBAC 관리permission catalog, tenant role, role-permission 교체, user-role 부여/해제, warehouse scope 기반 user-role을 관리합니다.
권한 확인GET /api/auth/permissions/summary와 POST /api/auth/permissions/check로 BFF 메뉴 노출과 도메인 실행 권한을 지원합니다.
이벤트auth.user.created, auth.user.updated, auth.user.statusChanged, auth.permission.created, auth.role.created, auth.role.updated, auth.role.permissionsReplaced, auth.userRole.assigned, auth.userRole.removed
후속 범위system_admin bootstrap 분리 강화, permission catalog 운영 UI와 검증 증적 축적, ABAC 확장 검토

상세 기능

Auth/IAM은 사용자 identity와 권한 판단의 기준점입니다. 도메인 서비스는 권한을 자체 해석하지 않고 이 서비스의 check API를 호출합니다.

기능입력/행위시스템 처리결과/이벤트
로그인사용자가 email/password를 제출합니다.사용자 상태, password hash, tenantId, userType을 확인하고 access token과 refresh token을 발급합니다.성공 시 JWT와 refresh token을 반환하고, 실패 시 AUTH_INVALID_CREDENTIALS 또는 계정 상태 오류를 반환합니다.
Token refresh클라이언트가 refresh token으로 새 access token을 요청합니다.서버 저장 refresh token을 조회하고 만료/폐기 여부를 확인한 뒤 refresh token rotation을 수행합니다.새 access token과 새 refresh token을 반환하고 이전 refresh token은 폐기합니다.
Logout사용자가 로그아웃합니다.전달된 refresh token 또는 사용자 세션을 폐기합니다.이후 같은 refresh token 재사용은 실패합니다.
내 정보 조회gateway가 검증한 사용자 context로 GET /api/auth/me를 호출합니다.사용자 기본 정보와 tenant/userType/status를 조회합니다.BFF app/admin shell에서 사용할 사용자 정보를 반환합니다.
사용자 생성관리자가 tenant 사용자 email, displayName, password를 입력합니다.tenant scope와 userType 규칙을 검증하고 password를 hash해 저장합니다.auth.user.created outbox 이벤트를 저장합니다. email 중복은 AUTH_USER_EMAIL_CONFLICT입니다.
사용자 수정/상태 변경관리자가 사용자 이름, email 또는 status를 변경합니다.tenant scope로 대상 사용자를 찾고 수정 가능 필드를 검증합니다.auth.user.updated 또는 auth.user.statusChanged 이벤트를 저장합니다.
Permission catalog 관리관리자가 새 permission code를 등록하거나 목록을 조회합니다.{domain}.{resource}.{action} 형식의 code 중복을 검증합니다.생성 시 auth.permission.created 이벤트를 저장합니다.
Role 관리관리자가 tenant role을 생성, 조회, 수정합니다.tenant scope 안에서 role code/name을 검증하고 저장합니다.auth.role.created, auth.role.updated 이벤트를 저장합니다.
Role permission 교체관리자가 role에 포함될 permission code 목록을 저장합니다.요청된 permission 존재 여부를 확인하고 role-permission mapping을 전체 교체합니다.auth.role.permissionsReplaced 이벤트를 저장하고 최종 permission 목록을 반환합니다.
User role 부여/해제관리자가 사용자에게 role을 부여하거나 제거합니다.tenant scope, role/user 존재 여부, optional warehouse scope를 검증합니다.auth.userRole.assigned, auth.userRole.removed 이벤트를 저장합니다.

발행 대상local/mock, EventBridge, SQS publisher를 지원하며 audit event queue로 업무 이벤트를 전달할 수 있습니다.
처리 정책pending event claim, lock timeout, retry count, failed 상태 기록, published target과 provider message id 기록을 담당합니다.
운영 APIGET /internal/operations/outbox-relay/status, GET /internal/operations/outbox-relay/sources로 worker와 source별 pending/failed 상태를 확인합니다.
후속 범위source별 운영 alert, DLQ replay 정책, EventBridge schema registry, 멀티 source 장애 격리

상세 기능

outbox-relay-service는 각 서비스가 transaction 안에 저장한 outbox event를 외부 메시징으로 옮기는 신뢰성 계층입니다.

기능입력/상태시스템 처리결과/운영 기준
Source 등록환경 변수에 source 이름과 DB URL이 설정됩니다.source registry가 각 DB의 outbox adapter를 구성합니다.설정 누락 source는 readiness에서 확인 대상이 됩니다.
Pending event pollingsource DB에 pending outbox row가 있습니다.worker가 batch size만큼 조회하고 lock을 획득합니다.다른 relay worker가 같은 event를 동시에 처리하지 못하게 합니다.
Event envelope 검증outbox row의 column 값과 payload 값이 들어옵니다.eventId, eventType, schemaVersion, tenantId, requestId, aggregateId 등 payload와 outbox column 일치 여부를 확인합니다.불일치하면 발행하지 않고 실패 처리합니다.
SQS 발행publisher type이 SQS로 설정되어 있습니다.event payload를 MessageBody로 보내고 eventId/eventType/tenantId/requestId를 message attributes에 넣습니다.발행 성공 시 outbox row를 published로 갱신하고 target/message id를 기록합니다.
EventBridge 발행publisher type이 EventBridge로 설정되어 있습니다.event bus, source prefix, detailType 기준으로 이벤트를 발행합니다.성공 시 published target을 기록합니다.
Retry 처리일시적 네트워크 오류나 publisher 오류가 발생합니다.retryCount를 증가시키고 nextRetryAt을 계산합니다.최대 재시도 전까지 pending 상태로 재시도하며, 초과하면 failed로 전환합니다.
Lock timeout 복구worker가 event를 lock한 뒤 비정상 종료됩니다.lockedAt이 timeout을 넘은 row를 다시 claim 가능하게 봅니다.중단된 이벤트가 영구 정체되지 않게 합니다.
운영 상태 조회운영자가 relay 상태 API를 호출합니다.worker enabled, publisher type, 처리 카운터, 마지막 성공/실패 정보를 반환합니다.운영 대시보드나 수동 점검에서 사용합니다.
Source 상태 조회운영자가 source별 backlog를 확인합니다.pending/failed count와 oldest pending age를 집계합니다.특정 서비스 outbox 적체를 빠르게 식별합니다.

web-admin

운영자와 관리자를 위한 React + Vite 웹 앱입니다. Admin BFF API를 사용해 tenant와 접근제어 운영 화면을 제공합니다.

착수 전
주요 사용자system_admin, tenant_admin, 운영 담당자
연결 서비스gateway-service를 통해 admin-bff-service의 /api/admin/** API를 호출합니다.
Tenant 화면tenant 목록, 상세, 생성, 상태 변경, module 변경, domain 관리 화면을 제공합니다.
사용자/RBAC 화면사용자 목록/상세/생성/상태 변경, role 관리, permission catalog, role permission 교체, user-role 부여/해제 화면을 제공합니다.
Audit 화면action, actor, 기간, resource, requestId 기준 검색과 evidence 상세 확인 화면을 제공합니다.
UX 기준위험 작업은 실행 전 확인을 받고, 완료 후 requestId와 변경 결과를 확인할 수 있어야 합니다.
후속 범위로그인 이후 admin shell, tenant 관리 화면, access-control 화면, audit-log 화면 순서로 구현합니다.

상세 기능

web-admin은 운영자가 시스템 설정과 접근제어를 반복적으로 처리하는 업무 도구입니다. 화면은 dense하고 비교/검색/확인이 쉬운 구조를 우선합니다.

화면/기능사용자 행위표시/처리 내용완료 기준
로그인/세션관리자가 계정으로 로그인하고 관리자 앱에 진입합니다.gateway의 auth API를 통해 token을 받고 admin shell 진입 시 사용자/tenant context를 불러옵니다.인증 실패, 만료, 권한 부족 상태가 명확히 분리되어 표시됩니다.
Admin shell관리자가 메뉴를 이동합니다.사용자 정보, 현재 tenant, 권한 요약, 관리 메뉴를 표시합니다.권한이 없는 메뉴는 숨기거나 disabled 상태로 표현합니다.
Tenant 목록tenant를 검색, 정렬, 상태별 필터링합니다.tenant code, 이름, 상태, domain, enabled module, 생성/수정 시간을 표로 보여줍니다.검색 결과와 page 상태가 URL/query와 동기화됩니다.
Tenant 생성관리자가 tenant 이름, domain, module을 입력합니다.필수값 검증, domain 형식 검증, idempotency key 기반 생성 요청을 보냅니다.생성 후 tenantId, tenant.code, requestId를 확인할 수 있습니다.
Tenant 상세/상태 변경관리자가 tenant 상태나 기본 정보를 수정합니다.상태 변경 전 확인 UI를 제공하고 변경 사유를 입력받습니다.완료 후 변경된 상태와 audit 추적용 requestId를 보여줍니다.
Module/Domain 관리관리자가 tenant별 module과 domain을 조정합니다.현재 enabled module과 domain 목록을 비교 가능하게 보여주고 변경 요청을 보냅니다.추가/비활성화 결과와 충돌 오류를 화면에서 확인합니다.
사용자 관리관리자가 tenant 사용자를 검색, 생성, 수정, 비활성화합니다.email, 표시명, 상태, role 요약, userType을 표시하고 general_user 생성만 허용합니다.system_admin 생성 시도는 화면/서버 양쪽에서 차단됩니다.
Role/Permission 관리관리자가 role을 만들고 permission을 선택해 저장합니다.permission catalog를 domain/resource/action 기준으로 검색하고 role permission set을 편집합니다.저장 후 최종 permission 목록과 requestId를 보여줍니다.
User-role 관리관리자가 사용자에게 role과 optional warehouse scope를 부여합니다.사용자 상세에서 현재 role 목록과 scope를 보여주고 부여/해제를 실행합니다.중복 부여와 존재하지 않는 role/scope 오류를 명확히 표시합니다.
Audit log 조회관리자가 기간, action, actor, resource, requestId로 검색합니다.감사 로그 목록, action, actor, resource, result, occurredAt, requestId, details를 표시합니다.업무 변경 후 evidence를 requestId로 추적할 수 있습니다.

web-wms

WMS 사용자와 현장 작업자를 위한 React + Vite 웹 앱입니다. User BFF의 app shell과 WMS screen-data API를 기준으로 화면을 구성합니다.

착수 전
주요 사용자창고 관리자, 재고 담당자, 입고 담당자, 출고/포장 담당자, 현장 작업자
연결 서비스gateway-service를 통해 user-bff-service의 /api/app/** API를 호출합니다.
App shell로그인 후 사용자, tenant, module, permission summary, navigation을 표시합니다.
WMS Dashboard창고 현황, 재고 요약, 입고/출고 작업 상태, 예외 작업 알림을 보여줍니다.
기준정보 화면창고, 로케이션, 품목 조회와 필터링을 제공합니다.
재고 화면현재고 검색, 가용 수량, 재고 조정 요청, 과거 스냅샷 조회를 제공합니다.
입출고/포장 화면입고 확정, 출고 할당, 포장 작업, 패키지 등록, 포장 확정, 출하 완료 흐름을 화면화합니다.
후속 범위User BFF WMS screen-data API 확정 후 dashboard, inventory, 기준정보, 작업 화면 순서로 구현합니다.

상세 기능

web-wms는 현장 업무 속도를 우선하는 화면입니다. 재고와 작업 상태를 빠르게 스캔하고, 실수하기 쉬운 수량/상태 변경은 명확한 확인과 evidence를 남겨야 합니다.

화면/기능사용자 행위표시/처리 내용완료 기준
앱 진입사용자가 로그인 후 WMS 앱에 접근합니다.app context와 navigation을 불러와 tenant, 사용자, module, 권한별 메뉴를 표시합니다.WMS module이 비활성 tenant이면 업무 메뉴를 노출하지 않습니다.
WMS dashboard사용자가 오늘의 창고 운영 상태를 확인합니다.현재고 요약, 입고 확정 대기, 출고 할당/포장/출하 상태, 예외 작업을 카드나 표로 보여줍니다.각 요약에서 상세 목록으로 이동할 수 있습니다.
창고/로케이션 조회사용자가 창고나 로케이션 기준으로 재고를 필터링합니다.창고, 로케이션 type, 사용 가능 상태, 품목 수량 연결 정보를 표시합니다.선택한 창고/로케이션 조건이 재고/입출고 화면으로 이어집니다.
품목 조회사용자가 SKU나 품명으로 자재를 검색합니다.SKU, 품명, 단위, 상태, 재고 존재 여부를 목록으로 보여줍니다.선택한 품목으로 현재고와 스냅샷 조회에 진입할 수 있습니다.
현재고 조회사용자가 품목/창고/로케이션 조건으로 재고를 조회합니다.quantity, allocatedQuantity, availableQuantity를 함께 보여주고 부족/음수 불가 상태를 구분합니다.재고 조정 권한이 있는 사용자에게만 조정 액션을 표시합니다.
재고 조정사용자가 실사 수량 차이를 입력하고 조정 사유를 선택합니다.수량 증감, reason, referenceNo를 받아 WMS 조정 API를 호출합니다.완료 후 변경 전/후 수량과 requestId, outbox/audit 추적 정보를 확인합니다.
과거 스냅샷 조회사용자가 특정 날짜의 보유 자재 수량을 조회합니다.snapshotDate와 필터를 User BFF/WMS snapshot API로 전달하고 과거 수량을 표시합니다.현재고와 스냅샷 수량이 구분되어 표시됩니다.
입고 확정입고 담당자가 도착 수량을 확인하고 확정합니다.품목, 창고, 로케이션, 수량, referenceNo를 입력받아 WMS 입고 확정 API를 호출합니다.확정 후 현재고 증가와 requestId를 표시합니다.
출고 할당출고 담당자가 주문 수량을 할당합니다.availableQuantity를 화면에서 먼저 보여주고 부족 가능성을 안내한 뒤 WMS 할당 API를 호출합니다.부족 시 WMS_INSUFFICIENT_STOCK 메시지와 현재 가용 수량을 표시합니다.
포장 작업포장 담당자가 allocation을 선택해 packing을 만들고 패키지를 등록합니다.포장 대상 수량, 등록된 박스, 품목별 포장 수량, 미포장 잔량을 계속 계산합니다.합계 불일치 시 포장 확정을 막고 어떤 품목이 부족/초과인지 표시합니다.
출하 완료담당자가 allocation 또는 packing 기준으로 출하를 확정합니다.출하 전 상태와 수량을 확인시키고 WMS shipment API를 호출합니다.완료 후 현재고/할당 수량 차감 결과와 requestId를 표시합니다.
권한별 액션 제어사용자마다 가능한 업무 액션이 다릅니다.permission summary를 기준으로 조정, 입고 확정, 출고 할당, 포장, 출하 버튼 노출을 제어합니다.권한이 없는 사용자는 화면에서 실행 버튼을 볼 수 없고, 서버에서도 다시 차단됩니다.
