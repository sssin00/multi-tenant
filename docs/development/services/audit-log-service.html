audit-log-service 개발과정

사용자와 관리자의 중요한 업무 행위를 append-only 감사 로그로 저장하고 조회합니다.

서비스/앱 역할

위치apps/audit-log-service
카테고리Backend Service · Audit
책임감사 로그 저장, 검색/조회, 보존 정책, evidence 추적, requestId 기반 시스템 로그 연결
연동auth-iam-service와 WMS 등 도메인 서비스가 내부 API로 audit record 저장을 요청하고, admin-bff-service가 운영자 조회 API로 위임합니다.
주의감사 로그는 시스템 로그와 분리하고, 수정/삭제가 아닌 append-only 저장 원칙을 따릅니다.

구현 상태

상태착수 전
선행 조건감사 로그 저장 API 계약, 내부 서비스 인증, 저장소 보존 정책, Admin BFF 조회 계약을 확정해야 합니다.
API 문서docs/apis/audit-log-service/index.html에 내부 저장/조회 API 초안을 기록했습니다.

기능별 작업 내역

Storage감사 로그는 append-only로 저장하며 tenantId, requestId, actor, action, resource, result를 필수로 둡니다.
Internal API저장 API는 내부 서비스 인증을 요구하고, 조회 API는 Admin BFF 전용 내부 호출로 제한합니다.
Security비밀번호, token, secret, SQL, stack trace, 민감 개인정보 원문은 저장하지 않습니다.
Querytenant, actor, action, resource, result, 기간, requestId 기준 검색과 page 기반 pagination을 제공합니다.

테스트 시나리오

AUDIT-001내부 인증이 유효한 서비스는 감사 로그를 저장할 수 있다.
AUDIT-002필수 필드가 누락된 audit record는 VALIDATION_FAILED로 거절된다.
AUDIT-003Admin BFF는 tenant, action, 기간 조건으로 감사 로그를 조회할 수 있다.
AUDIT-004저장된 감사 로그는 수정/삭제 API 없이 append-only 원칙을 유지한다.

후속 작업

현재 audit-log-service 착수 전 이어서 확정할 작업입니다. 이 표의 작업이 진행되면 관련 코드와 문서를 함께 갱신합니다.

우선순위 작업 범위 완료 기준 함께 갱신할 문서
P0 내부 저장 API 계약 확정 POST /api/internal/audit/logs의 request body, 내부 인증 header, 실패 시 처리 기준을 확정합니다. auth-iam-service가 사용자/권한 변경 성공 후 audit record를 안정적으로 저장 요청할 수 있습니다. audit-log-service API 문서, auth-iam-service 개발 문서, logging/audit 규칙
P0 Admin BFF 조회 계약 확정 GET /api/internal/audit/logs의 검색 조건, pagination, 권한, tenant scope 기준을 정합니다. Admin BFF의 GET /api/admin/audit-logs가 실제 audit-log-service 조회 API로 연결됩니다. admin-bff-service 개발 문서, admin-bff-service API 문서, audit-log-service API 문서
P1 보존 정책과 masking 기준 감사 로그 보존 기간, archive, PII masking, 검색 인덱스와 비용 기준을 정합니다. 운영 환경에서 감사 로그가 과도한 비용이나 민감 정보 노출 없이 보존됩니다. logging/audit 규칙, security/AWS 규칙, AWS ECS 기술 문서
