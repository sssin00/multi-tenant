Development Process

audit-log-service 개발과정

사용자와 관리자의 중요한 업무 행위를 append-only 감사 로그로 저장하고 조회합니다.

서비스/앱 역할

위치apps/audit-log-service
카테고리Backend Service · Audit
책임감사 로그 저장, 검색/조회, 보존 정책, evidence 추적, requestId 기반 시스템 로그 연결
연동auth-iam-service, tenant-service, WMS 등 도메인 서비스는 감사 대상 이벤트를 outbox에 저장하고, audit-log-service가 EventBridge/SQS 또는 raw event envelope를 소비해 audit record를 저장합니다. user-bff-service는 자체 outbox가 없는 app shell 감사 이벤트를 EventBridge로 best-effort 발행할 수 있으며, 내부 POST API는 local/호환 fallback으로 유지합니다. admin-bff-service는 운영자 조회 API를 위임하며, gateway-service는 audit-log-service를 직접 호출하지 않습니다.
주의감사 로그는 시스템 로그와 분리하고, 수정/삭제가 아닌 append-only 저장 원칙을 따릅니다.

기능 정의

감사 기록 저장권한 변경, tenant 설정 변경, 사용자 변경, 재고 조정 등 중요한 업무 행위를 append-only audit record로 저장합니다.
감사 기록 조회Admin BFF가 tenant, 기간, actor, action, resource, result, requestId 조건으로 조회할 수 있는 내부 API를 제공합니다. tenant 없는 system_admin 요청은 전체 감사 로그 조회를 위해 tenant 필터를 생략할 수 있습니다.
Evidence 연결requestId, actor, resource, result, details를 통해 시스템 로그와 업무 행위를 추적할 수 있게 합니다.
보존/마스킹감사 로그 보존 정책과 민감정보 저장 금지, 필요한 식별자의 마스킹 기준을 적용합니다.
내부 호출 보호저장/조회 API는 internal auth를 요구하고 gateway에서 직접 노출하지 않습니다.
제외 범위일반 시스템 로그 저장소, 실시간 모니터링 metric, 도메인 업무 상태 변경은 audit-log-service 기능이 아닙니다.

첫 구현 범위

Service Shell

NestJS 앱 기본 구조, health/readiness, config, request context, access log, exception filter를 구성합니다.

Audit Store

PostgreSQL + Prisma 기반 audit_logs 저장소를 만들고 append-only 원칙을 적용합니다.

Audit Event Consumer

업무 행위 완료 이벤트를 소비해 append-only 감사 record를 저장합니다.

GET /api/internal/audit/logs

Admin BFF가 tenant 범위 또는 슈퍼 관리자 전체 범위의 감사 로그를 조건별로 조회하는 API입니다.

POST /api/internal/audit/logs

User BFF app shell과 Admin BFF 관리자 변경성 action 감사 이벤트를 internal-api fallback으로 저장하는 내부 API입니다.

Admin BFF 연동

GET /api/admin/audit-logs를 실제 audit-log-service 조회 API로 연결합니다.

Security

password, token, secret, Authorization, SQL, stack trace, 민감 개인정보 원문 저장을 금지합니다.

개발 순서

1
NestJS 기본 서비스 구성

apps/audit-log-service를 실제 NestJS 앱으로 전환하고 config, health, readiness를 먼저 만듭니다.

2
Audit 데이터 모델 설계

audit_logs 모델과 tenant/request/action/resource/result 기준 index를 정의합니다.

3
이벤트 기반 저장 구현

outbox relay가 발행한 감사 대상 이벤트를 소비하고 eventId 기준 중복 저장 방지를 적용합니다.

4
내부 조회 API 구현

GET /api/internal/audit/logs를 구현해 Admin BFF가 page 기반으로 조회할 수 있게 합니다.

5
Admin BFF 조회와 이벤트 저장 연동

Admin BFF 조회 API는 audit-log-service 내부 조회 API로 연결하고, Auth/IAM 저장 경로는 outbox event 기반으로 전환합니다.

6
Docker와 Local 검증

local compose, env, DB reset, Postman collection을 갱신하고 시나리오 기반으로 검증합니다.

보안/저장 제외

실패 정책도메인 업무 API는 audit-log-service HTTP 저장 응답에 결합하지 않습니다. 저장 실패는 outbox-relay-service의 retry, failed 상태, DLQ 또는 replay 정책으로 처리합니다. User BFF app shell 감사는 best-effort HTTP 기록이며 실패 시 warning log만 남깁니다.
권한외부 조회는 Admin BFF의 audit.logs.read permission을 통과한 요청만 audit-log-service로 전달됩니다.

운영 DB Secret과 Migration 기준

GitHub secretAUDIT_DATABASE_URL_SECRET_ARN에는 AWS Secrets Manager secret ARN만 저장합니다. DB URL 원문은 GitHub secret이나 repository 파일에 두지 않습니다.
Secrets Manager 이름권장 이름은 multi-tenant-{env}-audit-log-service-database-url입니다.
SecretString 형식plain PostgreSQL URL 또는 JSON { "DATABASE_URL": "postgresql://..." }, { "databaseUrl": "postgresql://..." }, { "url": "postgresql://..." } 형식을 허용합니다.
Runtime 주입CDK context auditDatabaseUrlSecretArn을 ECS task secret DATABASE_URL로 주입합니다.
Migration 명령배포 DB schema 변경은 pnpm --filter audit-log-service prisma:migrate:deploy만 사용합니다. 운영/staging/dev 배포 DB에 prisma db push 또는 prisma migrate dev를 사용하지 않습니다.
Workflowdeploy-audit-log-service.yml 수동 실행 시 runAuditPrismaMigration=true를 선택하면 image deploy 전에 Secrets Manager에서 DB URL을 읽어 migration을 실행합니다.
네트워크 예외DB가 private subnet이라 GitHub-hosted runner에서 접근할 수 없으면 승인된 self-hosted runner, bastion, 또는 ECS one-off task에서 같은 명령을 실행합니다.
실패 정책migration 실패 시 ECS deploy를 진행하지 않습니다. prisma migrate resolve는 운영자 확인 후 실패 이력 정리에만 사용합니다.

테스트 시나리오

AUDIT-001내부 인증이 유효한 user-bff-service는 app shell 감사 로그를 저장할 수 있다.
AUDIT-002필수 필드가 누락된 audit record는 VALIDATION_FAILED로 거절된다.
AUDIT-003Admin BFF는 tenant, action, 기간 조건으로 감사 로그를 조회할 수 있다.
AUDIT-004저장된 감사 로그는 수정/삭제 API 없이 append-only 원칙을 유지한다.
AUDIT-005다른 tenant의 감사 로그는 조회 결과에 포함되지 않는다.
AUDIT-006password/token/secret이 details에 포함되면 저장 전 거절하거나 제거한다.
AUDIT-007auth-iam-service의 사용자 생성 성공 후 audit record가 저장된다.
AUDIT-008Admin BFF의 audit 조회 API는 audit.logs.read 권한이 없으면 거절된다.
AUDIT-009auth-iam-service의 role/permission 변경 성공 후 audit record가 저장된다.
