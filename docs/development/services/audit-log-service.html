Development Process

audit-log-service 개발과정

사용자와 관리자의 중요한 업무 행위를 append-only 감사 로그로 저장하고 조회합니다.

서비스/앱 역할

위치apps/audit-log-service
카테고리Backend Service · Audit
책임감사 로그 저장, 검색/조회, 보존 정책, evidence 추적, requestId 기반 시스템 로그 연결
연동auth-iam-service와 WMS 등 도메인 서비스가 내부 API로 audit record 저장을 요청하고, admin-bff-service가 운영자 조회 API로 위임합니다. gateway-service가 audit-log-service를 직접 호출하지 않습니다.
주의감사 로그는 시스템 로그와 분리하고, 수정/삭제가 아닌 append-only 저장 원칙을 따릅니다.

기능 정의

감사 기록 저장권한 변경, tenant 설정 변경, 사용자 변경, 재고 조정 등 중요한 업무 행위를 append-only audit record로 저장합니다.
감사 기록 조회Admin BFF가 tenant, 기간, actor, action, resource, result, requestId 조건으로 조회할 수 있는 내부 API를 제공합니다.
Evidence 연결requestId, actor, resource, result, details를 통해 시스템 로그와 업무 행위를 추적할 수 있게 합니다.
보존/마스킹감사 로그 보존 정책과 민감정보 저장 금지, 필요한 식별자의 마스킹 기준을 적용합니다.
내부 호출 보호저장/조회 API는 internal auth를 요구하고 gateway에서 직접 노출하지 않습니다.
제외 범위일반 시스템 로그 저장소, 실시간 모니터링 metric, 도메인 업무 상태 변경은 audit-log-service 기능이 아닙니다.

첫 구현 범위

Service Shell

NestJS 앱 기본 구조, health/readiness, config, request context, access log, exception filter를 구성합니다.

Audit Store

PostgreSQL + Prisma 기반 audit_logs 저장소를 만들고 append-only 원칙을 적용합니다.

POST /api/internal/audit/logs

내부 서비스가 업무 행위 완료 후 audit record를 저장하는 API입니다.

GET /api/internal/audit/logs

Admin BFF가 tenant 범위의 감사 로그를 조건별로 조회하는 API입니다.

Admin BFF 연동

GET /api/admin/audit-logs를 실제 audit-log-service 조회 API로 연결합니다.

Security

password, token, secret, Authorization, SQL, stack trace, 민감 개인정보 원문 저장을 금지합니다.

개발 순서

1
NestJS 기본 서비스 구성

apps/audit-log-service를 실제 NestJS 앱으로 전환하고 config, health, readiness를 먼저 만듭니다.

2
Audit 데이터 모델 설계

audit_logs 모델과 tenant/request/action/resource/result 기준 index를 정의합니다.

3
내부 저장 API 구현

POST /api/internal/audit/logs를 구현하고 internal auth, validation, 민감정보 저장 금지 기준을 적용합니다.

4
내부 조회 API 구현

GET /api/internal/audit/logs를 구현해 Admin BFF가 page 기반으로 조회할 수 있게 합니다.

5
Admin BFF와 Auth/IAM 연동

Admin BFF 조회 API와 Auth/IAM audit 저장 client를 실제 audit-log-service로 연결합니다.

6
Docker와 Local 검증

local compose, env, DB reset, Postman collection을 갱신하고 시나리오 기반으로 검증합니다.

보안/저장 제외

저장 금지password, accessToken, refreshToken, Authorization, cookie, secret, API key, private key, SQL, stack trace
마스킹email, phone, 사업자등록번호처럼 운영상 식별이 필요한 값은 원문 대신 마스킹 값이나 별도 identifier만 저장합니다.
실패 정책1차는 audit 저장 실패 시 caller가 오류를 알 수 있게 응답합니다. 비즈니스 처리와 audit 저장 실패의 최종 정책은 운영 단계에서 재검토합니다.
권한외부 조회는 Admin BFF의 audit.logs.read permission을 통과한 요청만 audit-log-service로 전달됩니다.

테스트 시나리오

AUDIT-001내부 인증이 유효한 서비스는 감사 로그를 저장할 수 있다.
AUDIT-002필수 필드가 누락된 audit record는 VALIDATION_FAILED로 거절된다.
AUDIT-003Admin BFF는 tenant, action, 기간 조건으로 감사 로그를 조회할 수 있다.
AUDIT-004저장된 감사 로그는 수정/삭제 API 없이 append-only 원칙을 유지한다.
AUDIT-005다른 tenant의 감사 로그는 조회 결과에 포함되지 않는다.
AUDIT-006password/token/secret이 details에 포함되면 저장 전 거절하거나 제거한다.
AUDIT-007auth-iam-service의 사용자 생성 성공 후 audit record가 저장된다.
AUDIT-008Admin BFF의 audit 조회 API는 audit.logs.read 권한이 없으면 거절된다.

후속 작업

현재 audit-log-service 착수 전 이어서 확정할 작업입니다. 이 표의 작업이 진행되면 관련 코드와 문서를 함께 갱신합니다.

우선순위 작업 범위 완료 기준 함께 갱신할 문서
P0 내부 저장 API 계약 확정 POST /api/internal/audit/logs의 request body, 내부 인증 header, 실패 시 처리 기준을 확정합니다. auth-iam-service가 사용자/권한 변경 성공 후 audit record를 안정적으로 저장 요청할 수 있습니다. audit-log-service API 문서, auth-iam-service 개발 문서, logging/audit 규칙
P0 Admin BFF 조회 계약 확정 GET /api/internal/audit/logs의 검색 조건, pagination, 권한, tenant scope 기준을 정합니다. Admin BFF의 GET /api/admin/audit-logs가 실제 audit-log-service 조회 API로 연결됩니다. admin-bff-service 개발 문서, admin-bff-service API 문서, audit-log-service API 문서
P1 보존 정책과 masking 기준 감사 로그 보존 기간, archive, PII masking, 검색 인덱스와 비용 기준을 정합니다. 운영 환경에서 감사 로그가 과도한 비용이나 민감 정보 노출 없이 보존됩니다. logging/audit 규칙, security/AWS 규칙, AWS ECS 기술 문서
