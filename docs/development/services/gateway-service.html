Development Process

gateway-service 개발과정

ALB 뒤에서 외부 요청을 받아 BFF와 내부 서비스로 전달하는 공통 출입구입니다.

서비스 역할

위치apps/gateway-service
책임request id 발급, tenant 식별, JWT 검증, 라우팅, rate limit, 공통 로그와 보안 정책 적용
라우팅/api/auth/**, /api/admin/**, /api/app/**를 각 서비스/BFF로 전달
주의재고 차감, 입고 완료, 출고 할당 같은 업무 로직과 DB 직접 접근은 넣지 않습니다.

첫 구현 범위

Health

GET /health, GET /ready

Context

x-request-id, x-tenant-id 추출과 전파

Auth

JWT guard 기본 구조와 public/private route 구분

Routing

user-bff, admin-bff, auth-iam으로 프록시 라우팅

Log

tenant_id, request_id, user_id 기반 access log

Policy

Redis rate limit, tenant status check, 보안 header/CORS 운영 기준

개발 기록

2026-06-01 NestJS 기본 구성

  • apps/gateway-service를 NestJS + TypeScript 실행 앱으로 구성했습니다.
  • 초기 API는 GET /health, GET /ready만 구현합니다.
  • 실행 포트는 GATEWAY_PORT 환경 변수로 받고, 기본값은 3000입니다.
  • 이번 단계에서는 request context, JWT guard, proxy routing, access log는 만들지 않고 다음 단계에서 추가합니다.

2026-06-08 Health API 보강

  • GET /health는 gateway-service 프로세스 생존 확인용 endpoint로 둡니다.
  • 응답에는 status, service, timestamp를 포함합니다.
  • DB, Redis, BFF, Auth/IAM 같은 외부 의존성 확인은 /health에 넣지 않고 /ready에서 다룹니다.
  • API 상세 문서는 docs/apis/gateway-service/get-health.html에 기록합니다.

2026-06-08 Docker 구성 추가

  • docker/services/Dockerfile.nest를 monorepo NestJS 앱 실행에 맞게 조정했습니다.
  • docker/local/docker-compose.yml에 gateway-service 서비스를 추가했습니다.
  • 로컬 개발 compose 프로젝트 이름은 gateway-service-local로 고정했습니다.
  • docker/deploy/docker-compose.yml을 단독 서버나 VM 실행용 배포 템플릿으로 추가했습니다.
  • AWS ECS 배포는 compose가 아니라 infra/cdk에서 관리합니다.

2026-06-08 Request Context와 Gateway 기본 보안 흐름

  • 요청별 requestId, tenantId, userId를 담는 request context를 추가했습니다.
  • X-Request-Id가 없으면 gateway-service가 UUID를 생성하고 응답 header와 upstream 요청에 전파합니다.
  • X-Tenant-Id는 request context에 먼저 담고, 인증 후 JWT tenantId와 불일치하면 403 TENANT_MISMATCH로 차단합니다.
  • 요청 완료 시 JSON access log를 남기며 method, path, statusCode, durationMs, requestId, tenantId, userId를 포함합니다.
  • /health, /ready는 public route로 유지하고, /api/**는 기본 private route로 처리합니다.
  • JWT verifier는 local-dev 기본 구조로 시작하며 sub, tenantId, exp를 확인합니다. roles, permissions는 JWT에서 사용하지 않습니다.
  • /api/auth/**, /api/admin/**, /api/app/** proxy routing 뼈대를 추가했습니다.
  • Proxy route API 문서는 docs/apis/gateway-service/proxy-routes.html에 기록합니다.
GWS-CTX-001GET /health 호출 시 X-Request-Id가 응답 header로 전파되고 access log에 남는다.
GWS-AUTH-001/api/app/** 호출 시 bearer token이 없으면 401 UNAUTHORIZED로 차단된다.
GWS-TENANT-001JWT tenantId와 X-Tenant-Id가 다르면 403 TENANT_MISMATCH로 차단된다.
GWS-PROXY-001JWT와 tenant header가 일치하면 request context header를 upstream으로 전파하는 proxy 흐름까지 진입한다.

2026-06-08 Proxy Body와 공통 에러 응답

  • Gateway에서 request body를 먼저 소비하지 않도록 NestJS bodyParser를 비활성화했습니다.
  • GET, HEAD를 제외한 proxy 요청은 원본 request stream을 upstream으로 전달합니다.
  • Proxy 요청 body와 query string은 내부 서비스 경로로 함께 전달합니다.
  • Global exception filter를 추가해 실패 응답을 success, requestId, timestamp, error envelope로 통일했습니다.
  • 에러 응답의 error는 code, message, details를 포함합니다.

2026-06-08 JWT 서명 검증

  • JWT verifier에 서명 검증을 추가했습니다.
  • 로컬 개발 기본값은 JWT_ALGORITHM=HS256와 랜덤 생성한 JWT_SECRET입니다.
  • HS256은 JWT_SECRET으로 HMAC 서명을 검증합니다.
  • RS256은 JWT_PUBLIC_KEY로 RSA SHA-256 서명을 검증할 수 있게 구성했습니다.
  • 서명이 맞지 않거나 허용되지 않은 algorithm이면 401 AUTH_INVALID_TOKEN으로 차단합니다.
  • 서명 검증 후에도 JWT payload에서 request context에 반영하는 값은 sub, tenantId만 유지합니다.

2026-06-08 JWT Issuer/Audience 검증

  • JWT_ISSUER 설정을 추가하고 토큰의 iss claim과 일치하는지 검증합니다.
  • JWT_AUDIENCE 설정을 추가하고 토큰의 aud claim과 일치하는지 검증합니다.
  • aud claim이 배열이면 설정된 audience가 배열에 포함되어야 통과합니다.
  • issuer 또는 audience가 맞지 않으면 401 AUTH_INVALID_TOKEN으로 차단합니다.
  • 검증 후 request context에 반영하는 값은 계속 sub, tenantId만 유지합니다.

2026-06-09 Ready API 기준 강화

  • GET /ready가 gateway-service의 트래픽 수용 가능 여부를 판단하도록 readiness check를 추가했습니다.
  • config check는 gateway port와 request context header 설정을 확인합니다.
  • jwt check는 signing key 설정과 JWT_ISSUER, JWT_AUDIENCE 설정을 확인합니다.
  • proxyRoutes check는 Auth/IAM, Admin BFF, User BFF upstream URL 형식을 확인합니다.
  • check 중 하나라도 실패하면 503 Service Unavailable과 status=not_ready를 반환합니다.

2026-06-09 Upstream Timeout과 Retry 정책

  • proxy route별 upstream timeout을 추가했습니다.
  • /api/auth/**는 기본 3000ms, /api/admin/**와 /api/app/**는 기본 5000ms를 적용합니다.
  • GET, HEAD, OPTIONS만 timeout, network error, upstream 502, 503, 504 응답에 대해 기본 1회 재시도합니다.
  • POST, PUT, PATCH, DELETE는 중복 실행 위험 때문에 재시도하지 않습니다.
  • 최종 timeout은 504 GATEWAY_TIMEOUT, 최종 연결 실패는 503 UPSTREAM_UNAVAILABLE로 응답합니다.
  • upstream 시도별 로그에 requestId, tenantId, userId, method, path, upstream, attempt, durationMs, statusCode, errorCode를 남깁니다.

2026-06-09 Redis 기반 Rate Limit

  • Redis fixed window 방식의 rate limit guard를 추가했습니다.
  • 제한 기준은 인증 후 tenantId + userId + routeGroup, 인증 전 IP + routeGroup입니다.
  • Redis key는 인증 후 rate-limit:{env}:gateway:{tenantId}:{userId}:{routeGroup} 형식을 사용합니다.
  • Redis key는 인증 전 rate-limit:{env}:gateway:anonymous:{ip}:{routeGroup} 형식을 사용합니다.
  • route group은 auth, admin, app으로 나눕니다.
  • 로컬 Docker 환경은 compose Redis를 사용하고, AWS 배포 환경은 ElastiCache Redis를 사용합니다.
  • 기본 제한은 auth=60/min, admin=300/min, app=600/min입니다.
  • 제한 초과 시 429 RATE_LIMIT_EXCEEDED, Redis 저장소 장애 시 503 RATE_LIMIT_UNAVAILABLE로 응답합니다.
  • 응답 header에 X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset을 추가합니다.

2026-06-09 AWS ElastiCache Redis 연동

  • gateway-service CDK stack에 ElastiCache Redis cache cluster를 추가했습니다.
  • Redis subnet group은 gateway-service app subnet을 사용합니다.
  • Redis security group은 gateway-service task security group에서 오는 6379 포트만 허용합니다.
  • ECS task 환경 변수 REDIS_URL은 CDK가 ElastiCache endpoint로 주입합니다.
  • 기본 Redis 노드 타입은 cache.t4g.micro이며 CDK context gatewayRedisNodeType으로 조정할 수 있습니다.

2026-06-09 Tenant Status Check 연동

  • tenant status check guard를 추가했습니다.
  • 기능 활성화 시 TENANT_SERVICE_URL의 GET /internal/tenants/{tenantId}/status를 호출합니다.
  • tenant-service 내부 API 계약에 맞춰 X-Internal-Service-Id, X-Internal-Timestamp, X-Internal-Signature HMAC 인증 header를 전송합니다.
  • tenant-service 응답 envelope의 data.status를 읽어 tenant 상태를 판단합니다.
  • tenant status는 Redis에 tenant-status:{env}:{tenantId} key로 기본 30초 캐시합니다.
  • active tenant만 통과하고 provisioning, suspended, deleted는 403 TENANT_NOT_READY로 차단합니다.
  • tenant status 조회 실패는 503 TENANT_STATUS_UNAVAILABLE로 응답합니다.
  • 로컬 gateway 환경은 tenant-service 연동 검증을 위해 GATEWAY_TENANT_STATUS_CHECK_ENABLED=true로 활성화합니다.

2026-06-11 tenant-service 실제 연결 정리

  • gateway-service의 tenant status check는 관리 API 우회가 아니라 업무 API 진입 전 active tenant 확인 용도로만 tenant-service를 직접 호출합니다.
  • GET /internal/tenants/{tenantId}/status HMAC 서명에서 GET 요청 body hash를 tenant-service guard의 canonical body 기준과 맞췄습니다.
  • login과 token refresh는 multi-tenant 예외 API이므로 tenant status check 대상에서 제외했습니다.
  • local compose 기준 TENANT_SERVICE_URL=http://tenant-service:3000, GATEWAY_TENANT_STATUS_CHECK_ENABLED=true, 공유 TENANT_INTERNAL_AUTH_SECRET로 연결합니다.

2026-06-09 Security Headers와 CORS 운영 기준

  • gateway-service 모든 응답에 공통 보안 header를 추가했습니다.
  • X-Content-Type-Options, X-Frame-Options, Referrer-Policy, Permissions-Policy, Content-Security-Policy, Cross-Origin-Opener-Policy, Cross-Origin-Resource-Policy를 적용합니다.
  • HTTPS 요청에는 Strict-Transport-Security를 추가합니다.
  • CORS는 GATEWAY_CORS_ALLOWED_ORIGINS에 명시된 origin만 허용합니다.
  • 운영 기준에 맞춰 wildcard origin은 readiness check에서 실패 처리합니다.
  • 로컬 환경은 localhost 개발 origin을 허용하고, dev/staging/prod는 환경별 명시 origin을 사용합니다.

2026-06-09 HTTPS, ACM, Route53 연결 기준

  • gateway-service CDK stack에 선택형 HTTPS listener 구성을 추가했습니다.
  • gatewayAcmCertificateArn context가 있으면 ALB 443 listener를 만들고 ACM 인증서를 연결합니다.
  • HTTPS listener가 활성화되면 ALB 80 listener는 HTTPS로 리다이렉트합니다.
  • Route53 hosted zone이 아직 없으므로 CDK는 DNS record를 생성하지 않습니다.
  • gatewayDomainName context를 전달하면 CDK output에 연결 대상 ALB DNS 이름을 함께 노출합니다.
  • Route53 도입 전에는 외부 DNS 제공자에서 subdomain CNAME 또는 ALIAS/ANAME을 ALB DNS 이름으로 연결합니다.
  • Route53 도입 후에는 hosted zone 기반 A/AAAA Alias record를 CDK 관리 리소스로 추가합니다.

2026-06-09 GitHub Actions 배포 자동화

  • .github/workflows/deploy-gateway-service.yml을 추가했습니다.
  • 현재 push 자동 배포는 당분간 비활성화되어 있으며, GitHub Actions 수동 실행으로만 배포합니다.
  • 자동 배포를 다시 켜면 dev 브랜치는 dev 환경, staging 브랜치는 staging 환경, main 브랜치는 prod 환경으로 배포합니다.
  • 배포는 GitHub OIDC로 AWS role을 assume하는 방식이며, 환경별 GitHub Environment secret AWS_ROLE_ARN을 사용합니다.
  • 첫 배포도 가능하도록 CDK를 gatewayDesiredCount=0으로 먼저 실행해 ECR/기본 인프라를 만들고, 이미지를 ECR에 push한 뒤 실제 desired count로 다시 배포합니다.
  • gateway workflow는 같은 CDK 스택의 auth/tenant 값을 함께 갱신하므로 AUTH_DESIRED_COUNT, AUTH_IMAGE_TAG, TENANT_DESIRED_COUNT, TENANT_IMAGE_TAG를 명시하지 않으면 실패합니다.
  • 이미지 tag는 {env}-{gitSha12} 형식을 사용합니다.
  • CDK stack은 GitHub Environment variable로 ACM 인증서 ARN, gateway domain, CORS origin, upstream service URL을 받을 수 있습니다.
  • JWT secret은 GitHub Environment secret GATEWAY_JWT_SECRET_ARN으로 Secrets Manager ARN을 전달하고, ECS task secret으로 주입합니다.

2026-06-09 Gateway 테스트 시나리오 문서화

  • gateway-service 1차 검증 시나리오를 개발 문서에 추가했습니다.
  • 시나리오는 health/readiness, JWT, tenant mismatch, tenant status, proxy 전달, timeout/retry, rate limit, security header/CORS, error envelope, branch deploy mapping을 포함합니다.
  • 각 시나리오는 scenarioId, priority, 기대 결과, 증거 기준을 함께 기록합니다.
  • 검증 증거는 response JSON, header, access log, upstream mock 수신 내용, Redis key, CloudFormation/GitHub Actions 결과를 기준으로 남깁니다.

2026-06-09 향후 서비스 개발 연동 작업 정리

  • 다른 서비스가 구현될 때 gateway-service에서 함께 확인할 작업 목록을 추가했습니다.
  • 대상은 Auth/IAM, tenant-service, Admin BFF, User BFF, WMS API, public/private route, 운영 도메인, Route53, 관측성, 배포 안정화입니다.
  • gateway-service는 업무 로직을 갖지 않고 인증, tenant, context, routing, 보안 경계만 담당한다는 기준을 유지합니다.

2026-06-08 Gateway 환경 분리

  • gateway-service 환경명을 local, dev, staging, prod로 통일했습니다.
  • 환경별 compose 설정은 docker/env/gateway-service/.env.{env}에서 관리합니다.
  • 로컬 개발 compose는 .env.local을 사용하고, 배포용 compose는 선택한 환경 env 파일을 주입하는 구조로 분리했습니다.
  • 애플리케이션 설정 검증의 APP_ENV 허용값을 local, dev, staging, prod로 맞췄습니다.
  • CDK 기본 환경명은 dev로 유지하고, 운영 환경 판정은 prod 기준으로 정리했습니다.
