Development Process

gateway-service 개발과정

ALB 뒤에서 외부 요청을 받아 BFF와 내부 서비스로 전달하는 공통 출입구입니다.

서비스 역할

위치apps/gateway-service
책임request id 발급, tenant 식별, JWT 검증, 라우팅, rate limit, 공통 로그와 보안 정책 적용
라우팅/api/auth/**, /api/admin/**, /api/app/**를 각 서비스/BFF로 전달
주의재고 차감, 입고 완료, 출고 할당 같은 업무 로직과 DB 직접 접근은 넣지 않습니다.

첫 구현 범위

Health

GET /health, GET /ready

Context

x-request-id, x-tenant-id 추출과 전파

Auth

JWT guard 기본 구조와 public/private route 구분

Routing

user-bff, admin-bff, auth-iam으로 프록시 라우팅

Log

tenant_id, request_id, user_id 기반 access log

Later

Redis rate limit, tenant status check, 보안 헤더

개발 기록

2026-06-01 NestJS 기본 구성

  • apps/gateway-service를 NestJS + TypeScript 실행 앱으로 구성합니다.
  • 초기 API는 GET /health, GET /ready만 구현합니다.
  • 실행 포트는 GATEWAY_PORT 환경 변수로 받고, 기본값은 3000입니다.
  • pnpm --filter gateway-service typecheck, pnpm --filter gateway-service build를 통과했습니다.
  • 로컬 실행 후 /health, /ready 응답을 확인했습니다.
  • 이번 단계에서는 request context, JWT guard, proxy routing, access log는 만들지 않고 다음 단계에서 추가합니다.
