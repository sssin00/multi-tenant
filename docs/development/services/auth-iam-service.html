서비스/앱 역할

위치apps/auth-iam-service
카테고리Backend Service · Auth/IAM
책임사용자 인증, JWT 발급, refresh token rotation, 사용자/역할/권한 관리, permission summary/check 제공
연동gateway-service의 /api/auth/** proxy route 뒤에서 동작하고, BFF와 도메인 서비스에 권한 조회 API를 제공합니다.

P2 Login throttling과 account lock 정책 Redis 기반 실패 횟수 추적, 임시 lock, 운영 alert 기준을 정합니다. 반복 로그인 실패가 rate/lock 정책으로 제어되고 정상 사용자 복구 경로가 문서화됩니다. Auth/IAM API 문서, security/AWS 규칙, 운영 Runbook

Login, JWT, Refresh Token
Login

POST /api/auth/login은 tenant context, email, password를 검증하고 active 사용자에게만 token을 발급합니다.

Password

password hash는 scrypt와 사용자별 random salt를 사용합니다. 저장 형식은 scrypt$salt$key이며 비교는 timing-safe 방식으로 처리합니다.

JWT

access token은 기본 30분이며 sub, tenantId, type, iat, exp, iss, aud만 포함합니다.

Refresh

POST /api/auth/token/refresh는 기존 refresh token을 폐기하고 새 access/refresh token을 발급합니다. 재사용된 token은 401 AUTH_INVALID_TOKEN으로 실패합니다.

Revoke

POST /api/auth/logout은 refresh token 단건 폐기 또는 gateway가 전달한 사용자 context 기반 전체 token 폐기를 지원합니다.

Me

GET /api/auth/me는 gateway가 전달한 X-Tenant-Id, X-User-Id 기준으로 현재 사용자, role, permission 요약을 반환합니다.

Gateway

gateway-service의 JWT_ALGORITHM, JWT_SECRET/JWT_PUBLIC_KEY, JWT_ISSUER, JWT_AUDIENCE 검증과 일치해야 합니다.

User CRUD와 상태 관리
Create

POST /api/auth/users는 X-Tenant-Id 기준으로 사용자를 생성합니다. email은 같은 tenant 안에서만 unique이며 password는 hash만 저장합니다.

Read

GET /api/auth/users와 GET /api/auth/users/{userId}는 항상 tenantId 조건을 포함합니다. cross-tenant 조회는 404 AUTH_USER_NOT_FOUND로 실패합니다.

Update

PATCH /api/auth/users/{userId}는 email과 displayName 변경을 지원합니다. email 중복은 tenant 안에서 409 AUTH_USER_EMAIL_CONFLICT로 차단합니다.

Status

PATCH /api/auth/users/{userId}/status는 active, inactive, locked를 지원하며 active가 아닌 상태로 바뀌면 refresh token을 revoke합니다.

Delete

DELETE /api/auth/users/{userId}는 hard delete가 아니라 inactive 전환으로 처리하고 active refresh token을 폐기합니다.

Envelope

list 응답은 data.items, data.page, data.size, data.total만 사용합니다. 사용자 응답에는 passwordHash를 포함하지 않습니다.

Role과 Permission
Permission

POST/GET /api/auth/permissions는 전역 permission catalog를 관리합니다. permission code는 {domain}.{resource}.{action} 형식을 사용합니다.

Role

POST/GET/PATCH /api/auth/roles는 X-Tenant-Id 기준 tenant-scoped role을 관리합니다.

Mapping

PUT /api/auth/roles/{roleId}/permissions는 role의 permission mapping을 교체합니다.

Assignment

POST /api/auth/users/{userId}/roles는 사용자에게 role을 부여하고 optional warehouseId scope를 둘 수 있습니다.

Summary

GET /api/auth/permissions/summary는 BFF 화면 구성용 role/permission 요약을 반환합니다.

Check

POST /api/auth/permissions/check는 도메인 서비스가 업무 수행 전 permission과 warehouse scope를 확인하는 API입니다.

Internal Auth

summary와 check는 X-Internal-Service-Id, X-Internal-Timestamp, X-Internal-Signature 기반 HMAC 인증을 통과해야 합니다.

Caller Scope

summary는 admin-bff-service, user-bff-service만 허용하고, check는 wms-service까지 허용합니다.

Admin Scope

system_admin은 tenant-scoped role API에서 생성할 수 없습니다. system.* permission은 tenant role에 mapping하거나 tenant permission check에서 사용할 수 없습니다.

Tenant Admin

tenant_admin은 tenant 전체 scope role입니다. 사용자에게 부여할 때 warehouseId를 함께 전달하면 403 AUTH_ADMIN_SCOPE_MISMATCH로 실패합니다.

Error Envelope, Audit, Outbox
Error

실패 응답은 success, requestId, timestamp, error.code, error.message 구조로 통일했습니다.

Audit

auth-iam-service는 audit_logs 테이블을 소유하지 않습니다. 사용자/역할/권한 변경은 별도 audit-log-service에 저장 요청을 보냅니다.

Outbox

사용자/permission/role 변경과 outbox_events 저장을 같은 DB transaction 안에서 처리합니다. Relay 발행은 이후 outbox-relay-service가 담당합니다.

Secret

password, token, Authorization, private key, SQL, stack trace는 응답과 로그에 남기지 않습니다.

테스트 시나리오

1차 검증은 자동 E2E 코드가 아니라 시나리오 기반 AI 검증으로 진행합니다. 성공 판정에는 응답 JSON, access log, JWT payload, refresh token row, permission row, audit/outbox row 중 필요한 증거를 남깁니다.

2026-06-09 Password Hash와 Login API 구현

  • PasswordHasher를 추가해 scrypt, random salt, timing-safe 비교 방식으로 password hash를 처리합니다.
  • POST /api/auth/login을 구현했고 body/header tenant mismatch는 403 TENANT_MISMATCH로 차단합니다.
  • 성공 응답은 표준 envelope에 accessToken, expiresIn, refreshToken, refreshExpiresIn, tokenType을 담아 200으로 반환합니다.
  • access token은 HS256 또는 RS256 설정을 지원하며 JWT payload에는 sub, tenantId, type, iat, exp, iss, aud만 포함합니다.
  • refresh token은 opaque token으로 반환하고 DB에는 SHA-256 hash만 저장합니다. rotation과 revoke API는 별도 단계에서 이어서 구현했습니다.
  • pnpm --filter auth-iam-service typecheck와 pnpm --filter auth-iam-service build가 통과했습니다.
  • 로컬 DB/Redis 연결 상태에서 로그인 성공, JWT claim 확인, refresh token hash 저장, 잘못된 password의 401 AUTH_INVALID_CREDENTIALS 응답을 확인했습니다.

2026-06-09 Refresh Token Rotation과 Revoke 구현

  • POST /api/auth/token/refresh를 추가해 기존 refresh token을 revokedAt과 replacedBy로 폐기하고 새 access/refresh token을 발급합니다.
  • 존재하지 않거나 만료되었거나 이미 폐기된 refresh token, 비활성 사용자 token은 401 AUTH_INVALID_TOKEN으로 실패합니다.
  • POST /api/auth/logout을 추가해 refresh token 단건 revoke와 X-Tenant-Id, X-User-Id 기반 전체 active refresh token revoke를 지원합니다.
  • refresh/logout 요청에서 body tenant와 X-Tenant-Id가 다르면 403 TENANT_MISMATCH로 차단합니다.
  • pnpm --filter auth-iam-service typecheck와 pnpm --filter auth-iam-service build가 통과했습니다.
  • 로컬 DB/Redis 연결 상태에서 login, refresh rotation, 이전 refresh token 재사용 실패, logout revoke, 폐기된 refresh token 재사용 실패를 확인했습니다.
  • DB row 확인 결과 첫 refresh token은 replacedBy로 새 token을 가리키고, logout 대상 token은 revokedAt이 설정된 것을 확인했습니다.

  • 첫 인프라 배포에서 비어 있는 ECR 때문에 task가 실패하지 않도록 authDesiredCount=0을 기본값으로 사용합니다.
  • 서비스 실행은 이미지 push와 runtime secret 준비 후 authDesiredCount=1, authImageTag=...로 재배포해 시작합니다.
  • Auth/IAM은 ALB target으로 직접 노출하지 않고 Cloud Map 내부 DNS와 auth base path를 포함한 auth-iam-service.{envName}.multi-tenant.local:3000/api/auth로 gateway-service가 호출합니다.
  • DATABASE_URL, JWT secret/key, 내부 호출 인증 secret은 Secrets Manager ARN context로 받아 ECS task secret에 주입합니다.
  • security group은 gateway-service에서 Auth/IAM 3000 포트 접근만 허용하고, Auth/IAM에서 Redis 6379 포트 접근만 허용합니다.
  • deploy-gateway-service.yml도 Auth/IAM task definition context를 함께 갱신하므로 AUTH_DESIRED_COUNT와 AUTH_IMAGE_TAG를 명시해야 합니다. Auth/IAM image build/push와 실행 배포는 수동 실행 전용 deploy-auth-iam-service.yml에서 처리합니다.
  • deploy-auth-iam-service.yml은 gateway/tenant/admin-bff desired count와 image tag가 비어 있으면 실패해, 같은 스택의 다른 서비스를 암묵적으로 0대로 낮추지 않습니다.
  • pnpm --filter @multi-tenant/infra-cdk typecheck와 pnpm --filter @multi-tenant/infra-cdk exec cdk synth -c gatewayDesiredCount=0 -c authDesiredCount=0 -c tenantDesiredCount=0 -c adminBffDesiredCount=0 -c gatewayUseNatGateway=false가 통과했습니다.
