Development Process

tenant-service 개발과정

tenant-service는 테넌트 기본 정보, 상태, 사용 모듈, 도메인, DB 전략을 관리합니다. 1차 목표는 gateway-service가 tenant 상태를 확인하고, 이후 WMS와 BFF가 tenant context를 안정적으로 사용할 수 있게 만드는 것입니다.

서비스 역할

위치apps/tenant-service
책임테넌트 생성, 상태 관리, 활성 module 관리, domain 관리, tenant 설정 조회
초기 연동gateway-service가 내부 API로 tenant 상태를 조회하고, 이후 admin-bff-service와 user-bff-service가 tenant context를 조합합니다.
핵심 엔티티tenants, tenant_domains, tenant_modules, tenant_settings, outbox_events
Tenant IDtenantId는 UUID 기반 내부 불변 식별자입니다. 사람이 읽는 운영 식별자는 tenant.code로 분리합니다.
초기 DB 전략Shared DB + tenantId 전략만 구현합니다. schema/database 분리는 later로 둡니다.
주의tenant-service는 WMS 재고/입출고 업무 규칙을 소유하지 않습니다. tenant 상태와 설정 경계만 소유합니다.

1차 개발 목표

Tenant Status

gateway-service가 업무 API 진입 전에 tenant 상태를 확인할 수 있도록 내부 상태 API를 제공합니다.

Tenant Identity

tenantId는 UUID로 생성하고 JWT, header, DB 접근 조건에 사용합니다. 화면과 운영자 입력에는 별도 tenant.code를 사용합니다.

Tenant Admin

운영자가 사용하는 외부 API는 admin-bff-service가 제공하고, tenant-service는 BFF가 호출하는 내부 관리 API를 제공합니다.

Module Scope

초기 module은 wms를 중심으로 관리하고, 이후 MES/QMS/FMS/EMS 확장을 준비합니다.

Outbox

tenant 생성, 상태 변경, module 변경은 outbox event로 남겨 이후 outbox-relay-service와 연결합니다.

개발 순서

1
NestJS 기본 서비스 구성

apps/tenant-service를 실제 NestJS 앱으로 전환하고 config, health, readiness를 먼저 만듭니다.

2
Tenant 데이터 모델 설계

tenants, tenant_domains, tenant_modules, tenant_settings, outbox_events 모델을 정의합니다.

3
Gateway 연동 API 구현

GET /internal/tenants/{tenantId}/status를 먼저 구현해 gateway tenant status check와 연결합니다.

4
내부 관리 API 구현

admin-bff-service가 호출할 tenant 생성, 목록, 상세, 상태 변경, module 변경 내부 API를 구현합니다.

5
Event와 Outbox 추가

tenant 관련 상태 변경을 outbox event로 저장하고, 이후 outbox-relay-service에서 발행할 수 있게 준비합니다.

6
Docker와 Local 연동

local compose에 tenant-service를 추가하고 gateway의 TENANT_SERVICE_URL을 실제 tenant-service로 연결합니다.

Tenant 기능 TODO

tenant-service 기능은 이 체크리스트를 기준으로 구현 내역에 맞춰 갱신합니다.

Tenant Code 생성 규칙

목적tenant.code는 운영자와 화면에서 읽기 쉬운 tenant 식별자이며, 내부 식별자인 tenantId와 분리합니다.
형식대문자 base 4자리와 sequence 4자리를 붙인 8자리 문자열입니다. 구분자는 사용하지 않습니다.
Domain 우선생성 요청에 domain이 있으면 등록 도메인의 이름 label을 우선 사용합니다. 예: acme.example.com은 EXAM0001, factory.acme.co.kr은 ACME0001
Name fallbackdomain이 없으면 name을 대문자 ASCII로 정규화해 base를 만듭니다. 예: ACME 한국은 ACME0001
한글 fallbackASCII label이 없고 한글이 있으면 첫 한글 음절을 로마자로 변환해 base로 사용합니다. 4자리가 모자라면 hash로 채웁니다. 예: 한국공장은 HANF0001
Hash fallbackASCII label도 한글도 없으면 SHA-256 hash의 가운데 4글자를 base로 사용합니다.
충돌 처리동일 base가 이미 있으면 sequence를 0002, 0003처럼 증가시켜 unique code를 생성합니다.

추천 API 초안

외부 관리자 API는 admin-bff-service의 /api/admin/** surface가 소유합니다. tenant-service는 gateway와 BFF가 호출하는 내부 API를 우선 구현합니다.

BFF 외부 API 경계

POST /api/admin/tenants, GET /api/admin/tenants 같은 외부 관리자 API는 admin-bff-service API 문서에서 관리합니다. tenant-service API 문서는 내부 계약을 중심으로 작성합니다.

상태와 이벤트

Tenant Status

provisioning, active, suspended, deleted를 기본 상태로 사용합니다.

Gateway 허용 기준

업무 API는 active tenant만 허용합니다. 그 외 상태는 403 TENANT_NOT_READY 또는 적절한 tenant error code로 차단합니다.

1차 이벤트

tenant.created, tenant.status.changed, tenant.module.enabled, tenant.module.disabled

Event Payload

event에는 eventId, eventType, schemaVersion, tenantId, requestId, occurredAt, source, data를 포함합니다.

1차에서 미룰 것

VIP DB 자동화

tenant별 DB 생성, schema 생성, migration 자동화는 WMS 이후 별도 단계에서 결정합니다.

Custom Domain 인증서

ACM 인증서 자동 발급과 Route53 자동 연결은 Route53 hosted zone 도입 후 처리합니다.

Billing과 Plan

요금제, 계약, quota, 사용량 집계는 초기 WMS 흐름 이후 분리합니다.

Tenant Proxy Access

대리 접속은 승인, 감사 로그, 권한 제한 규칙이 확정되기 전까지 제공하지 않습니다.

Feature Flag 고도화

초기에는 module 활성화 정도만 관리하고, tenant별 세부 feature flag는 이후로 미룹니다.

운영자 UI

초기에는 내부 API와 문서 중심으로 검증하고, web-admin 화면은 admin-bff-service 안정화 후 연결합니다.

테스트 시나리오 초안

TS-TENANT-001active tenant는 gateway tenant status check를 통과한다.
TS-TENANT-002suspended tenant는 gateway에서 업무 API 접근이 차단된다.
TS-TENANT-003deleted tenant는 gateway에서 업무 API 접근이 차단된다.
TS-TENANT-004tenant 생성 시 기본 module wms 활성화 여부가 기대대로 저장된다.
TS-TENANT-005tenant 상태 변경 시 outbox event가 같은 transaction에서 저장된다.
TS-TENANT-006tenant domain 중복 등록은 TENANT_DOMAIN_CONFLICT로 실패한다.
TS-TENANT-007tenant 생성 시 domain/name/한글 fallback 규칙에 맞춰 8자리 대문자 tenant.code가 생성되고, 충돌 시 sequence가 증가한다.

2026-06-10 서비스별 Database 분리

  • 로컬 PostgreSQL 인스턴스는 공유하되 서비스 저장소는 database 단위로 분리하기로 정리했습니다.
  • tenant-service는 tenant database를 사용하고, auth-iam-service는 auth_iam database를 사용합니다.
  • 테넌트별 저장 전략인 Shared DB + tenantId는 각 서비스 database 내부의 tenant-owned data 격리 기준이며, 서비스 간 database 공유를 의미하지 않습니다.
  • 로컬 compose에 service-databases-init job을 추가해 기존 PostgreSQL volume에서도 서비스별 database를 idempotent하게 생성합니다.

2026-06-11 Tenant ID 생성 규칙 정리

  • tenantId는 시스템 내부 불변 식별자이며 UUID로 생성합니다.
  • 회사명, 사업자번호, domain, 사람이 읽는 code는 tenantId로 사용하지 않습니다.
  • 운영자와 화면에서 읽기 쉬운 식별자는 별도 unique tenant.code로 관리합니다.
  • JWT claim, X-Tenant-Id, tenant-scoped DB 접근 조건에는 tenantId를 사용합니다.

2026-06-11 Tenant Code 자동 생성 구현

  • POST /internal/admin/tenants를 추가해 admin-bff-service가 tenant 생성을 요청할 수 있게 했습니다.
  • tenant.code 형식은 대문자 base 4자리와 sequence 4자리를 붙인 8자리 문자열로 통일했습니다.
  • 요청 body에 domain이 있으면 등록 도메인의 이름 label을 기반으로 tenant.code를 생성하고, domain이 없으면 name 기반으로 fallback합니다.
  • domain은 ASCII hostname으로 정규화해 저장하며, 이미 등록된 domain은 409 TENANT_DOMAIN_CONFLICT로 차단합니다.
  • ASCII 정규화 결과가 비어 있고 한글이 있으면 첫 한글 음절을 로마자로 변환해 base로 사용하며, 4자리가 모자라면 hash로 채웁니다. 한글도 없으면 SHA-256 hash의 가운데 4글자를 base로 사용합니다.
  • 예: acme.example.com은 EXAM0001, factory.acme.co.kr은 ACME0001, 한국공장은 HANF0001, ACME 한국은 ACME0001입니다.
  • 동일 base가 이미 있으면 0002, 0003처럼 sequence를 증가시켜 unique code를 생성합니다.
  • tenant 생성과 tenant.created outbox event 저장을 같은 DB transaction에서 처리합니다.
  • Tenant Code 생성 규칙 섹션과 테스트 시나리오 TS-TENANT-007을 추가해 구현 규칙과 검증 기준을 문서화했습니다.
  • 검증: pnpm --filter tenant-service typecheck, pnpm --filter tenant-service build를 실행해 통과했습니다.

2026-06-10 내부 Tenant 조회 API 구현

  • GET /internal/tenants/{tenantId}/status를 추가했습니다. gateway-service만 HMAC internal auth로 호출할 수 있으며, 응답은 tenant id, code, name, status를 포함합니다.
  • GET /internal/tenants/{tenantId}/modules를 추가했습니다. gateway-service, admin-bff-service, user-bff-service, wms-service가 호출할 수 있으며, 응답은 tenant status와 정렬된 enabled module code 목록을 포함합니다.
  • 잘못된 tenant id는 400 TENANT_INVALID_ID, 존재하지 않는 tenant는 404 TENANT_NOT_FOUND로 응답합니다.
  • 응답 envelope는 success, requestId, timestamp, data 구조를 유지합니다.
  • 검증: pnpm --filter tenant-service typecheck와 pnpm --filter tenant-service build를 순차 실행해 통과했습니다.

2026-06-10 AWS 배포 준비

  • CDK 스택에 tenant-service용 ECR repository, ECS Fargate task/service, CloudWatch log group, security group, Cloud Map service discovery를 추가했습니다.
  • tenant-service는 ALB에 직접 노출하지 않고 gateway-service가 http://tenant-service.{env}.multi-tenant.local:3000 내부 DNS로 호출합니다.
  • ECS task에는 DATABASE_URL과 TENANT_INTERNAL_AUTH_SECRET을 Secrets Manager ARN context로 주입하고, REDIS_URL은 같은 스택의 ElastiCache Redis endpoint를 사용합니다.
  • 초기 배포 실패와 비용 발생을 막기 위해 기본 tenantDesiredCount는 0이며, 이미지 push와 runtime secret 준비 후에만 1 이상으로 올립니다.
  • 자동 push 배포 없이 수동 실행 전용 deploy-tenant-service.yml을 추가했습니다. 이 workflow는 gateway/auth desired count와 image tag가 비어 있으면 실패해, 같은 스택의 다른 서비스를 암묵적으로 0대로 낮추지 않습니다.
