Development Process

web-admin 개발과정

운영자와 관리자를 위한 React + Vite 프론트엔드 앱입니다.

서비스/앱 역할

위치apps/web-admin
카테고리Frontend App · Admin UI
대상시스템 운영자, 테넌트 관리자
연결브라우저는 gateway-service의 /api/admin/**만 호출하고, gateway가 admin-bff-service로 proxy합니다.
초기 화면관리 대시보드, 고객사 관리(시스템 관리자 전용), 사용자 관리(조회/추가), 역할/권한, 감사 로그, 위험 작업

기능 정의

관리자 App Shell로그인 후 관리자 navigation, tenant scope, 사용자 context, route guard를 제공합니다.
공통 화면 디자인관리 대시보드와 같은 흰 패널, 얇은 라인, 파란 포인트, 8px radius 기준으로 고객사 관리, 사용자 관리, 역할/권한, 감사 로그, 위험 작업 화면을 통일합니다.
Tenant 관리 화면고객사 목록, 통합 검색, 상태 필터, 모듈 필터, 상단 버튼 기반 고객사 생성 모달, 행 선택 기반 상세/수정 모달을 Admin BFF API와 연결합니다. 서비스 플랜 필드는 사용하지 않고 활성 module 정보로 표시/필터링합니다.
사용자 관리 화면시스템 내 사용자 목록 조회와 사용자 추가를 제공합니다. 이름/이메일 통합 검색, 테넌트, 역할, 상태 필터는 Admin BFF와 Auth/IAM query에서 처리하고, 목록에는 소속 테넌트, 역할, 상태, 마지막 로그인 시간을 표시합니다.
감사 로그 화면기간, actor, action, resource, result, requestId 조건 검색과 상세 evidence 표시를 제공합니다.
권한 기반 UIpermission summary를 기준으로 메뉴, 버튼, 변경성 action 노출 여부를 제어합니다.
제외 범위도메인 검증, token 발급, 직접 DB 접근, BFF를 우회한 내부 서비스 호출은 하지 않습니다.

구현 상태

상태Desktop 1차 관리자 조회/운영 화면 완료
선행 조건admin-bff-service의 tenant/user/role/permission/audit 조회 API가 준비되어 Desktop 1차 화면 연동 기준으로 사용합니다.
로컬 Dockerdocker/local/docker-compose.yml의 web-admin 서비스가 http://localhost:5173에서 nginx로 정적 파일을 제공하고 API base URL은 http://localhost:3000 gateway로 빌드합니다.
API 문서화면 연동 기준은 docs/apis/admin-bff-service/index.html를 따릅니다.

기술 스펙 결정

web-admin은 운영자가 테넌트와 권한을 다루는 관리 콘솔이므로, web-wms와 같은 프론트엔드 기반을 사용하되 위험 작업 확인, requestId evidence, 권한 기반 UI를 더 강하게 드러냅니다.

영역 선택 이유
런타임React + Vite + TypeScript프로젝트 공통 규칙과 web-wms 구현 방식에 맞춰 관리 화면도 같은 개발/빌드 경험을 유지합니다.
라우팅React Router관리 대시보드, 고객사 관리, 사용자 관리, 역할/권한, 감사 로그, 위험 작업 화면을 URL과 route guard로 나눕니다. 고객사 관리는 시스템 관리자 navigation에만 포함합니다.
서버 상태TanStack QueryAdmin BFF 조회 API의 loading, error, retry, cache invalidation을 화면 단위로 일관되게 처리합니다.
API 클라이언트fetch 기반 래퍼 + zod envelope 검증관리 API의 success, requestId, timestamp, error.code 규칙을 한 곳에서 처리하고 evidence 표시 기준을 유지합니다.
스타일CSS Modules + global tokens아직 공통 UI 패키지가 없으므로 앱 내부에서 스타일을 격리하고, web-wms와 같은 카드/테이블/패널 패턴을 공유하기 쉽게 유지합니다.
아이콘lucide-react관리 메뉴, 권한, 감사, 위험 작업을 익숙한 아이콘으로 구분합니다.
검증TypeScript, build, 시나리오 기반 AI 검증관리 화면은 권한과 감사 evidence가 중요하므로 E2E 자동화 추가 전 시나리오별 화면 상태와 requestId 확인을 우선합니다.

기능별 작업 내역

Tenant Admin고객사 목록, 통합 검색(keyword), 상태, 활성 모듈(moduleCode) 필터, 고객사 생성 모달, 고객사 상세/수정 모달을 Admin BFF API와 연결합니다. 생성 모달은 여러 활성 모듈을 선택하고 생성 직후 module 교체 API로 반영합니다.
User Admin사용자 관리는 목록 조회와 사용자 추가를 제공합니다. GET /api/admin/access-control/screen-data로 사용자 목록을 조회하고 keyword, tenantFilter, roleCode, userStatus 필터를 백엔드 query로 전달합니다. 상단 사용자 추가 버튼은 현재 관리자 scope에 맞춰 tenant를 지정하고 POST /api/admin/users, 선택 시 POST /api/admin/users/{userId}/roles를 호출합니다.
Access Control역할/권한 화면은 BFF의 screen-data 조합 API를 사용하고, role permission 전체 교체 같은 변경성 요청은 API query와 idempotency 기준을 따릅니다. 권한 chip과 안내 panel은 파란 포인트 계열로 통일했습니다.
Auditaudit-log-service 조회 API, action/resourceType/requestId 검색, 변경성 action audit evidence를 연결했습니다. 감사 로그 table과 검색 bar는 대시보드 스타일의 흰 panel, 얇은 line, compact row 기준으로 표시합니다.
Risk Actions위험 작업 화면은 대시보드 스타일의 흰 카드, 파란 아이콘/버튼, 회색 설명문 기준으로 표시합니다.

개발 기록

2026-06-25 고객사 상세 모달 버튼 위치 조정

  • 고객사 상세 조회 모달의 조회 모드 수정하기 버튼을 footer에서 header 우측 action 영역으로 이동했습니다.
  • 조회 모드 footer는 닫기만 표시하고, 수정 모드 footer는 취소/저장하기만 표시하도록 정리했습니다. 모바일에서는 footer 버튼이 하나일 때 전체 폭을 사용하도록 반응형 grid를 보강했습니다.
  • 검증: pnpm --filter web-admin typecheck 기준으로 JSX와 CSS module 참조가 정상임을 확인했습니다.

2026-06-25 사용자 상세 모달 수정 모드 추가

  • 사용자 상세 모달에 조회 모드/수정 모드를 추가하고, 조회 모드의 수정하기 버튼을 header 우측 action 영역에 배치했습니다.
  • 수정 가능한 필드는 사용자 이름, 이메일, 상태입니다. 사용자 ID, 소속 테넌트, 사용자 유형, 역할, 등록/수정/마지막 로그인 일자는 읽기 전용으로 유지합니다.
  • 저장 시 이름/이메일은 PATCH /api/admin/users/{userId}, 상태 변경이 있으면 PATCH /api/admin/users/{userId}/status를 호출합니다. 성공 후 사용자 목록, 사용자 요약, 대시보드, 감사 로그 cache를 갱신합니다.
  • 수정 모드는 서버 원본과 분리된 draft 값을 사용하며, 취소를 누르면 입력 중인 값은 버리고 조회 모드 원본값으로 복귀합니다.
  • 검증: pnpm --filter web-admin typecheck, Docker 로컬 web-admin 재빌드 및 재기동을 완료했고, multi-tenant-web-admin 컨테이너가 healthy 상태임을 확인했습니다.
  • 저장 버튼 클릭 시 시스템 관리자가 테넌트 사용자를 수정하는 요청에 X-Tenant-Id가 빠져 400 TENANT_REQUIRED가 발생하던 문제를 수정했습니다. 사용자 기본정보 수정과 상태 변경 요청 모두 선택 사용자 tenantId를 전달하며, gateway 검증 requestId=verify-user-update-with-tenant에서 200 성공을 확인했습니다.
  • 테넌트가 없는 system_admin 계정은 일반 테넌트 사용자 수정 API 대상이 아니므로 상세 모달에서 수정 모드 진입을 비활성화했습니다. 화면에는 별도 시스템 관리자 관리 흐름에서 처리해야 한다는 안내를 표시합니다.
  • 현재 로그인한 슈퍼 관리자는 자기 자신의 이름과 이메일을 수정할 수 있도록 예외를 추가했습니다. 상태와 역할은 전역 관리자 관리 흐름 대상이므로 사용자 상세 모달에서는 읽기 전용으로 유지합니다. gateway 검증 requestId=verify-super-self-update에서 tenant 없는 system_admin 본인 수정이 200 성공했습니다.

2026-06-25 사용자 상세 조회 모달 추가

  • 사용자 관리 목록 행을 고객사 관리 목록과 같은 클릭 가능한 행 패턴으로 변경했습니다. 마우스 클릭과 키보드 Enter/Space로 사용자 상세 모달을 열 수 있습니다.
  • 상세 모달은 고객사 상세 모달과 같은 customerDetailModal 스타일을 사용하며 조회 모드 전용입니다. 표시 항목은 사용자 이름, 사용자 ID, 이메일, 소속 테넌트, 사용자 유형, 역할, 등록 일자, 수정 일자, 마지막 로그인입니다.
  • 모달은 헤더/푸터를 유지하고 본문만 스크롤되는 기존 상세 모달 구조를 재사용합니다. footer에는 닫기 버튼만 제공해 사용자 조회 화면의 읽기 전용 정책과 맞췄습니다.
  • 검증: pnpm --filter web-admin typecheck, Docker 로컬 web-admin 재배포, 브라우저에서 /access-control 목록 5행 표시와 첫 행 클릭 시 사용자 상세 정보/조회 모드 모달 표시를 확인했습니다. 사용자 ID/이메일/닫기 버튼이 표시되고 모달이 viewport 안에 들어오며 console error는 0건입니다.

2026-06-24 사용자 관리 화면 개편과 추가 버튼 연결

  • 사용자 관리 화면을 대시보드 스타일로 재구성했습니다. 상단 지표 카드, 통합 검색, 테넌트/역할/상태 필터, 사용자 목록 table, pagination을 제공하고 고객사 관리의 고객사 추가 버튼과 같은 위치에 사용자 추가 버튼을 배치했습니다.
  • Admin BFF GET /api/admin/access-control/screen-data는 keyword, tenantFilter, roleCode, userStatus query를 Auth/IAM 사용자 목록 API로 전달합니다. 시스템 관리자는 전체 테넌트 사용자를 조회할 수 있고, 테넌트 관리자는 본인 tenant scope로 제한됩니다.
  • Auth/IAM 사용자 목록 응답에 roleCodes와 lastLoginAt을 추가해 화면에서 역할 badge와 마지막 로그인 시간을 표시할 수 있게 했습니다. keyword는 이름/이메일 부분 검색, tenantFilter는 시스템 관리자용 tenant 필터, roleCode는 사용자 역할 기준 필터입니다.
