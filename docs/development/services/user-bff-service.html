Development Process

user-bff-service 개발과정

WMS 사용자 화면과 PDA Web에 필요한 데이터를 조합하는 사용자용 BFF입니다.

서비스/앱 역할

위치apps/user-bff-service
카테고리Backend Service · BFF
책임사용자 정보, 현재 테넌트, 사용 가능 메뉴, WMS 화면 초기 데이터, 대시보드 응답 조합
제작 시점2026-06-18에 app shell과 navigation 기본 구조를 구현했고, 2026-06-19에 프론트 제작용 WMS 화면 조회 API를 추가했습니다.
Tenant 연동업무 API 처리 전 tenant-service의 GET /internal/tenants/{tenantId}/status로 active 상태를 확인하고, 화면 구성에는 module/context API를 사용합니다.
Audit 연동User BFF가 소유한 app shell 이벤트는 USER_BFF_APP_AUDIT_PUBLISHER_TYPE 설정에 따라 EventBridge 또는 기존 audit-log-service 내부 기록 API로 best-effort 저장합니다. 감사 저장 실패는 사용자 응답을 차단하지 않고 warning log로 남깁니다.
주의재고 차감, 입고 완료, 출고 할당 정책 같은 도메인 업무 로직은 WMS에 둡니다. BFF는 직접 DB에 접근하지 않습니다.

기능 정의

사용자 Context/api/app/me에서 사용자, tenant, 활성 module, permission summary를 조합해 제공합니다.
Navigationtenant module과 Auth/IAM permission summary를 기준으로 WMS 사용자 메뉴와 버튼 노출 기준을 구성합니다.
WMS 화면 APIdashboard, warehouse selector, inventory summary 등 화면 중심 응답을 wms-service 조회 API로부터 조합합니다.
Tenant 보호업무 API 처리 전 tenant-service 상태 확인을 수행하고 requestId, tenantId, userId를 downstream으로 전파합니다.
감사 로그/api/app/me, /api/app/navigation 성공 시 userBff.appContext.loaded, userBff.navigation.loaded 이벤트를 생성합니다. EventBridge bus가 설정된 환경은 EventBridge로 발행하고, 기존 local/호환 환경은 내부 기록 API로 저장합니다.
장애 처리downstream 장애를 표준 error envelope로 변환하고 사용자 화면에서 처리 가능한 오류로 정리합니다.
제외 범위WMS 재고 계산, 입출고 상태 전이, 권한 원장 관리, 직접 DB 접근은 담당하지 않습니다.

개발 순서

1
NestJS 서비스 골격

apps/user-bff-service를 admin-bff-service 패턴과 맞춰 config, health, readiness, context, access log, error filter, security headers부터 만듭니다.

2
Downstream client와 Guard

Auth/IAM, tenant-service, WMS internal client를 만들고 tenant active check와 app permission check를 전역 guard로 연결합니다.

3
App Context API

/api/app/me, /api/app/navigation을 먼저 구현해 web-wms app shell이 사용자, tenant, module, permission context를 받을 수 있게 합니다.

4
WMS 화면 조합 API

WMS 내부 조회 API를 사용해 warehouse selector, dashboard, inventory summary를 조합합니다. BFF에는 재고 계산과 입출고 상태 전이를 넣지 않습니다.

5
Gateway와 Local Compose 연결

/api/app/** proxy, USER_BFF_SERVICE_URL, route group rate limit, compose service와 포트를 추가합니다.

6
문서와 시나리오 검증

API 문서, 개발 문서, gateway 문서, 테스트 시나리오를 갱신하고 로컬 전체 시스템 검증에 User BFF 항목을 추가합니다.

Tenant Boundary와 Downstream 전파
Tenant업무 API 처리 전 tenant-service의 status API로 active tenant만 통과시키고 module 목록에 wms가 있는지 확인합니다.
전파Auth/IAM, tenant-service, wms-service 내부 client는 X-Request-Id, X-Tenant-Id, X-User-Id와 internal auth header를 downstream으로 전파합니다.
장애downstream timeout/unavailable은 표준 error envelope로 변환하고 원본 secret, SQL, stack trace를 응답에 포함하지 않습니다.
검증응답 JSON, access log, downstream log, requestId, tenantId propagation을 evidence로 남깁니다.

2026-06-19 App Audit EventBridge 전환

  • AppAuditService가 audit-log-service 내부 POST client를 직접 호출하지 않고 AppAuditPublisher를 통해 감사 이벤트를 발행하도록 분리했습니다.
  • USER_BFF_APP_AUDIT_PUBLISHER_TYPE은 eventbridge, internal-api, disabled를 지원합니다. 값이 없으면 USER_BFF_AUDIT_EVENTBRIDGE_BUS_NAME이 있을 때 EventBridge, 없을 때 기존 내부 API를 사용합니다.
  • EventBridge 모드는 USER_BFF_AUDIT_EVENTBRIDGE_BUS_NAME, USER_BFF_AUDIT_EVENT_SOURCE_PREFIX, 선택적 USER_BFF_AUDIT_EVENTBRIDGE_ENDPOINT를 사용해 userBff.appContext.loaded, userBff.navigation.loaded를 PutEvents로 발행합니다.
  • CDK에 userBffAppAuditPublisherType, userBffAuditEventBridgeBusName context, User BFF task role의 events:PutEvents grant, EventBridge rule → audit event SQS target 연결을 추가했습니다. 기본 context는 기존 배포 호환을 위해 internal-api로 유지합니다.
  • 전체 시스템 테스트에 OBS-AUDIT-003을 추가해 EventBridge 전환 설정의 CDK synth, detail-type routing, audit SQS target, events:PutEvents 권한을 함께 검증합니다.
  • pnpm --filter user-bff-service typecheck, pnpm --filter @multi-tenant/infra-cdk typecheck로 설정 타입과 CDK 합성을 위한 타입 계약을 검증했습니다.

2026-06-11 BFF 경계 최신화

  • user-bff-service는 사용자 화면 조합 API만 담당하고, tenant module 활성화 상태는 tenant-service 내부 API로 조회하는 기준을 정리했습니다.
  • 권한 summary는 auth-iam-service, WMS 업무 데이터는 wms-service에서 가져오며 BFF에는 도메인 업무 규칙을 두지 않습니다.
  • 제작 시점은 WMS 1차 도메인 조회 API 이후로 정리했습니다. WMS 업무 API가 없는 상태에서 User BFF를 먼저 만들면 mock 응답이 많아져 계약이 흔들릴 수 있습니다.

2026-06-12 착수 조건 점검

  • Auth/IAM permission summary/check, tenant status/module 조회 조건은 준비된 상태로 체크했습니다.
  • gateway /api/app/** proxy 조건은 실행 가능한 user-bff-service 구현 시 함께 추가하는 남은 준비 항목으로 되돌렸습니다.
  • 남은 착수 조건은 첫 web-wms 화면 데이터 형태 정리입니다. WMS 1차 내부 API는 구현과 로컬 연계 검증이 완료되었습니다.
  • gateway-service는 user-bff-service까지만 호출하고, user-bff-service가 tenant-service와 WMS 같은 downstream service를 호출하는 구조를 기준으로 유지합니다.

2026-06-12 전역 BFF 경유 원칙 확정

  • 사용자용 업무 API는 /api/app/**로 받고, gateway-service는 user-bff-service까지만 proxy합니다.
  • User BFF는 업무 API 처리 전 tenant-service 내부 status API로 active tenant를 확인합니다. WMS 화면 API가 추가될 때 wms-service downstream 호출을 연결합니다.
  • 이후 WMS, PDA, 현장 작업 API가 늘어나도 gateway-service에서 domain service로 직접 route를 만들지 않고 User BFF 계약을 확장합니다.
