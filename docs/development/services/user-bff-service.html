WMS 사용자 화면과 PDA Web에 필요한 데이터를 조합하는 사용자용 BFF입니다.

서비스/앱 역할

위치apps/user-bff-service
카테고리Backend Service · BFF
책임사용자 정보, 현재 테넌트, 사용 가능 메뉴, WMS 화면 초기 데이터, 대시보드 응답 조합
제작 시점wms-service 1차 도메인 조회 API가 잡힌 뒤, web-wms 화면 연결 직전에 시작
Tenant 연동화면 구성에는 tenant-service의 GET /internal/tenants/{tenantId}/modules와 이후 context API를 사용합니다.
주의재고 차감, 입고 완료, 출고 할당 정책 같은 도메인 업무 로직은 WMS에 둡니다. BFF는 직접 DB에 접근하지 않습니다.

기능별 작업 내역

App Context/api/app/me와 navigation API는 Auth/IAM permission summary와 tenant module 정보를 조합합니다.
WMS DashboardWMS 요약 지표, 입출고 대기, 작업 현황은 wms-service 조회 API를 통해 가져옵니다.
Tenant BoundaryBFF는 JWT/header tenant context를 전파하고, WMS 업무 규칙과 tenant business rule은 소유하지 않습니다.
API ClientAuth/IAM, tenant-service, wms-service 내부 client는 requestId, tenantId, userId를 downstream으로 전파합니다.

테스트 시나리오

USER-BFF-001활성 tenant 사용자는 /api/app/me에서 사용자, tenant, permission summary를 받을 수 있다.
USER-BFF-002비활성 module 또는 권한이 없는 메뉴는 navigation 응답에서 제외된다.
USER-BFF-003WMS dashboard API는 downstream 장애 시 표준 error envelope와 requestId를 반환한다.

우선순위 작업 범위 완료 기준 함께 갱신할 문서
P0 WMS Contract 확정 wms-service 1차 API 계약이 확정되면 dashboard와 inventory summary 조합 규칙을 구체화합니다. User BFF가 mock 응답 없이 실제 WMS 조회 API를 조합하고 tenant/user context를 downstream으로 전파합니다. wms-service 개발 문서, user-bff-service API 문서, web-wms 개발 문서

P1 권한 summary와 메뉴 표시 기준 Auth/IAM permission summary를 사용해 화면별 메뉴, 버튼, action availability를 조합합니다. JWT에 roles/permissions를 넣지 않고 BFF 응답으로 화면 권한 표시를 제어합니다. Auth/IAM 개발 문서, user-bff-service 개발 문서, auth 규칙

개발 기록

2026-06-11 BFF 경계 최신화

  • user-bff-service는 사용자 화면 조합 API만 담당하고, tenant module 활성화 상태는 tenant-service 내부 API로 조회하는 기준을 정리했습니다.
  • 권한 summary는 auth-iam-service, WMS 업무 데이터는 wms-service에서 가져오며 BFF에는 도메인 업무 규칙을 두지 않습니다.
  • 제작 시점은 WMS 1차 도메인 조회 API 이후로 정리했습니다. WMS 업무 API가 없는 상태에서 User BFF를 먼저 만들면 mock 응답이 많아져 계약이 흔들릴 수 있습니다.

2026-06-12 착수 조건 점검

  • Auth/IAM permission summary/check, tenant status/module 조회, gateway /api/app/** proxy 조건은 준비된 상태로 체크했습니다.
  • 남은 착수 조건은 WMS 조회 API와 첫 web-wms 화면 데이터 형태 정리입니다.
