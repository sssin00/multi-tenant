Development Process

outbox-relay-service 개발과정

서비스 DB의 Outbox 이벤트를 읽어 외부 이벤트 인프라로 안전하게 전달하고, 발행 상태를 추적하는 이벤트 중계 전담 서비스입니다.

서비스/앱 역할

위치apps/outbox-relay-service
카테고리Backend Service · Event Relay
핵심 역할각 서비스 DB에 저장된 outbox_events를 읽어 EventBridge/SQS 또는 local mock publisher로 발행하고, 성공/실패 상태를 원본 outbox row에 기록합니다.
책임미발행 outbox event 조회, event envelope 검증, publish, retry, 실패 기록, 발행 지연/실패 관측
초기 대상Auth/IAM 사용자·권한 변경 이벤트, tenant 상태·모듈 변경 이벤트, 이후 WMS 재고 변경, 입고 완료, 출고 완료 이벤트
주의도메인 트랜잭션은 각 서비스 DB에서 끝나고, relay는 발행만 담당합니다.

역할 상세

한 문장 정의outbox-relay-service는 서비스 내부 DB와 AWS 이벤트 인프라 사이의 안정적인 배송 담당자입니다.
입력Auth/IAM, tenant-service, 이후 WMS 등 각 서비스 DB의 outbox_events row입니다. 입력 이벤트는 이미 업무 트랜잭션과 함께 저장된 사실이어야 합니다.
처리pending event를 source별로 polling하고, batch claim/lock, envelope validation, publisher 호출, retry/backoff, failed 전환을 수행합니다.
출력local에서는 mock publisher log와 outbox 상태 변경, AWS에서는 EventBridge event 또는 SQS message입니다.
상태 책임발행 성공 시 published, 실패 시 retry count와 실패 사유를 남기고, 최대 재시도 초과 시 failed로 전환합니다.
관측 책임source, eventId, eventType, tenantId, requestId, publish target, duration, result를 JSON log와 metric으로 남깁니다.
하지 않는 일업무 이벤트 생성, 도메인 상태 변경, 다른 서비스 DB의 업무 데이터 수정, consumer 업무 처리, 관리자 화면 제공은 담당하지 않습니다.
경계다른 서비스의 내부 TypeScript source를 import하지 않고, source별 outbox table 계약과 publisher interface만 의존합니다.

기능 정의

Outbox 수집각 서비스 DB의 outbox_events에서 pending event를 batch 단위로 조회하고 lock 기준을 적용합니다.
Event 발행packages/events의 event envelope 필수 필드를 검증한 뒤 EventBridge 또는 SQS로 발행합니다.
상태 전환성공 시 published, 실패 시 retryable/failed 상태와 last error, retry count를 기록합니다.
중복 방지eventId 기반 idempotency와 lock timeout 기준으로 중복 발행 위험을 줄입니다.
운영 관측발행 성공/실패 수, oldest pending age, retry count, DLQ 전환 수를 metric/log로 남깁니다.
제외 범위도메인 트랜잭션 처리, event payload 생성, 다른 서비스 DB의 업무 데이터 수정은 담당하지 않습니다.

구현 계획 요약

1차 목표Auth/IAM과 tenant-service의 outbox_events pending row를 안전하게 가져와 EventBridge 또는 SQS로 발행하고, 성공/실패 상태를 원본 DB에 기록합니다.
서비스 경계다른 서비스의 내부 TypeScript source나 Prisma generated client를 import하지 않습니다. relay는 source별 DB connection과 SQL adapter로 outbox table만 다룹니다.
초기 sourceauth-iam-service, tenant-service. 이후 wms-service가 outbox table을 갖추면 source adapter를 추가합니다.
초기 publisherlocal은 mock publisher로 payload와 상태 전환을 검증하고, AWS dev 이상은 감사 로그 저장을 위해 SQS를 기본 대상으로 둡니다. EventBridge는 여러 consumer fan-out이 필요한 이벤트에 사용합니다.
운영 방식HTTP API 중심 서비스가 아니라 worker 성격입니다. /health, /ready, 내부 운영 조회/replay API만 제공합니다.
성공 기준동일 이벤트가 중복 발행되어도 소비자는 eventId로 멱등 처리하고, relay는 published/failed/retry 상태와 requestId를 추적할 수 있습니다.

개발 순서

1
NestJS 기본 서비스 구성

apps/outbox-relay-service를 실제 NestJS 앱으로 전환하고 config, health, readiness, logger, graceful shutdown을 추가합니다.

2
Outbox schema 보강 기준 확정

Auth/IAM과 tenant-service의 outbox_events에 lock/retry 운영에 필요한 필드를 추가할지 확정합니다. 1차 권장 필드는 lockedAt, lockedBy, nextRetryAt, lastError, publishedTarget입니다.

3
Source adapter 구현

서비스별 DB URL을 받아 outbox_events pending batch를 claim하고, 성공/실패 상태를 원본 DB에 반영하는 SQL adapter를 만듭니다.

4
Publisher abstraction 구현

mock, eventbridge, sqs publisher interface를 분리하고 EventBridge event envelope를 packages/events 타입과 맞춥니다.

5
Relay worker loop 구현

source별 polling interval, batch size, max retry, backoff, concurrency를 config로 제어하고 worker loop에서 반복 처리합니다.

6
운영 API와 metric 추가

내부 운영용 상태 조회, source별 pending/failed count, 수동 replay 후보 조회 API를 추가하고 CloudWatch metric/alarm 기준을 문서화합니다.

7
Docker/CDK/Postman 연결

local compose, deploy compose, CDK ECS service, EventBridge/SQS 리소스, Postman 운영 API 요청을 함께 갱신합니다.

기능별 작업 내역

Pollingsource별 DB에서 pending event를 createdAt 오름차순으로 조회합니다. batch size와 polling interval은 source별로 설정합니다.
Claim여러 worker가 동시에 떠도 같은 event를 처리하지 않도록 lock 기준을 둡니다. 운영 권장은 lockedAt, lockedBy, nextRetryAt 보강 후 claim update를 사용하는 방식입니다.
Publishevent schema 필수 필드 검증 후 EventBridge bus 또는 SQS queue로 발행합니다. local에서는 mock publisher로 상태 전환만 검증할 수 있게 합니다.
Retry일시 장애는 exponential backoff로 재시도하고, 반복 실패는 failed 상태와 lastError로 운영자가 확인 가능한 상태로 분리합니다.
Idempotencyrelay는 eventId를 publisher idempotency key로 사용하고, consumer는 eventId unique key로 중복 소비를 방지합니다.
Observability발행 성공/실패 수, 지연 시간, source별 pending/failed count, oldest pending age를 metric과 JSON log로 남깁니다.

Outbox DB 보강 초안

Auth/IAM과 tenant-service의 outbox_events에는 relay 운영 안정성을 위해 아래 lock/retry 필드를 반영했습니다. 필드는 nullable이라 기존 outbox 저장 코드와 호환됩니다.

필드목적1차 필요도비고
lockedAtworker가 event를 claim한 시각P0lock timeout 이후 재처리 가능
lockedByclaim한 worker instance 식별자P0ECS task id 또는 generated worker id
nextRetryAt다음 재시도 가능 시각P0backoff 구현에 필요
lastError마지막 발행 실패 사유P0secret, token, SQL, stack trace는 저장 금지
publishedTargetEventBridge bus 또는 SQS queue 식별자P1운영 추적용

Audit Relay 준비 계약

감사 로그 저장은 직접 REST 호출이 아니라 outbox event 발행으로 연결합니다. outbox-relay-service가 추가되면 아래 필드를 그대로 EventBridge detail 또는 SQS message body로 발행합니다.

공통 타입packages/events의 DomainEventEnvelope, AuditLogEvent, EventBridgeEventEnvelope, SqsEventEnvelope를 기준으로 합니다.
Outbox 필드eventId, eventType, schemaVersion, tenantId, requestId, source, aggregateType, aggregateId, payload
Payload 필수payload.eventId, payload.eventType, payload.schemaVersion, payload.tenantId, payload.requestId, payload.occurredAt, payload.source, payload.aggregateType, payload.aggregateId, payload.actor, payload.data
Audit 소비audit-log-service의 SqsAuditEventWorker가 AUDIT_EVENT_QUEUE_URL에서 메시지를 polling하고, AuditLogEventConsumer.handleSqsEvent가 SQS Records[].body JSON을 감사 로그로 저장합니다. CDK 기본값은 관리형 audit-events-queue를 outbox SQS publisher와 audit consumer가 함께 사용합니다.
중복 처리consumer는 eventId를 audit_logs.event_id unique key로 저장하고, 같은 이벤트 재전달 시 기존 auditId를 반환합니다.
ID 타입eventId와 tenantId는 UUID를 유지합니다. aggregateId는 도메인별 문자열 ID를 허용하도록 outbox schema에서 UUID 제약을 두지 않습니다.

CloudWatch metric/alarm 기준

1차 구현은 구조화 로그와 운영 API를 우선 제공합니다. dev AWS 연결 후 custom metric은 아래 이름으로 방출하고 alarm을 연결합니다.

MetricDimensions기준권장 alarm
OutboxPublishedCountenv, source, publisher발행 성공 건수참고 metric
OutboxPublishFailedCountenv, source, publisher발행 실패 건수5분 합계 1 이상이면 warning
OutboxFailedEventCountenv, sourcefailed 상태 누적 건수1 이상이면 수동 확인
OutboxOldestPendingAgeSecondsenv, source가장 오래된 pending event age300초 초과 2회 연속이면 warning, 900초 초과면 critical
OutboxClaimedCountenv, sourceworker가 claim한 event 수참고 metric

Alarm 알림 대상은 dev에서는 운영 채널, staging/prod에서는 장애 대응 채널로 분리합니다. metric payload와 log에는 secret, SQL, stack trace를 넣지 않습니다.

테스트 시나리오

OUTBOX-001pending event를 한 번만 claim하고 publisher 성공 후 published 상태와 publishedAt을 기록한다.
OUTBOX-002publisher 장애 시 retry count, last error, nextRetryAt을 기록하고 재시도 가능 상태를 유지한다.
OUTBOX-003동일 event를 중복 polling해도 외부 발행은 idempotent하게 처리한다.
OUTBOX-004max retry를 초과한 event는 failed 상태로 전환되고 운영 조회 API에서 확인된다.
OUTBOX-005EventBridge publisher는 detail-type, source, detail을 event 규칙과 맞게 구성한다.
OUTBOX-006SQS publisher는 message body에 event envelope를 보존하고 FIFO 사용 시 aggregateId 기반 group key를 적용한다.
OUTBOX-007audit-log-service consumer는 relay가 전달한 이벤트를 eventId 기준으로 멱등 저장한다.
OUTBOX-008source DB 하나가 장애여도 다른 source polling은 계속 진행되고 readiness는 degraded 또는 not_ready 기준을 명확히 반환한다.

  • 검증용 auth.local.verification.created event를 auth_iam.outbox_events에 삽입한 뒤 published, published_target=mock, retry_count=0으로 전환되는 것을 확인했습니다.
  • OUTBOX_SQS_ENDPOINT를 추가하고 일반 SQS queue에는 FIFO 전용 MessageGroupId/MessageDeduplicationId를 보내지 않도록 수정했습니다.
  • LocalStack SQS queue와 audit-log-service SQS consumer를 사용해 auth.local.sqsAudit.verified event가 audit_log.audit_logs에 저장되는 것을 확인했습니다.
  • scripts/test/verify-local-outbox-sqs.mjs와 pnpm test:outbox:sqs:local 명령을 추가해 같은 검증을 재실행할 수 있게 했습니다.
  • 기존 로컬 DB에 남아 있던 과거 포맷 pending event 14건은 payload.aggregateType does not match outbox aggregateType 검증 오류로 최대 재시도 후 failed로 전환되었습니다.
  • 이번 로컬 검증 로그는 2026-06-17 Outbox Relay 로컬 검증 로그에 기록했습니다.
  • 후속 보완은 idempotency 반복 검증과 dev AWS SQS/DLQ/IAM/CloudWatch 증적 확보입니다.

2026-06-17 감사 이벤트 SQS 전달 경로 연결

  • 감사 로그 저장 경로를 outbox-relay-service의 SQS publisher와 audit-log-service의 SQS polling consumer로 연결했습니다.
  • CDK는 기본 audit-events-queue와 DLQ를 생성하고, outboxPublisherType=sqs일 때 OUTBOX_SQS_QUEUE_URL을 관리형 audit queue로 주입합니다.
  • 별도 queue를 써야 하는 환경은 outboxSqsQueueUrl context 또는 OUTBOX_SQS_QUEUE_URL GitHub Environment variable로 override합니다.
  • GitHub Actions의 SQS publisher 검증은 queue URL 필수 조건을 제거했습니다. 값이 없으면 CDK 관리형 audit queue를 사용하고, EventBridge publisher만 bus 이름을 필수로 검증합니다.
  • dev AWS 검증은 pending outbox row 발행, SQS message 수신, audit_logs 저장, 실패 시 DLQ redrive 증적을 확보하는 시나리오로 남겼습니다.

2026-06-17 Outbox lock/retry schema 보강

  • Auth/IAM과 tenant-service의 OutboxEvent schema에 lockedAt, lockedBy, nextRetryAt, lastError, publishedTarget nullable 필드를 추가했습니다.
  • pending claim과 retry 조회를 위해 status, nextRetryAt, createdAt 복합 index와 lockedAt index를 추가했습니다.
  • 새 필드는 nullable이므로 기존 업무 transaction의 outbox event 저장 코드는 그대로 호환됩니다.
  • pnpm --dir apps/auth-iam-service prisma:validate, pnpm --dir apps/tenant-service prisma:validate로 schema 유효성을 확인했습니다.

2026-06-17 Source DB URL/secret 정책 확정

  • Outbox Relay가 source별 전용 DB URL secret을 사용하도록 AUTH_IAM_OUTBOX_DATABASE_URL, TENANT_OUTBOX_DATABASE_URL, WMS_OUTBOX_DATABASE_URL 기준을 확정했습니다.
  • GitHub Environment에는 secret 원문이 아니라 AUTH_IAM_OUTBOX_DATABASE_URL_SECRET_ARN, TENANT_OUTBOX_DATABASE_URL_SECRET_ARN, WMS_OUTBOX_DATABASE_URL_SECRET_ARN만 저장하는 정책으로 정리했습니다.
  • CDK context 이름은 authIamOutboxDatabaseUrlSecretArn, tenantOutboxDatabaseUrlSecretArn, wmsOutboxDatabaseUrlSecretArn으로 확정했습니다.
  • OUTBOX_WORKER_ENABLED=true일 때 OUTBOX_SOURCES에 포함된 source별 DB URL secret이 모두 있어야 readiness가 통과하는 기준을 문서화했습니다.
