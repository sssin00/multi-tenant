outbox-relay-service 개발과정

서비스 DB의 Outbox 이벤트를 읽어 EventBridge/SQS로 안정적으로 발행합니다.

서비스/앱 역할

위치apps/outbox-relay-service
카테고리Backend Service · Event Relay
책임미발행 outbox event 조회, publish, retry, 실패 기록
초기 대상WMS 재고 변경, 입고 완료, 출고 완료 이벤트
주의도메인 트랜잭션은 각 서비스 DB에서 끝나고, relay는 발행만 담당합니다.

구현 상태

상태착수 전
선행 조건각 도메인 서비스가 outbox_events에 pending event를 저장하고, relay가 읽을 DB 연결/권한 정책이 확정되어야 합니다.
API 문서내부 운영 API가 생기면 docs/apis/outbox-relay-service/index.html에 기록합니다.

Polling

Publish

EventBridge event bus로 발행

Retry

실패 횟수, DLQ 연계, 수동 replay 기준

기능별 작업 내역

Pollingpending event 조회, batch size, visibility/lock timeout, 중복 relay 방지 기준을 구현합니다.
Publishevent schema 필수 필드 검증 후 EventBridge bus 또는 SQS queue로 발행합니다.
Retry일시 장애는 재시도하고, 반복 실패는 실패 사유와 함께 운영자가 확인 가능한 상태로 분리합니다.
Observability발행 성공/실패 수, 지연 시간, oldest pending age를 metric으로 남깁니다.

테스트 시나리오

OUTBOX-001pending event를 한 번만 발행하고 published 상태로 변경한다.
OUTBOX-002publisher 장애 시 retry count와 last error를 기록하고 재시도 가능 상태를 유지한다.
OUTBOX-003동일 event를 중복 polling해도 외부 발행은 idempotent하게 처리한다.

후속 작업

현재 Outbox Relay 착수 전 이어서 확정할 작업입니다. 이 표의 작업이 진행되면 관련 코드와 문서를 함께 갱신합니다.

우선순위 작업 범위 완료 기준 함께 갱신할 문서
P0 Relay 대상과 DB 연결 경계 확정 Auth/IAM과 tenant-service의 outbox_events를 어떤 connection pool과 polling 기준으로 읽을지 정합니다. 서비스별 DB가 분리되어도 relay 대상, schema, connection secret, polling interval이 명확히 관리됩니다. outbox-relay-service 개발 문서, environment 규칙, AWS ECS 기술 문서
P0 EventBridge 또는 SQS 발행 정책 pending event를 발행할 AWS 대상, event envelope, retry, DLQ, published/failed 상태 전환 기준을 확정합니다. 이벤트 발행 성공/실패가 DB 상태와 CloudWatch 로그로 추적되고 소비자가 idempotent하게 처리할 수 있습니다. events/outbox 규칙, AWS ECS 기술 문서, smart-factory-aws-ecs.drawio
P1 Replay Tool 설계 운영자가 실패 event를 조건별로 재발행하는 API와 승인, 감사 로그, 권한 제한 흐름을 설계합니다. 실패 이벤트 재처리가 임의 DB 수정 없이 승인과 audit evidence를 남기며 수행됩니다. admin-bff-service 개발 문서, logging/audit 규칙, 운영 Runbook
