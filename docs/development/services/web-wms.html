Development Process

web-wms 개발과정

WMS 운영자 Desktop 1차 화면을 제공하는 React + Vite 프론트엔드 앱입니다. Desktop 조회 화면과 재고/입고/출고 mutation 작업 패널은 구현 완료했고, PDA Web은 후속 범위입니다.

서비스/앱 역할

위치apps/web-wms
카테고리Frontend App · WMS UI
대상창고 운영자 Desktop 사용자. 현장 작업자와 PDA Web 사용자는 후속 범위입니다.
연결브라우저는 gateway-service의 /api/app/**만 호출하고, gateway가 user-bff-service로 proxy합니다.
현재 화면로그인, 대시보드, 재고 현황, 재고 스냅샷, 기준정보, 출고 작업, 권한 제한

기능 정의

WMS App Shell로그인 후 사용자, tenant, 활성 module, WMS navigation, route guard를 표시합니다.
대시보드재고 현황, 출고 할당/포장 작업, 사용자 권한 기반 visible action을 사용자용 BFF API로부터 조회해 표시합니다.
기준정보 화면창고, 로케이션, 품목 기준정보를 조회하고 창고 코드, 로케이션 창고/코드, 품목 SKU 필터를 API query로 전달합니다.
재고 화면현재고와 일 단위 재고 스냅샷을 조회하고 창고, 로케이션, 품목, 스냅샷 일자 필터를 API query로 전달합니다. 재고 조정과 입고 확정 mutation을 같은 화면에서 처리합니다.
출고 작업 화면출고 할당과 포장 상태를 조회합니다. 할당 상태와 포장 상태는 각 API enum에 맞춰 별도 필터로 전달하고, 출고 할당, 포장 생성, 패키지 등록, 포장 확정, 출하 처리를 같은 화면에서 수행합니다.
제외 범위WMS 업무 규칙 계산, 권한 원장 관리, BFF를 우회한 도메인 서비스 직접 호출은 하지 않습니다.

구현 상태

상태Desktop 1차 조회/mutation API 연동 완료
선행 조건wms-service의 1차 조회/mutation API와 user-bff-service 화면 조합/proxy API가 준비되었습니다.
로컬 Dockerdocker/local/docker-compose.yml의 web-wms 서비스가 http://localhost:5174에서 nginx로 정적 파일을 제공하고 API base URL은 http://localhost:3000 gateway로 빌드합니다.
API 문서사용자 화면 API 계약은 docs/apis/user-bff-service/index.html의 /api/app/wms/** endpoint를 기준으로 합니다.

현재 개발 순서

web-wms는 실제 로그인 API부터 고정하고, app shell 권한 초기화, 조회 화면, mutation, PDA 순서로 진행합니다.

순서 작업 상태 완료 기준
1로그인 API 실연동완료POST /api/auth/login 성공 응답만 세션으로 저장하고 실패 시 오류 code와 requestId를 표시합니다.
2App Shell 세션/권한 초기화완료GET /api/app/me, GET /api/app/navigation으로 tenant, 사용자, permission, 메뉴를 구성합니다.
3Desktop 조회 화면 API 검증완료대시보드, 재고, 스냅샷, 기준정보, 출고 조회 API와 화면 필터가 gateway 경유 query 계약에 맞춰 동작합니다.
4업무 mutation 화면 연결완료재고 조정, 입고 확정, 출고 할당/포장/출하 API에 화면 action과 자동 Idempotency-Key를 연결했습니다.
5PDA Web 구현후속Desktop 1차 안정화 후 별도 PDA 화면으로 scan, retry, confirm 흐름을 구현합니다.

기술 스펙 결정

web-wms는 창고 운영자와 PDA Web 작업자가 같은 WMS 업무 흐름을 쓰되, 화면 밀도와 조작 방식이 달라지는 앱입니다. 첫 구현은 서버 계약을 그대로 반영하는 얇은 프론트엔드로 시작하고, 업무 규칙은 wms-service와 user-bff-service에 둡니다.

영역 선택 이유
런타임React + Vite + TypeScript프로젝트 공통 규칙과 AWS ECS 프론트엔드 배포 방향에 맞고, WMS 화면을 기능 단위로 빠르게 구성할 수 있습니다.
라우팅React Router로그인, dashboard, inventory, snapshot, master data, outbound, PDA 작업 화면을 URL과 권한 guard로 분리하기 쉽습니다.
서버 상태TanStack QueryUser BFF 조회 API의 loading, error, retry, cache invalidation을 화면별로 일관되게 다룰 수 있습니다.
클라이언트 상태컴포넌트 local state 우선, 필요 시 React Context현재 범위의 로그인 세션, 검색 조건, route 상태는 앱 내부 state로 충분하며 전역 업무 상태가 커질 때만 Context로 확장합니다.
API 클라이언트fetch 기반 래퍼 + zod 응답 검증응답 envelope, requestId, 오류 code, ISO timestamp 규칙을 한 곳에서 처리하고 런타임 계약 불일치를 조기에 발견합니다.
폼React controlled input + zod API 응답 검증현재 구현은 로그인과 조회 필터 중심이라 제어 컴포넌트로 충분합니다. mutation 입력이 커질 때 form 라이브러리 도입을 재검토합니다.
스타일CSS Modules + global design tokens아직 공통 UI 패키지가 없으므로 앱 내부에서 예측 가능한 스타일 격리를 유지하고, 이후 공유 컴포넌트로 승격하기 쉽습니다.
아이콘lucide-react버튼, navigation, 상태 표시용 아이콘을 일관되게 제공하고 번들 부담이 작습니다.
검증TypeScript, build, Vitest/RTL, 시나리오 기반 AI 검증프로젝트 테스트 규칙상 첫 단계는 E2E 자동화보다 시나리오 evidence를 우선하며, UI 단위 회귀는 가벼운 테스트로 보강합니다.

2026-06-23 WMS mutation 화면과 API 연동

  • 재고 화면에 재고 조정과 입고 확정 작업 패널을 추가했습니다. 현재고 첫 행을 작업 폼에 적용할 수 있고, mutation 성공 시 requestId를 화면에 표시합니다.
  • 출고 작업 화면에 출고 할당, 포장 생성, 패키지 등록, 포장 확정, 출하 처리 패널을 추가했습니다. 할당/포장 첫 행 적용 버튼과 작업 성공 후 query invalidation을 연결했습니다.
  • API client는 모든 POST mutation에 자동 Idempotency-Key를 생성하고, 실패 시 error.code와 requestId를 화면 오류로 표시합니다.
  • 검증 명령: pnpm --filter web-wms typecheck, pnpm --filter web-wms build, pnpm --filter user-bff-service typecheck, pnpm --filter user-bff-service build.
  • gateway 경유 실 API run web-wms-mut-mqpsvn8i에서 재고 조정, 입고 확정, 직접 출하, 포장 생성/패키지/확정/출하가 모두 성공했습니다. 상세 evidence는 docs/test-logs/2026-06-23/web-wms-mutation-verification.html에 기록했습니다.

  • 재고, 스냅샷, 기준정보, 출고 화면의 검색 폼을 실제 query 상태와 연결했습니다. 기준정보는 창고 코드, 로케이션 창고/코드, 품목 SKU를 각각 해당 API로 전달합니다.
