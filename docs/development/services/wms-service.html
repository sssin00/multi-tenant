Development Process

wms-service 개발과정

WMS 기준정보, 재고, 일 단위 스냅샷, 입고, 출고, 포장, 출하를 담당하는 첫 메인 도메인 서비스입니다. 1차 구현은 내부 API, tenant scope, 권한 확인, 재고 원장, outbox 이벤트 저장, 감사 로그 연계 기준까지 완료했습니다.

서비스/앱 역할

위치apps/wms-service
카테고리Backend Service · Domain Service
외부 노출직접 노출하지 않습니다. 사용자 화면 API는 이후 user-bff-service가 담당합니다.
초기 구조하나의 서비스 내부를 모듈형 모놀리스로 구성합니다.
1차 도메인warehouse, location, item, inventory ledger/balance/daily snapshot, inbound, outbound allocation, packing, direct/packing shipment
다음 확장read 권한 분리, Idempotency-Key 저장, task, lot/serial, wave picking, ERP 전송 retry, 운영 replay/approval

서비스 경계

소유WMS 기준정보, 재고 수량, 입고 확정, 출고 할당/출하, WMS outbox event
소유하지 않음로그인, 사용자/권한 원장, tenant lifecycle, 화면 데이터 조합, 감사 로그 저장소, 외부 ERP orchestration
호출 방향BFF 또는 내부 서비스가 WMS 내부 API를 호출합니다. WMS는 tenant-service와 auth-iam-service의 내부 API만 사용합니다.
이벤트 방식WMS 업무 DB 변경과 outbox_events 저장을 같은 transaction에서 처리합니다. 감사 로그 저장은 relay/SQS/audit-log-service가 담당합니다.

포장과 출하 구현

기존 allocation 직접 출하 흐름은 유지하면서, allocation -> packing -> shipment 단계로 출고 업무를 세분화했습니다. 포장 확정과 포장 기반 출하는 WMS DB row와 outbox event를 같은 transaction에서 저장합니다.

단계 목표 주요 데이터 이벤트
1 포장 작업 생성 allocated 상태의 allocation을 포장 대상으로 묶습니다. packingId, outboundOrderId, allocationIds, status, packedBy -
2 박스/패키지 등록 포장 박스별 품목, 수량, 중량, 부피, 바코드/운송장 후보 정보를 기록합니다. packageId, packageNo, itemId, quantity, weight, dimensions -
3 포장 확정 포장 수량이 할당 수량과 일치하는지 검증하고 packing 상태를 confirmed로 변경합니다. confirmedAt, confirmedBy, package summary wms.outbound.packed
4 출하 확정 포장 확정 건을 출하 처리하고 재고/할당 수량을 최종 차감합니다. shipmentId, carrierCode, trackingNo, shippedAt wms.outbound.shipped
호환성기존 POST /api/internal/wms/outbound/shipments의 allocationId 직접 출하 흐름은 유지합니다. 포장 기반 출하는 같은 endpoint의 packingId branch로 처리합니다.
권한포장 생성/목록/패키지 등록/확정은 wms.outbound.pack, 출하는 기존 wms.outbound.ship를 사용합니다. read 권한 분리는 후속 결정입니다.
트랜잭션포장 확정과 출하 확정은 각각 업무 row와 outbox event 저장을 같은 DB transaction에서 처리합니다.
감사포장 확정과 출하 확정은 audit-log-service 직접 호출이 아니라 outbox relay/SQS/audit consumer 경로로 기록합니다.
wms.outbound.packed payloadpackingId, outboundOrderId, allocationIds, packageIds, packageCount, totalQuantity, confirmedAt를 포함하고, 품목 상세는 소비자가 꼭 필요한 최소 필드만 추가합니다.

일 단위 재고 스냅샷 구현 기준

과거 일자별 보유 자재 수량 조회를 위해 재고 변경 원장과 일 단위 스냅샷 read model을 추가합니다. 현재고 테이블을 단순 복사하면 배치 지연, 재실행, 과거일 재생성에서 정합성이 흔들릴 수 있으므로 모든 재고/할당 수량 변경을 먼저 원장에 기록하고, 스냅샷은 지정 일자 마감 시점의 원장을 기준으로 생성합니다.

상태Implemented 원장 기반 스냅샷과 수동 생성 API를 추가했습니다.
조회 기준snapshotDate 하루 단위로 tenant business date를 사용합니다. tenant timezone 설정이 도입되기 전 기본값은 Asia/Seoul로 확정합니다.
마감 기준해당 business date의 종료 시각까지 발생한 원장 중 각 warehouse/location/item 조합의 마지막 balance를 스냅샷으로 저장합니다.
수량 필드quantity, allocatedQuantity, availableQuantity를 문자열 decimal로 응답합니다.
정합성 원칙재고 mutation, 원장 insert, outbox event 저장은 같은 DB transaction 안에서 처리합니다. 스냅샷 생성은 원장 기준으로 멱등 재실행 가능해야 합니다.
과거 데이터원장 도입 전 기간은 정확한 역산을 보장하지 않습니다. 배포일 기준 baseline snapshot을 만들고, 정확 보장 시작일을 문서와 API 응답에 명확히 표시합니다.
구현 모델 주요 필드 목적
wms_inventory_ledger ledgerId, tenantId, warehouseId, locationId, itemId, movementType, quantityChange, allocatedQuantityChange, balanceQuantityAfter, balanceAllocatedQuantityAfter, sourceType, sourceId, effectiveDate, occurredAt, correctedLedgerId, correctionReason, requestId 모든 현재고/할당 수량 변경을 transaction 안에서 append-only로 남기는 정합성 기준입니다.
wms_inventory_daily_snapshots snapshotId, tenantId, snapshotDate, snapshotAt, warehouseId, locationId, itemId, quantity, allocatedQuantity, availableQuantity, sourceLedgerId, runId, previousSnapshotId, isCurrent, generatedAt 일자별 조회 성능과 재실행 안정성을 위한 materialized read model입니다. 보정 재생성 시 기존 row는 isCurrent=false로 보관하고 새 row를 current로 저장합니다.
wms_inventory_snapshot_runs runId, tenantId, snapshotDate, snapshotAt, warehouseId, mode, status, startedAt, finishedAt, generatedCount, unchangedCount, errorCode, requestId 일자별 생성/재생성 lock, 실패 추적, 운영 증적을 남깁니다.
순서 처리 정합성 기준
1재고 조정, 입고 확정, 출고 할당, 출하 확정에서 balance를 변경합니다.balance 변경과 ledger insert를 같은 transaction으로 묶습니다.
2일 마감 배치 또는 승인된 수동 명령이 tenant/date 단위 snapshot run lock을 획득합니다.같은 tenant/date에 대해 동시에 두 생성 작업이 실행되지 않아야 합니다.
3마감 cutoff 이하의 ledger에서 각 재고 키별 마지막 balance를 조회합니다.배치가 다음 날 실행되어도 과거 일자 결과가 바뀌지 않아야 합니다.
4daily snapshot은 변경이 있을 때 기존 current row를 보관하고 새 current row를 생성합니다.같은 입력 원장으로 재실행하면 새 row 없이 unchangedCount만 증가해야 합니다.
5조회 API는 snapshot table만 읽고 필터/페이지네이션을 적용합니다.tenantId 조건과 API envelope 규칙을 지킵니다.
조회 APIGET /api/internal/wms/inventory/snapshots?snapshotDate=YYYY-MM-DD&warehouseId=&locationId=&itemId=&page=&size=, 권한은 기존 wms.inventory.read를 사용합니다.
생성 APIPOST /api/internal/wms/inventory/snapshots/generate는 운영/관리 목적의 내부 명령입니다. 권한명은 wms.inventory.snapshot.generate를 신설합니다.
이벤트스냅샷 조회 자체는 이벤트를 만들지 않습니다. 생성/재생성 완료는 wms.inventory.snapshot.generated outbox event로 남깁니다.
보정 정책과거일 보정은 허용합니다. 보정 원장은 effectiveDate, correctedLedgerId, correctionReason을 남기고, 재생성된 스냅샷은 이전 current row를 보관해 보정 전/후 수량 추적이 가능해야 합니다.

연계 구조

user-bff-service 또는 admin-bff-service
  -> wms-service /api/internal/wms/**
    -> tenant-service /internal/tenants/{tenantId}/modules
    -> auth-iam-service /api/auth/permissions/check
    -> WMS DB transaction
      -> WMS domain rows
      -> outbox_events
        -> outbox-relay-service
          -> SQS
            -> audit-log-service
tenant 확인업무 API마다 X-Tenant-Id를 요구하고, tenant-service modules 응답에 wms가 포함되어야 합니다.
권한 확인중요 action 전 auth-iam-service의 permission check를 호출합니다. warehouse scope가 있으면 scope.warehouseId를 전달합니다.
감사 로그WMS는 audit-log-service를 직접 호출하지 않습니다. outbox event가 relay와 SQS를 거쳐 audit-log-service에 저장됩니다.

  • pnpm test:wms:local을 다시 실행해 WMS 정상 흐름, 포장/출하, 스냅샷, 오류 회귀 시나리오 22개 통과를 확인했습니다.
  • 로컬 compose의 기본 outbox-relay-service가 WMS outbox를 mock으로 선점하지 않도록 검증 스크립트가 필요한 경우 해당 컨테이너를 일시 중지하고 종료 시 복구하도록 보완했습니다.
  • 검증 증적은 wms.outbound.packed, wms.outbound.shipped, wms.inventory.snapshot.generated의 LocalStack SQS publish와 audit row 저장을 포함합니다.

2026-06-19 일 단위 재고 스냅샷 구현

  • 과거 보유 자재 수량 조회를 위해 현재고 복사 방식이 아니라 wms_inventory_ledger 기반 스냅샷 생성 방식을 구현했습니다.
  • wms_inventory_daily_snapshots와 wms_inventory_snapshot_runs를 추가해 일자별 조회, 재실행 멱등성, 운영 증적을 분리했습니다.
  • 기본 business date는 Asia/Seoul, 수동 생성 권한은 wms.inventory.snapshot.generate, 생성 완료 이벤트는 wms.inventory.snapshot.generated로 확정했습니다.
  • 과거일 보정은 effectiveDate, correctedLedgerId, correctionReason 원장으로 허용하고, 이전 스냅샷 row는 isCurrent=false로 보관합니다.

2026-06-19 포장과 출하 구현

  • wms_outbound_packings, wms_outbound_packages, wms_outbound_package_items, wms_outbound_shipments 모델을 추가했습니다.
  • 포장 생성, 포장 목록, 패키지 등록, 포장 확정, 포장 기반 출하 API를 추가했습니다.
  • 포장 확정은 wms.outbound.packed, 출하 확정은 wms.outbound.shipped outbox event를 저장합니다.

2026-06-19 포장과 출하 확장 계획

  • 기존 출고 흐름을 깨지 않고 allocation -> packing -> shipment 단계로 확장하기로 했습니다.
  • 포장 작업, 박스/패키지, 포장 확정, 포장 기반 출하를 별도 계획 항목과 테스트 시나리오로 추가했습니다.
  • 새 이벤트는 wms.outbound.packed를 추가하고, 출하 완료는 기존 wms.outbound.shipped 이벤트를 유지합니다.

2026-06-19 개선 전 테스트 준비

  • scripts/test/verify-local-wms-flow.mjs의 requestId를 실행 단위로 고유하게 만들어 이전 outbox/audit row가 검증 결과에 섞이지 않도록 했습니다.
  • pnpm test:wms:local이 기존 WMS 정상 흐름 5개에 더해 tenant 누락, invalid UUID, 중복 창고/품목, 음수 재고 조정, 재고 부족 출고 할당, 출하 중복 방지까지 확인하도록 준비했습니다.
  • 실패 시나리오는 표준 error envelope와 실패 requestId 기준 outbox 미생성을 증거로 남깁니다.

2026-06-17 1차 설계 및 작업 분해

  • WMS를 직접 외부 노출하지 않는 내부 도메인 서비스로 확정했습니다.
  • 1차 범위를 기준정보, 재고 조정, 입고 확정, 출고 할당/출하로 제한했습니다.
  • tenant module check, Auth/IAM permission check, outbox relay, audit-log-service 연계를 테스트 범위에 포함했습니다.
  • 구현 담당 서브에이전트는 apps/wms-service/**만 수정하고, 문서/연계/검증은 통합 단계에서 별도로 처리합니다.

2026-06-17 1차 구현 완료

  • apps/wms-service에 NestJS shell, config, health/readiness, request context, access log, exception filter, internal auth guard를 추가했습니다.
  • Prisma schema와 migration으로 WMS 기준정보, 재고 balance, inventory adjustment, inbound receipt, outbound order/allocation, outbox_events를 구성했습니다.
  • tenant-service module check와 auth-iam-service permission check를 업무 API마다 호출하도록 연결했습니다.
  • 재고 조정, 입고 확정, 출고 할당, 출하 완료 시 WMS DB 변경과 outbox 저장을 같은 transaction에서 처리합니다.
  • 첫 로컬 검증에서 UUID 정규식이 네 번째 그룹과 다섯 번째 그룹 사이의 하이픈을 누락해 정상 tenantId를 거부하는 문제를 발견했고 수정했습니다.
  • Postman collection에는 WMS 내부 직접 호출 폴더를 추가했습니다. 제품 API 경로는 계속 Gateway → BFF → WMS 원칙을 따릅니다.
