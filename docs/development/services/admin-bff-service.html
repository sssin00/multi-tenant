Development Process

admin-bff-service 개발과정

운영자/관리자 화면에 필요한 테넌트, 사용자, 권한, 설정, 감사 로그 API를 조합합니다.

서비스/앱 역할

위치apps/admin-bff-service
카테고리Backend Service · BFF
책임시스템 관리자 전용 고객사 관리, 사용자/권한 관리, 운영 설정, 감사 로그 조회 화면 API 조합
제작 시점tenant-service 1차 완료 후, WMS 본격 개발 전에 운영자용 tenant/user/role 관리 surface부터 제작
Tenant 관리 경계외부 /api/admin/tenants surface를 소유하고, 실제 tenant 생성/상태/module/domain 변경은 tenant-service 내부 API로 위임합니다.
주의권한의 최종 판단은 Auth/IAM에 두고, tenant business rule은 tenant-service에, WMS 업무 정책은 WMS에 둡니다. BFF는 직접 DB에 접근하지 않습니다.

기능 정의

관리자 API 조합운영자/관리자 화면에서 필요한 tenant, user, role, permission, audit 데이터를 내부 서비스 호출로 조합합니다.
Tenant 관리 Surface/api/admin/tenants 계열 외부 API를 소유하고 실제 변경은 tenant-service 내부 API로 위임합니다.
Access Control 화면Auth/IAM의 role, permission, user API를 조합해 관리 화면용 응답 형태를 제공합니다.
Audit 조회운영자 감사 로그 조회 API를 제공하고 audit-log-service 내부 조회 API로 위임합니다.
권한 확인관리자 API별 permission을 Auth/IAM에 확인하고 tenant status check를 통과한 요청만 처리합니다.
제외 범위도메인 데이터 저장, 인증 토큰 발급, tenant/WMS business rule 직접 판단은 하지 않습니다.

기능별 작업 내역

RuntimeNestJS + TypeScript ESM 구조입니다. pnpm --filter admin-bff-service dev, typecheck, build, start script를 제공합니다.
HealthGET /health, GET /ready를 제공합니다. readiness는 config, downstream URL, internal auth secret, CORS/security 기준을 확인합니다.
ContextX-Request-Id, X-Tenant-Id, X-User-Id를 request context에 저장하고 응답 header로 request/tenant id를 전파합니다.
Logging요청 완료 시 JSON access log에 requestId, tenantId, userId, method, path, statusCode, durationMs를 남깁니다.
Security기존 gateway-service와 같은 보안 header와 명시 origin 기반 CORS 규칙을 사용합니다.
Internal authauth-iam-service와 tenant-service 호출에 사용할 HMAC signer를 추가했습니다.
Permission check@AdminPermission(...) decorator와 전역 guard를 추가했습니다. decorator가 붙은 route는 auth-iam-service의 POST /api/auth/permissions/check를 internal auth로 호출해 관리자 권한을 확인합니다. tenant 없는 system_admin 요청은 Auth/IAM의 전역 permission check 결과를 그대로 사용합니다.
Admin SessionGET /api/admin/me를 추가했습니다. Auth/IAM의 현재 사용자 정보와 tenant-service의 tenant 상태/module 정보를 조합하고, permission summary 기준으로 관리자 navigation을 반환합니다. tenant 없는 system_admin은 tenant-service 상태 조회를 생략하고 tenantId=null, status=system_admin, enabledModules=["admin"]을 반환합니다.
Tenant clientTenantInternalClient를 추가했습니다. tenant 목록/생성/상세/수정/상태/module/domain 내부 API를 호출하고, 변경성 요청에는 Idempotency-Key를 전파합니다.
DashboardGET /api/admin/dashboard를 추가했습니다. tenant 목록, Auth/IAM permission summary와 사용자/role/permission 목록, audit log 최근 목록을 병렬 조회해 관리 대시보드용 요약 응답으로 조합합니다. system_admin은 tenant 없이 전체 tenant/user/role/audit 요약을 조회합니다.
Audit clientAuditLogInternalClient와 GET /api/admin/audit-logs controller를 구현했습니다. 로컬 compose 기준 AUDIT_LOG_SERVICE_URL=http://audit-log-service:3000와 AUDIT_INTERNAL_AUTH_SECRET로 GET /api/internal/audit/logs에 query, request context, internal auth header를 전파합니다.
Admin audit writeAdminAuditService가 tenant/user/role 변경성 action 성공 후 POST /api/internal/audit/logs로 admin.* audit event를 best-effort 기록합니다. details에는 password, token, Authorization, email 원문을 포함하지 않습니다.
Infraapps/admin-bff-service/Dockerfile을 추가했습니다. ECS 컨테이너 기본 포트는 3000이고 gateway는 Cloud Map 내부 URL과 /api/admin base path를 포함한 ADMIN_BFF_SERVICE_URL로 /api/admin/**를 전달합니다.
DeployCDK에 ECR repository, CloudWatch log group, security group, ECS Fargate task/service, Cloud Map service를 추가했습니다. 기본 adminBffDesiredCount는 0이며, 이미지 push 후 deploy-admin-bff-service.yml 수동 workflow로 desired count를 올립니다.

테스트 시나리오

ADMIN-BFF-001관리자 권한이 있는 사용자는 tenant 목록/상세/생성/수정/상태/module/domain API를 호출할 수 있다.
ADMIN-BFF-002권한이 없는 사용자는 각 API의 @AdminPermission 기준에 따라 403 FORBIDDEN으로 차단된다.
ADMIN-BFF-003변경성 요청은 Idempotency-Key가 없으면 차단되고, downstream 내부 호출에도 header가 전파된다.
ADMIN-BFF-004audit-log-service URL이 설정된 환경에서 audit log 조회는 GET /api/internal/audit/logs로 위임되고 표준 list envelope를 반환한다.
ADMIN-BFF-005audit-log-service URL이 없는 선택 환경에서 audit log 조회는 AUDIT_LOG_SERVICE_NOT_CONFIGURED로 실패한다.
ADMIN-BFF-006관리자 변경성 action 성공 후 같은 requestId로 audit log를 조회하면 admin.* audit row가 1건 이상 반환된다.

후속 작업

현재 Admin BFF의 1차 구현 이후 이어서 진행할 작업입니다. 이 표의 작업이 진행되면 관련 코드와 문서를 함께 갱신합니다.

우선순위 작업 범위 완료 기준 함께 갱신할 문서
P1 Admin Me API 구현 /api/admin/me에서 관리자 프로필, tenant scope, permission summary, 메뉴 표시용 권한 정보를 조합합니다. web-admin 초기화 화면이 별도 중복 호출 없이 현재 사용자와 권한 요약을 받아 렌더링할 수 있습니다. admin-bff-service API 문서, web-admin 개발 문서, Auth/IAM 개발 문서
P1 Audit 장애 처리 정책 보강 audit-log-service timeout, unavailable, 저장/조회 장애 시 Admin BFF와 upstream 서비스의 retry/fail-open/fail-closed 기준을 확정합니다. 운영 정책과 시나리오가 확정되고 실패 응답, retry, alert 기준이 문서와 구현에 반영됩니다. admin-bff-service API 문서, audit-log-service 개발 문서, logging/audit 규칙

2026-06-23 tenantless system_admin 관리자 surface 지원

  • AdminPermissionGuard와 Auth/IAM internal client가 tenant 없는 context를 그대로 전달하도록 바꿔 system_admin의 전역 permission check를 허용했습니다.
  • GET /api/admin/me는 tenant 없는 system_admin 요청에서 tenant-service status/module 조회를 생략하고 tenantId=null, status=system_admin, synthetic role/permission navigation을 반환합니다.
  • GET /api/admin/dashboard는 system_admin 요청에서 tenant/user/role/permission/audit 요약을 전체 scope로 조합합니다.
  • 검증 evidence: requestId=verify-super-admin-me-v3에서 tenant 없는 system_admin 세션이 tenantId=null, status=system_admin, 전체 permission 응답으로 성공했습니다. verify-admin-tenantless-me는 tenant 관리자 JWT의 resolved tenantId로 기존 tenant-scoped 세션을 반환했습니다.

2026-06-22 관리자 변경성 action audit 기록 보강

  • AdminAuditService를 추가해 tenant 생성/수정/상태/module/domain, 사용자 생성/수정/상태/role 부여/해제, role/permission 변경 성공 후 audit-log-service 내부 기록 API로 admin.* event를 저장합니다.
  • 감사 저장 장애는 warning log로 남기고 원 관리자 action 응답은 차단하지 않는 best-effort 정책으로 처리했습니다.
  • 로컬 재검증 run web-admin-audit-fix-mqorf18i에서 변경성 requestId 8개가 모두 GET /api/admin/audit-logs?requestId=... 기준 total=1을 반환했습니다.
  • 검증 명령: pnpm --filter admin-bff-service typecheck, docker compose -f docker/local/docker-compose.yml up -d --build admin-bff-service audit-log-service, gateway 경유 web-admin 시나리오 재검증.

2026-06-23 고객사 관리 scope 제한

  • GET /api/admin/me navigation에서 고객사 관리 메뉴는 system_admin scope에만 노출합니다. 테넌트 관리자는 tenant.tenants.read 계열 permission이 있더라도 좌측 메뉴와 직접 route 접근 기준에서 제외됩니다.
  • /api/admin/tenants/** 고객사 관련 API는 Admin BFF에서 tenant scope가 있는 관리자 요청을 403 AUTH_ADMIN_SCOPE_MISMATCH로 차단합니다. 시스템 관리자는 tenantless JWT와 기존 permission check를 통과해야 호출할 수 있습니다.
  • 로컬 gateway 검증: verify-scope-system-me에서 고객사 관리 navigation과 verify-scope-system-tenants 200을 확인했고, verify-scope-tenant-me에서는 메뉴 미노출, verify-scope-tenant-tenants에서는 403 AUTH_ADMIN_SCOPE_MISMATCH를 확인했습니다.

2026-06-11 BFF 경계 최신화

  • admin-bff-service는 외부 관리자 API surface를 담당하고, tenant 생성은 tenant-service의 POST /internal/admin/tenants 내부 API로 위임하는 기준을 명확히 했습니다.
  • gateway-service는 tenant-service를 직접 호출하지 않고, tenant status check와 tenant 관리 API는 Admin BFF가 tenant-service 내부 API로 위임하는 기준을 확정했습니다.
  • 제작 순서는 User BFF보다 앞입니다. WMS 업무 화면보다 tenant, 사용자, role, module 운영 기반을 먼저 준비합니다.

2026-06-11 API 계약 초안 작성

  • POST /api/admin/tenants를 첫 구현 후보로 두고, tenant-service의 추가 내부 API가 필요한 tenant 목록/상세/상태/module/domain API는 blocked 상태로 표시했습니다.

2026-06-11 NestJS 기본 구조 생성

  • Admin BFF는 직접 DB에 접근하지 않으므로 readiness는 downstream URL과 internal auth secret 중심으로 확인합니다.

2026-06-11 Auth/IAM 권한 check 레이어 추가

  • auth-iam-service의 GET /api/auth/permissions/summary, POST /api/auth/permissions/check를 호출하는 internal client를 추가했습니다.
  • @AdminPermission(...) decorator와 AdminPermissionGuard를 전역 guard로 등록해, 관리자 API별 permission check를 붙일 수 있게 했습니다.
  • 권한 확인은 propagated X-Tenant-Id, X-User-Id, X-Request-Id를 사용하고, 실패 시 표준 error envelope로 응답합니다.

2026-06-11 Tenant internal admin client 추가

  • tenant-service의 내부 API를 호출하는 TenantInternalClient를 추가했습니다.
  • 현재 구현된 tenant 생성과 module 조회 API를 호출할 수 있고, tenant 목록/상세/수정/상태/module/domain API는 tenant-service 선행 구현 후 바로 연결할 수 있도록 메서드를 준비했습니다.
  • tenant-service 호출도 X-Request-Id, X-Tenant-Id, X-User-Id와 HMAC internal auth header를 전파합니다.

2026-06-11 Tenant 목록/생성/상태 변경 API 구현

  • GET /api/admin/tenants, POST /api/admin/tenants, PATCH /api/admin/tenants/{tenantId}/status controller를 추가했습니다.
  • 각 API에 @AdminPermission을 적용해 auth-iam-service permission check를 거친 뒤 tenant-service 내부 API로 위임합니다.
  • Admin BFF 전역 tenant status guard를 추가해 GET /internal/tenants/{tenantId}/status로 active tenant만 관리자 API를 통과시킵니다.
  • 외부 응답은 Admin BFF envelope로 감싸고, request context와 internal auth header를 downstream으로 전파합니다.

  • GET/POST/PATCH /api/admin/roles/**, PUT /api/admin/roles/{roleId}/permissions, GET/POST /api/admin/permissions API를 추가했습니다.
  • GET /api/admin/access-control/screen-data는 사용자 목록, role 목록, permission 목록을 병렬 조회해 화면 초기 데이터를 조합합니다.
  • 사용자 상세 API는 사용자 상세와 user-role 목록을 함께 조합해 반환합니다.

  • 모든 API는 @AdminPermission으로 Auth/IAM permission check를 거친 뒤 auth-iam-service로 위임하고, 변경성 요청은 Idempotency-Key header를 필수로 확인합니다.

2026-06-18 사용자 타입 계약 반영
