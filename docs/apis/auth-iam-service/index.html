API Docs

auth-iam-service API 문서

로그인, token refresh, 사용자, 역할, 권한 조회 API의 구현 계약입니다. 외부 호출은 gateway-service의 /api/auth/** proxy route를 통해 들어옵니다.

API 공통 기준

Public base path/api/auth
Gateway upstreamAUTH_IAM_SERVICE_URL
HeadersAuthorization, Content-Type, Accept, X-Request-Id, X-Tenant-Id, Idempotency-Key
Internal headersInternal API는 X-Internal-Service-Id, X-Internal-Timestamp, X-Internal-Signature HMAC 인증 header가 필요합니다.
Internal signing서명 대상은 METHOD, originalUrl, timestamp, requestId, body SHA-256 hash를 줄바꿈으로 연결한 문자열입니다.
Envelope모든 응답은 success, requestId, timestamp를 포함합니다. list 응답은 data.items, data.page, data.size, data.total만 사용합니다.
JWTaccess token은 JWT이며 sub, tenantId, type, iat, exp, iss, aud만 포함합니다.
Refresh tokenopaque token을 사용하고 서버에는 hash만 저장합니다. refresh 사용 시 새 token을 발급하고 이전 token은 폐기합니다.
Auditauth-iam-service는 audit_logs 테이블을 소유하지 않습니다. 사용자/역할/권한 변경 API는 AUDIT_LOG_SERVICE_URL의 별도 audit-log-service로 audit record 저장을 요청합니다.
Outbox사용자/permission/role 변경 API는 업무 변경과 같은 DB transaction에서 outbox_events에 pending event를 저장합니다.

Health와 Readiness

GET /health인증 없이 호출할 수 있으며 프로세스 생존 상태를 반환합니다. 응답 header에는 X-Request-Id가 포함됩니다.
GET /ready트래픽 수용 가능 여부를 확인합니다. config, JWT, internal auth, database, Redis, security check를 포함합니다.
Success200과 status=ready를 반환합니다.
Failure필수 설정 또는 의존성 실패 시 503과 status=not_ready를 반환합니다. Readiness 응답은 일반 error envelope가 아니라 check 목록을 직접 반환합니다.

  }

Login

EndpointPOST /api/auth/login
Status성공 시 200, 인증 실패 시 401 AUTH_INVALID_CREDENTIALS, tenant mismatch 시 403 TENANT_MISMATCH를 반환합니다.
TenantX-Tenant-Id header 또는 body tenantId가 필요합니다. 둘 다 있으면 값이 같아야 합니다.
Password서버 저장 hash는 scrypt와 사용자별 random salt를 사용합니다. password 원문은 저장하거나 로그에 남기지 않습니다.
Refreshlogin 응답의 refresh token은 opaque token이며 DB에는 SHA-256 hash만 저장합니다.
Response dataaccessToken, expiresIn, refreshToken, refreshExpiresIn, tokenType
Errors400 VALIDATION_FAILED, 401 AUTH_INVALID_CREDENTIALS, 403 AUTH_ACCOUNT_LOCKED, 403 TENANT_MISMATCH
 
{
  "tenantId": "11111111-1111-4111-8111-111111111111",
  "email": "worker@example.com",
  "password": "user-password"
}

Refresh와 Logout

Refresh endpointPOST /api/auth/token/refresh
Refresh status성공 시 200으로 새 access token과 refresh token을 반환합니다. 이전 refresh token은 revokedAt, replacedBy로 폐기합니다.
Reuse failure존재하지 않거나 만료되었거나 이미 폐기된 refresh token은 401 AUTH_INVALID_TOKEN으로 실패합니다.
Logout endpointPOST /api/auth/logout
Logout behaviorbody의 refreshToken이 있으면 해당 token만 폐기합니다. 없으면 gateway가 전달한 X-Tenant-Id, X-User-Id 기준으로 active refresh token을 모두 폐기합니다.
Tenantbody tenantId와 X-Tenant-Id가 함께 있고 값이 다르면 403 TENANT_MISMATCH로 실패합니다.
Errors400 VALIDATION_FAILED, 401 AUTH_INVALID_TOKEN, 403 TENANT_MISMATCH

  "timestamp": "2026-06-09T00:00:00.000Z",
  "data": {
    "revoked": true,
    "revokedCount": 1
  }

User CRUD

Tenant모든 사용자 API는 X-Tenant-Id가 필요하며, DB 접근 조건에 tenantId를 포함합니다.
CreatePOST /api/auth/users는 email, displayName, password, optional status를 받습니다. 기본 상태는 active입니다.
ListGET /api/auth/users?page=1&size=20&status=active&email=worker 형식을 지원합니다. list 응답은 data.items, data.page, data.size, data.total만 사용합니다.
DetailGET /api/auth/users/{userId}는 다른 tenant 사용자를 404 AUTH_USER_NOT_FOUND로 숨깁니다.
UpdatePATCH /api/auth/users/{userId}는 email과 displayName 변경을 지원합니다.
StatusPATCH /api/auth/users/{userId}/status는 active, inactive, locked를 지원합니다. inactive 또는 locked 전환 시 active refresh token을 revoke합니다.
DeleteDELETE /api/auth/users/{userId}는 hard delete가 아니라 inactive 상태 전환으로 처리합니다.
Secret사용자 응답에는 passwordHash를 포함하지 않습니다.
Audit생성, 수정, 상태 변경은 auth.user.created, auth.user.updated, auth.user.statusChanged action으로 audit-log-service에 기록 요청을 보냅니다.

Role과 Permission

Permission catalogPOST/GET /api/auth/permissions로 전역 permission code catalog를 관리합니다. code는 {domain}.{resource}.{action} 형식을 사용합니다.
RolePOST/GET/PATCH /api/auth/roles는 X-Tenant-Id 기준 tenant-scoped role을 관리합니다.
Role mappingPUT /api/auth/roles/{roleId}/permissions는 role에 연결된 permission code 목록을 교체합니다.
User assignmentPOST /api/auth/users/{userId}/roles는 사용자에게 role을 부여합니다. optional warehouseId로 scope를 제한할 수 있습니다.
SummaryGET /api/auth/permissions/summary?userId={userId}는 BFF가 사용할 role과 permission 요약을 반환합니다.
CheckPOST /api/auth/permissions/check는 도메인 서비스가 업무 실행 전 permission과 warehouse scope를 확인할 때 사용합니다.
System adminsystem_admin은 tenant-scoped role API에서 생성할 수 없습니다. system.* permission은 tenant role mapping과 tenant permission check에서 403 AUTH_ADMIN_SCOPE_MISMATCH로 차단합니다.
Tenant admintenant_admin은 tenant 전체 scope role입니다. warehouseId가 포함된 user-role assignment는 403 AUTH_ADMIN_SCOPE_MISMATCH로 실패합니다.
Auditpermission 생성, role 생성/수정, role-permission 교체, user-role 부여/해제는 audit-log-service에 변경 record 저장을 요청합니다.
Outboxpermission 생성, role 생성/수정, role-permission 교체, user-role 부여/해제는 auth.permission.created, auth.role.*, auth.userRole.* event로 저장합니다.

운영 보완 사항

JWT 운영 알고리즘

로컬은 HS256로 충분하지만, 운영은 RS256와 key rotation 방식을 확정해야 합니다.

내부 호출 인증

Permission summary/check는 HMAC 기반 service-to-service 인증을 적용했습니다. Secret은 AUTH_INTERNAL_AUTH_SECRET로 주입합니다.

Bootstrap 경계

tenant 최초 관리자 생성 API와 system admin 권한 경계를 명확히 분리해야 합니다.

Audit 서비스 계약

현재 내부 저장 경로는 /api/internal/audit/logs로 두었습니다. 실제 audit-log-service 구현 시 인증 header, retry, 장애 시 처리 방식을 확정해야 합니다.

Outbox Relay

outbox_events 발행, retry, DLQ, published/failed 상태 전환은 outbox-relay-service 후속 작업에서 확정합니다.

후속 작업

상세 백로그는 auth-iam-service 개발 문서의 후속 작업 백로그를 기준으로 관리합니다.
