user-bff-service API 문서

user-bff-service는 사용자 업무 화면이 사용하는 /api/app/** surface를 소유합니다. tenant 상태, module, 권한 summary, WMS 조회 데이터를 조합하되 WMS 업무 규칙과 직접 DB 접근은 담당하지 않습니다.

현재 상태

기준일2026-06-21
Gateway 연결/api/app/**는 gateway-service의 app route group을 통해 USER_BFF_SERVICE_URL로 전달됩니다.
구현 완료GET /health, GET /ready, GET /api/app/me, GET /api/app/navigation, GET /api/app/wms/warehouses, GET /api/app/wms/materials, GET /api/app/wms/inventory-summary, GET /api/app/wms/dashboard
감사 로그/api/app/me, /api/app/navigation은 app shell 감사 이벤트를 best-effort 발행합니다. EventBridge bus가 설정된 환경은 EventBridge로 발행하고, 기존 local/호환 환경은 audit-log-service 내부 기록 API를 사용합니다.
다음 구현WMS mutation 화면 API, 단일 screen-data API 필요 여부, warehouse scope별 selector 필터링은 후속 검토 대상입니다.
최근 검증2026-06-21 로컬 Docker compose에서 gateway 경유 app shell, WMS 화면 조회 API, app shell 감사 로그 저장, EventBridge routing CDK synth를 33개 전체 시스템 시나리오로 검증했습니다.
CDK 상태로컬 compose, gateway env, AWS CDK ECS/ECR/Cloud Map/security group, gateway task USER_BFF_SERVICE_URL, User BFF task WMS_SERVICE_URL/WMS_INTERNAL_AUTH_SECRET 주입, 수동 배포 workflow가 준비되었습니다.

API 공통 기준

Public endpointGET /health, GET /ready만 public입니다.
App surface외부 사용자 API는 /api/app/**를 사용합니다. gateway-service가 JWT 검증과 tenant mismatch 1차 차단 후 User BFF로 전달합니다.
HeadersAuthorization, Content-Type, Accept, X-Request-Id, X-Tenant-Id, X-User-Id, Idempotency-Key
Context propagationdownstream 호출에는 X-Request-Id, X-Tenant-Id, X-User-Id와 internal auth header를 함께 전파합니다.
Envelope응답은 success, requestId, timestamp, data 또는 error 구조를 사용합니다. meta 필드는 사용하지 않습니다.
Tenant status guardUser BFF는 tenant-service의 GET /internal/tenants/{tenantId}/status를 호출해 active tenant만 업무 API를 통과시킵니다.
AuthorizationJWT에는 role이나 permission을 넣지 않고, auth-iam-service의 permission summary/check로 화면 메뉴와 액션 표시를 판단합니다.
Data ownershipUser BFF는 직접 DB에 접근하지 않습니다. 권한은 auth-iam-service, tenant 상태는 tenant-service, WMS 업무 데이터는 wms-service로 위임합니다.
AuditUser BFF가 소유한 app shell 이벤트는 USER_BFF_APP_AUDIT_PUBLISHER_TYPE에 따라 eventbridge, internal-api, disabled 중 하나로 처리합니다. 감사 로그 장애는 사용자 응답을 실패시키지 않고 warning log로 남깁니다.

Downstream 계약

대상 사용 API 용도 상태
tenant-service GET /internal/tenants/{tenantId}/status 사용자 업무 API 처리 전 tenant 상태 확인 Ready
tenant-service GET /internal/tenants/{tenantId}/modules 활성 module 기반 메뉴 구성 Ready
auth-iam-service GET /api/auth/permissions/summary, POST /api/auth/permissions/check 사용자 화면 권한 summary와 route/action 권한 확인 Ready
wms-service GET /api/internal/wms/warehouses, GET /api/internal/wms/items, GET /api/internal/wms/inventory WMS 화면 조회 데이터 조합과 dashboard action visibility 계산 Ready
EventBridge 또는 audit-log-service PutEvents 또는 POST /api/internal/audit/logs app shell 감사 이벤트 best-effort 발행/기록 Ready

Endpoint 목록

Endpoint 인증 설명 위임 대상 상태
GET /health Public 프로세스 생존 상태를 확인합니다. - Implemented
GET /ready Public config, downstream URL, internal auth, app audit publisher 준비 상태를 확인합니다. auth-iam-service, tenant-service, EventBridge 또는 audit-log-service Implemented
GET /api/app/me Bearer + active tenant 현재 사용자, tenant, 활성 module, role/permission summary를 조합합니다. auth-iam-service, tenant-service Implemented
GET /api/app/navigation Bearer + active tenant tenant module과 permission summary를 기준으로 사용 가능한 메뉴를 반환합니다. auth-iam-service, tenant-service Implemented
GET /api/app/wms/warehouses Bearer + wms.warehouses.manage 창고 목록 화면과 selector용 WMS warehouse 데이터를 반환합니다. Query: page, size, code. wms-service Implemented
GET /api/app/wms/materials Bearer + wms.items.manage 자재 목록 화면과 selector용 WMS item 데이터를 frontend 명칭인 material로 매핑합니다. Query: page, size, sku. wms-service Implemented
GET /api/app/wms/inventory-summary Bearer + wms.inventory.read 현재고 목록과 현재 페이지 기준 수량 합계를 반환합니다. Query: page, size, warehouseId, locationId, itemId. wms-service Implemented
GET /api/app/wms/dashboard Bearer + wms.inventory.read 첫 화면용 재고 balance 수, 샘플링된 페이지 합계, 현재 사용자의 WMS action visibility를 반환합니다. wms-service, auth-iam-service Implemented

감사 로그 side effect

GET /api/app/me 성공 시 userBff.appContext.loaded, GET /api/app/navigation 성공 시 userBff.navigation.loaded 감사 이벤트를 발행합니다. EventBridge 모드는 USER_BFF_AUDIT_EVENTBRIDGE_BUS_NAME으로 지정한 bus에 PutEvents를 호출하고, internal-api 모드는 POST /api/internal/audit/logs를 호출합니다. 실패해도 원 응답은 성공으로 유지하고 warning log에 record_app_audit를 남깁니다.
