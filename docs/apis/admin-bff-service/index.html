admin-bff-service API 문서

admin-bff-service는 운영자/관리자 화면이 사용하는 외부 /api/admin/** surface를 소유합니다. tenant, 사용자, role, permission, module, audit 화면 데이터를 조합하되, tenant business rule과 권한의 최종 판단은 각각 tenant-service와 auth-iam-service에 위임합니다.

API 공통 기준

Public endpointGET /health, GET /ready만 public입니다.
Admin surface외부 관리자 API는 /api/admin/**를 사용합니다. gateway-service가 JWT 검증과 tenant mismatch 1차 차단 후 Admin BFF로 전달합니다.
HeadersAuthorization, Content-Type, Accept, X-Request-Id, X-Tenant-Id, Idempotency-Key
Context propagationdownstream 호출에는 X-Request-Id, X-Tenant-Id, X-User-Id와 internal auth header를 함께 전파합니다.
Envelope응답은 success, requestId, timestamp, data 또는 error 구조를 사용합니다. meta 필드는 사용하지 않습니다.
List response목록 응답은 data.items, data.page, data.size, data.total만 사용합니다. page는 1부터 시작하고 기본 size는 20입니다.
Authorization중요 관리자 액션은 auth-iam-service의 permission check를 호출합니다. JWT에는 role이나 permission을 넣지 않습니다.
Permission guardAdmin BFF controller는 @AdminPermission("{domain}.{resource}.{action}") decorator로 필요한 권한을 선언합니다. 화면 조합 API처럼 여러 권한이 필요한 route는 배열로 선언하고 모두 통과해야 합니다.
Tenant status guardAdmin BFF는 tenant-service의 GET /internal/tenants/{tenantId}/status를 호출해 active tenant만 관리자 API를 통과시킵니다.
Data ownershipAdmin BFF는 직접 DB에 접근하지 않습니다. tenant 관리는 tenant-service, 사용자/role/permission 관리는 auth-iam-service로 위임합니다.

StatusCode의미

검증 시나리오
