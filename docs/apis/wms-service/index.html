wms-service API 문서

wms-service는 외부에 직접 노출하지 않는 WMS 도메인 서비스입니다. 1차 API는 BFF 또는 내부 서비스가 호출하는 /api/internal/wms/** 계약으로 작성합니다.

API 공통 기준

Public endpointGET /health, GET /ready만 public입니다.
Internal endpoint/api/internal/wms/**은 HMAC 기반 internal service auth가 필요합니다.
Allowed calleradmin-bff-service, user-bff-service만 WMS 내부 API 호출자로 허용합니다. Postman과 테스트 스크립트는 로컬에서 admin-bff-service caller를 시뮬레이션합니다.
HeadersContent-Type, Accept, X-Request-Id, X-Tenant-Id, X-User-Id, X-Internal-Service-Id, X-Internal-Timestamp, X-Internal-Signature, mutation은 Idempotency-Key 권장
Envelope응답은 success, requestId, timestamp, data 또는 error 구조를 사용합니다. meta 필드는 사용하지 않습니다.
Tenant모든 WMS 업무 API는 X-Tenant-Id가 필요하고 tenant module 목록에 wms가 있어야 합니다.

표준 에러

TENANT_REQUIREDX-Tenant-Id가 없습니다.
WMS_MODULE_NOT_ENABLEDtenant module 목록에 wms가 없습니다.
WMS_PERMISSION_DENIEDauth-iam permission check 결과가 거부되었습니다.
WMS_RESOURCE_NOT_FOUNDtenant 범위에서 창고/로케이션/품목/재고를 찾을 수 없습니다.
WMS_INSUFFICIENT_STOCK출고 할당 또는 출하에 필요한 가용/할당 재고가 부족합니다.
WMS_PACKING_NOT_FOUNDtenant 범위에서 포장 작업을 찾을 수 없습니다.
WMS_PACKING_ALREADY_CONFIRMED이미 확정된 포장 작업을 다시 수정하거나 확정하려고 했습니다.
WMS_PACKAGE_QUANTITY_MISMATCH패키지별 포장 수량 합계가 allocation 수량과 일치하지 않습니다.
WMS_SHIPMENT_NOT_READY포장 확정 전 출하를 시도했습니다.
WMS_SHIPMENT_ALREADY_SHIPPED이미 출하 완료된 포장/출고 건을 다시 출하하려고 했습니다.
WMS_SNAPSHOT_DATE_REQUIRED스냅샷 조회 또는 생성 요청에 snapshotDate가 없습니다.
WMS_SNAPSHOT_NOT_READY요청한 일자의 스냅샷이 아직 생성되지 않았습니다.
WMS_SNAPSHOT_REBUILD_CONFLICT같은 tenant/date의 스냅샷 생성 또는 재생성이 이미 진행 중입니다.
WMS_LEDGER_NOT_FOUND보정 대상 correctedLedgerId 원장을 찾을 수 없습니다.
WMS_LEDGER_REFERENCE_MISMATCH보정 대상 원장의 warehouse/location/item 조합이 요청과 다릅니다.

일 단위 재고 스냅샷 구현 기준

상태Implemented Postman collection과 구현을 함께 갱신합니다.
조회 목적특정 business date의 warehouse/location/item별 보유 수량, 할당 수량, 가용 수량을 조회합니다.
정합성 기준조회 API는 현재고를 역산하지 않고 wms_inventory_daily_snapshots read model만 읽습니다. 해당 read model은 wms_inventory_ledger의 cutoff balance에서 생성합니다.
날짜 기준snapshotDate는 YYYY-MM-DD 형식입니다. tenant timezone 설정 전까지 business date 기본값은 Asia/Seoul입니다.
권한조회는 wms.inventory.read, 생성/재생성은 신설 권한 wms.inventory.snapshot.generate를 사용합니다.
응답 envelope목록 응답은 data.items, data.page, data.size, data.total만 사용하고 meta는 사용하지 않습니다.
생성 명령자동 일마감 배치가 기본이며, 운영자가 수동으로 생성/재생성할 때는 mode를 generate 또는 rebuild로 보냅니다. 완료 시 wms.inventory.snapshot.generated 이벤트를 저장합니다.
보정 추적과거일 보정은 effectiveDate, correctedLedgerId, correctionReason으로 추적하고, 스냅샷은 previousSnapshotId와 isCurrent로 보정 전/후 수량을 보관합니다.

포장/출하 구현 기준

목표 흐름outbound allocation -> packing work -> package registration -> packing confirm -> shipment confirm
호환성기존 allocationId 직접 출하 요청은 유지하고, 포장 기반 출하는 같은 endpoint의 packingId branch로 처리합니다.
새 이벤트wms.outbound.packed를 추가합니다. 출하 완료 이벤트는 기존 wms.outbound.shipped를 유지합니다.
검증 상태pnpm test:wms:local 시나리오에 포장 생성, 포장 확정, 포장 기반 출하, 수량 불일치, 중복 출하 차단을 포함합니다.

연계 테스트 기준

Auth/IAMWMS mutation 전에 /api/auth/permissions/check 호출 evidence를 남깁니다.
Tenant/internal/tenants/{tenantId}/modules 응답에 wms가 있는지 확인합니다.
Outboxmutation마다 outbox_events에 pending row가 생기고 relay 처리 후 published가 됩니다.
AuditSQS consumer가 WMS event를 audit log로 저장합니다.
Local verificationpnpm test:wms:local로 WMS → Outbox Relay → LocalStack SQS → Audit Log 흐름을 검증합니다.
