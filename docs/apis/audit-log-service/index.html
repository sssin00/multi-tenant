audit-log-service API 문서

audit-log-service는 append-only 감사 로그 저장과 관리자 조회를 담당합니다. 현재 NestJS 기본 실행 구조와 health/readiness는 준비됐고, 감사 로그 저장소와 내부 저장/조회 API는 계약 초안 단계입니다.

서비스 상태

상태기본 실행 구조 완료 NestJS bootstrap, config, health/readiness, request context, access log, security headers, internal auth guard가 있습니다.
소유 범위감사 로그 저장, 조회, 보존 정책, evidence 추적
주요 호출자auth-iam-service, wms-service, tenant-service, admin-bff-service
배포 상태현재 CDK 스택에는 포함되지 않습니다. 향후 ECR, ECS Fargate service, CloudWatch log group, 내부 SG, DB secret을 추가합니다.

Endpoint 초안

MethodPathCallerStatus설명
GET /health Public 구현 프로세스 생존 확인용 endpoint입니다.
GET /ready Public 구현 config와 internal auth 설정 기준으로 트래픽 수용 가능 여부를 확인합니다.
POST /api/internal/audit/logs 도메인 서비스, Auth/IAM TODO 업무 변경 성공 후 감사 record를 append-only 저장소에 기록합니다.
GET /api/internal/audit/logs admin-bff-service TODO 운영자 화면에서 tenant, actor, action, 기간 기준으로 감사 로그를 조회합니다.

저장 요청 Payload 초안

{
  "tenantId": "tenant_001",
  "requestId": "req_01HZABC123",
  "occurredAt": "2026-06-12T00:00:00.000Z",
  "actor": {
    "type": "user",
    "userId": "user_001"
  },
  "action": "auth.user.created",
  "resource": {
    "type": "user",
    "id": "user_002"
  },
  "result": "success",
  "reason": "관리자 사용자 생성",
  "changes": {
    "after": {
      "email": "new-user@example.com"
    }
  }
}

민감정보, token, password, Authorization header, SQL, stack trace는 payload에 포함하지 않습니다.

응답과 오류 기준

Envelope모든 응답은 success, requestId, timestamp를 포함합니다.
List목록 응답은 data.items, data.page, data.size, data.total만 사용합니다.
주요 오류AUDIT_LOG_VALIDATION_FAILED, AUDIT_LOG_INTERNAL_AUTH_REQUIRED, AUDIT_LOG_WRITE_FAILED, AUDIT_LOG_QUERY_FAILED
장애 정책저장 실패 시 caller가 재시도 가능한 결과를 받을 수 있도록 timeout, retry, dead-letter 기준을 구현 전에 확정합니다.
