인증/인가 규칙

인증은 사용자가 누구인지 확인하는 과정이고, 인가는 사용자가 어떤 작업을 할 수 있는지 판단하는 과정입니다. 초기 권한 모델은 RBAC로 시작하고, ABAC는 이후 확장 항목으로 둡니다.

Access TokenJWT를 사용합니다. 기본 만료 시간은 30분을 권장합니다.
Refresh Tokenopaque token을 사용하고 저장소에 보관합니다. 기본 만료 시간은 14일을 권장합니다.
Refresh Rotationrefresh token 사용 시 새 token을 발급하고 이전 token은 폐기합니다.
User Type사용자는 system_admin과 general_user로 구분합니다. general_user는 반드시 tenant 값을 가지며, tenant-scoped 사용자 API는 general_user만 생성하고 관리합니다.
System Adminsystem_admin 계정은 별도 시스템 관리 또는 bootstrap 경로에서만 다루며 tenant-scoped 사용자 API와 섞지 않습니다.
권한 모델초기에는 RBAC를 사용합니다. 사용자 → 역할 → 권한 구조를 따릅니다.
Role/Permission Source역할과 권한의 source of truth는 auth-iam-service입니다.

JWT Claim 규칙

JWT에는 인증에 필요한 최소 식별 정보만 담고, 역할과 상세 권한은 넣지 않습니다.

{
  "sub": "user_001",
  "tenantId": "tenant_001",
  "type": "access",
  "iat": 1717220000,
  "exp": 1717221800,
  "iss": "auth-iam-service",
  "aud": "multi-tenant-api"
}
JWT에 넣는 것

sub, tenantId, type, iat, exp, iss, aud

JWT에 넣지 않는 것

roles, permissions, warehouse permission 상세, 사용자 프로필 상세, 메뉴 정보, 테넌트 설정

Permission 조회 규칙

JWT에는 roles와 permissions를 넣지 않습니다. BFF와 도메인 서비스는 필요한 시점에 auth-iam-service에서 role/permission summary 또는 permission check 결과를 조회합니다.

GatewayJWT 서명/만료 검증, sub, tenantId 추출, tenant mismatch 기본 차단
BFF화면 구성에 필요한 role/permission summary를 조회하고 메뉴/버튼 표시 여부를 구성
Domain Service실제 업무 수행 전 필요한 permission과 scope를 검증
Cache성능이 필요해지면 Redis에 role/permission cache를 두고, 역할 또는 권한 변경 시 무효화
 
permission cache key:
permission:{tenantId}:{userId}

Permission Naming

권한 이름은 {domain}.{resource}.{action} 형식을 사용합니다.

wms.inventory.read
	wms.inventory.adjust
	wms.inventory.snapshot.generate
	wms.inbound.manage
	wms.outbound.allocate
	wms.outbound.pack
	wms.outbound.ship
	tenant.users.manage
tenant.settings.manage
admin.tenants.manage
