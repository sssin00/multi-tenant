Events & Outbox

이벤트와 Outbox 규칙

도메인 이벤트, 1차 이벤트 범위, outbox 발행과 소비 규칙을 정리합니다.

이벤트 규칙

이벤트는 이미 발생한 도메인 사실을 표현합니다. 명령이 아니며, 상태 변경이 성공한 뒤 발행 대상으로 기록합니다. 서비스 간 직접 호출보다 이벤트 기반 연동이 자연스러운 경우에는 Outbox 패턴을 우선 사용합니다.

이름 형식{domain}.{resource}.{pastAction} 형식을 사용합니다. 예: wms.outbound.shipped
표현 방식이벤트 이름은 과거형으로 작성합니다. 예: created, adjusted, confirmed, allocated, shipped
금지createOutboundOrder, shipOutboundOrder, sendErp처럼 명령형 이름을 사용하지 않습니다.
필수 필드eventId, eventType, schemaVersion, tenantId, requestId, occurredAt, source, data
선택 필드actor, correlationId, causationId, traceId
Payload리소스 ID, 상태, 수량, 참조 ID, 변경 요약처럼 소비에 필요한 최소 데이터만 담습니다.
금지 데이터민감정보, 토큰, 비밀번호, 전체 사용자 프로필, 전체 권한 목록, 대용량 첨부파일은 이벤트에 넣지 않습니다.
 
{
  "eventId": "evt_01HZABC123",
  "eventType": "wms.outbound.shipped",
  "schemaVersion": 1,
  "tenantId": "tenant_001",
  "requestId": "req_01HZABC123",
  "occurredAt": "2026-06-02T01:40:00.000Z",
  "source": "wms-service",
  "actor": {
    "userId": "user_001",
    "type": "user"
  },
  "data": {
    "outboundOrderId": "out_001",
    "warehouseId": "wh_001",
    "status": "shipped"
  }
}

1차 이벤트 적용 범위

WMS 1차 개발에서는 모든 이벤트를 만들지 않고, 업무 상태 변경이 명확하고 이후 연동 가능성이 높은 이벤트만 먼저 적용합니다. 실제 외부 발행보다 outbox_events 저장까지 먼저 구현해도 충분합니다.

1차 필수 이벤트

wms.inventory.adjusted, wms.inventory.snapshot.generated, wms.inbound.confirmed, wms.outbound.allocated, wms.outbound.packed, wms.outbound.shipped

1차 구현 범위

packages/events envelope 타입, WMS 이벤트 이름 상수, wms-service outbox 저장, Auth/IAM 변경 outbox 저장, outbox-relay-service 기본 재시도 구조

Auth/IAM 이벤트

auth.user.created, auth.user.updated, auth.user.statusChanged, auth.permission.created, auth.role.created, auth.role.updated, auth.role.permissionsReplaced, auth.userRole.assigned, auth.userRole.removed

나중에 추가

wms.item.created, wms.item.updated, wms.inventory.reserved, wms.inventory.released, wms.inbound.created, wms.outbound.created, wms.outbound.cancelled

1차 연동 대상

감사 로그, 알림, ERP 연동, 분석 집계는 이벤트를 받을 수 있게 설계하되, WMS 첫 구현에서는 직접 완성하지 않아도 됩니다.

Outbox 이벤트 발행 규칙

상태 변경과 이벤트 저장은 같은 DB 트랜잭션에서 처리하고, 실제 발행은 outbox-relay-service가 담당합니다.

업무 트랜잭션도메인 상태 변경, outbox_events 저장, DB commit 순서로 처리합니다.
Relayoutbox-relay-service가 미발행 이벤트를 조회하고 EventBridge 또는 SQS로 발행합니다. 감사 로그 저장 경로는 SQS를 기본 대상으로 사용합니다.
BFF best-effort 이벤트자체 DB/outbox가 없는 BFF 소유 화면 이벤트는 사용자 응답을 차단하지 않는 범위에서 EventBridge 직접 발행을 사용할 수 있습니다. 정합성이 필요한 도메인 상태 변경은 반드시 outbox transaction 규칙을 따릅니다.
Source DB CredentialRelay는 source별 전용 DB URL secret으로 outbox_events만 다룹니다. 서비스의 일반 application DATABASE_URL을 재사용하지 않고, 다른 업무 테이블을 수정하지 않습니다.
성공 처리발행 성공 시 published 상태와 publishedAt을 기록합니다.
실패 처리발행 실패 시 retryCount를 증가시키고 재시도합니다. 반복 실패는 outbox failed 상태로 분리합니다. SQS 소비 실패는 메시지를 삭제하지 않아 visibility timeout 이후 재시도되고, redrive 정책에 따라 DLQ로 이동합니다.
 
{
  "outboxId": "outbox_01HZABC123",
  "eventId": "evt_01HZABC123",
  "eventType": "wms.outbound.shipped",
  "tenantId": "tenant_001",
  "aggregateType": "outboundOrder",
  "aggregateId": "out_001",
  "payload": {},
  "status": "pending",
  "retryCount": 0,
  "createdAt": "2026-06-02T01:40:00.000Z",
  "publishedAt": null
}

이벤트 소비 규칙

멱등성소비자는 eventId 기준으로 중복 처리를 방지합니다.
재시도이벤트 처리는 실패 후 재시도 가능해야 합니다.
순서순서가 중요한 처리는 aggregateId 기준으로 설계합니다.
서비스 경계이벤트 소비자가 다른 서비스의 DB를 직접 수정하지 않습니다.
상세 조회이벤트 payload가 부족하면 소비자가 API 또는 projection을 통해 필요한 상세 정보를 조회합니다.

이벤트 버전과 AWS 라우팅

Schema VersionschemaVersion은 필수입니다.
버전 유지필드 추가는 같은 version을 유지할 수 있습니다.
버전 증가필드 삭제, 타입 변경, 의미 변경은 version 증가가 필요합니다.
EventBridge여러 서비스가 구독하는 도메인 이벤트 라우팅에 사용합니다.
SQS특정 작업자가 안정적으로 처리해야 하는 작업, 재시도, DLQ가 중요한 작업에 사용합니다. 감사 이벤트는 CDK 관리형 audit event queue를 기본 경로로 사용합니다.
Audit Consumeraudit-log-service는 AUDIT_EVENT_QUEUE_URL을 polling하고, 저장 성공 메시지만 삭제합니다. eventId unique key로 중복 저장을 방지합니다.
