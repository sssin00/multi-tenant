환경 변수와 Secret 규칙

환경 변수 naming, AWS secret 관리, WMS 1차 환경 변수를 정리합니다.

환경 변수 규칙

환경 변수는 서비스별 설정을 명확히 분리하고, 비밀값은 AWS Secrets Manager 또는 SSM Parameter Store로 관리합니다. 실제 AWS 환경에서는 ECS Task Definition을 통해 주입합니다.

이름 규칙UPPER_SNAKE_CASE를 사용합니다.
공통 값모든 서비스가 공유하는 값은 prefix 없이 사용합니다. 예: APP_ENV, LOG_LEVEL
서비스 전용 값서비스 전용 값은 prefix를 붙입니다. 예: WMS_DEFAULT_PAGE_SIZE, OUTBOX_BATCH_SIZE
URL 이름서비스 간 호출 URL은 {SERVICE_NAME}_SERVICE_URL 형식을 사용합니다.
환경 파일환경 변수 파일 이름은 .env.local, .env.dev, .env.staging, .env.prod만 사용합니다. .env.example 등 다른 이름은 만들지 않습니다.
Secret실제 secret 값은 git에 올리지 않고, 배포 환경에서는 AWS Secrets Manager 또는 SSM Parameter Store로 관리합니다.
검증서비스 시작 시 환경 변수 validation을 수행합니다. 운영 환경에서 필수 값이 없으면 시작 실패로 처리합니다. zod 같은 schema validation 도입은 권장/향후 기준입니다.
기본값기본값은 코드 곳곳에 숨기지 않고 config module에 모읍니다. schema validation을 도입한 서비스는 schema에도 기본값을 명시합니다.

AWS Secret 관리

일반 설정값과 비밀값을 분리합니다. ECS Task Definition에서는 일반 설정은 environment, 비밀값은 secrets로 주입합니다.

environment서비스 포트, log level, endpoint, feature flag, event bus name, queue URL처럼 노출되어도 치명적이지 않은 설정값
secretsDB password, JWT private key, ERP API key, SMTP password, access key처럼 비밀로 관리해야 하는 값
Secrets ManagerDB password, JWT private key, 외부 API key, SMTP password 같은 민감정보를 관리합니다.
SSM Parameter Store일반 설정값, endpoint, feature flag, public config를 관리합니다.
로그 금지secret 값은 시스템 로그, 감사 로그, 에러 응답에 노출하지 않습니다.
 
# 일반 설정

FILE_BUCKET_NAME=multi-tenant-dev-files

Outbox Relay Source DB Secret

outbox-relay-service는 여러 서비스 DB의 outbox_events를 읽으므로 source별 DB URL을 별도 secret으로 주입합니다. 이 변수들은 source credential이어서 OUTBOX_ prefix 예외로 허용합니다.

ECS secretsAUTH_IAM_OUTBOX_DATABASE_URL, TENANT_OUTBOX_DATABASE_URL, WMS_OUTBOX_DATABASE_URL
GitHub secret ARNAUTH_IAM_OUTBOX_DATABASE_URL_SECRET_ARN, TENANT_OUTBOX_DATABASE_URL_SECRET_ARN, WMS_OUTBOX_DATABASE_URL_SECRET_ARN
ECS environmentOUTBOX_SOURCES, OUTBOX_WORKER_ENABLED, OUTBOX_PUBLISHER_TYPE, OUTBOX_EVENTBRIDGE_BUS_NAME, OUTBOX_SQS_QUEUE_URL, polling/retry 설정
권한전용 DB 계정을 사용하고, 가능한 경우 outbox_events 조회와 relay 상태 필드 갱신 권한으로 제한합니다.

Audit Event Consumer Environment

audit-log-service는 감사 이벤트 저장을 위해 SQS polling consumer를 가질 수 있습니다. CDK 배포 기본값은 consumer enabled와 관리형 audit event queue URL 주입이며, 로컬 기본값은 disabled입니다.

ECS environmentAUDIT_EVENT_CONSUMER_ENABLED, AUDIT_EVENT_QUEUE_URL, AUDIT_EVENT_POLL_INTERVAL_MS, AUDIT_EVENT_WAIT_TIME_SECONDS, AUDIT_EVENT_VISIBILITY_TIMEOUT_SECONDS, AUDIT_EVENT_BATCH_SIZE
Queue URLAUDIT_EVENT_QUEUE_URL이 비어 있으면 CDK 관리형 audit event queue URL을 주입합니다. 외부 queue 사용 시에만 override합니다.
Local endpointLocalStack 등 로컬 SQS를 사용할 때만 AUDIT_EVENT_SQS_ENDPOINT를 설정합니다.

환경 변수 검증

NestJS 서비스는 시작 시 환경 변수를 검증합니다. 현재 서비스는 config module에서 직접 검증하며, 향후 공통 schema validation으로 정리할 때는 zod 사용을 권장합니다.

const envSchema = z.object({
  NODE_ENV: z.enum(['development', 'test', 'production']),
  APP_ENV: z.enum(['local', 'dev', 'staging', 'prod']),
  SERVICE_NAME: z.string().min(1),
  GATEWAY_PORT: z.coerce.number().int().positive().optional(),
  AUTH_PORT: z.coerce.number().int().positive().optional(),
  LOG_LEVEL: z.enum(['debug', 'info', 'warn', 'error']).default('info'),
});
