API & Error

API와 에러 코드 규칙

API URL, 요청/응답 envelope, error code, HTTP status 매핑을 정리합니다.

API 규칙

기본 구조/api/v1/{surface}/{domain}/{resource} 형식을 사용합니다.
버전 위치API 버전은 /api 바로 뒤에 둡니다. 예: /api/v1/app/wms/items
기본 원칙URL은 명사 중심, 리소스는 복수형, 단어는 kebab-case를 사용합니다.
좋은 예GET /api/v1/admin/tenants, POST /api/v1/app/wms/outbound-orders
피할 예POST /createOutboundOrder, POST /adjustInventory
버전을 올리는 경우

응답 필드 삭제, 필드 타입 변경, 필수 요청값 추가, URL 구조 변경, 인증 방식 변경처럼 기존 클라이언트가 깨지는 변경입니다.

버전을 올리지 않는 경우

응답 필드 추가, optional query 추가, 새 endpoint 추가, 내부 로직 개선, 성능 개선, 버그 수정입니다.

요청/응답 규칙

Body 형식요청과 응답 body는 JSON을 기본으로 사용합니다.
JSON Key요청 body, 응답 body, query parameter는 camelCase를 사용합니다.
날짜/시간날짜/시간은 ISO 8601 UTC 문자열을 사용합니다. 예: 2026-06-01T10:00:00.000Z
HeaderAuthorization, Content-Type, Accept, X-Request-Id, X-Tenant-Id를 표준으로 사용합니다.
Idempotency중복 실행 위험이 있는 POST는 Idempotency-Key header를 사용합니다.

성공 응답 형식

meta 필드는 사용하지 않고, 공통 응답 필드는 top-level에 둡니다.

{
  "success": true,
  "requestId": "req_01HZ...",
  "timestamp": "2026-06-01T10:00:00.000Z",
  "data": {}
}
단건 응답

data는 객체 또는 null입니다.

삭제 응답

삭제도 envelope를 유지하고 data는 null로 둡니다.

목록 응답 형식

복수 데이터의 data는 items, page, size, total만 사용합니다.

{
  "success": true,
  "requestId": "req_01HZ...",
  "timestamp": "2026-06-01T10:00:00.000Z",
  "data": {
    "items": [],
    "page": 1,
    "size": 20,
    "total": 152
  }
}
items현재 페이지의 데이터 배열입니다.
page현재 페이지 번호입니다. 1부터 시작합니다.
size요청한 페이지 크기입니다. 기본값은 20, 최대값은 100을 권장합니다.
total필터 조건에 맞는 전체 item 개수입니다. totalPages는 내려주지 않습니다.

에러 응답 형식

{
  "success": false,
  "requestId": "req_01HZ...",
  "timestamp": "2026-06-01T10:00:00.000Z",
  "error": {
    "code": "VALIDATION_FAILED",
    "message": "Validation failed",
    "details": {}
  }
}
Error CodeUPPER_SNAKE_CASE를 사용합니다. 예: TENANT_NOT_FOUND, VALIDATION_FAILED
Validation Errorerror.details.fields 배열에 필드별 오류를 담습니다.

에러 코드 체계

에러 코드는 클라이언트 분기와 운영 추적을 위한 안정적인 식별자입니다. 화면 문구는 message가 아니라 error.code를 기준으로 별도 처리합니다.

네이밍UPPER_SNAKE_CASE를 사용합니다.
기본 구조공통 에러는 짧게, 도메인 에러는 {DOMAIN}_{RESOURCE}_{REASON} 형태를 우선 사용합니다.
분기 기준프론트엔드와 BFF는 message가 아니라 error.code로 분기합니다.
Message기본 설명용입니다. 다국어 사용자 문구는 프론트엔드 또는 별도 message catalog에서 처리합니다.
Details검증 실패 필드, 업무 처리 힌트, 외부 시스템 참조값만 넣고 민감정보는 넣지 않습니다.
보안내부 예외 메시지, SQL, stack trace는 API 응답에 노출하지 않습니다.
로그모든 에러 로그에는 requestId, tenantId, userId, error.code를 남깁니다.

HTTP Status 매핑

400요청값 오류. 예: INVALID_REQUEST, VALIDATION_FAILED
401인증 실패. 예: UNAUTHORIZED, AUTH_INVALID_TOKEN, AUTH_TOKEN_EXPIRED
403권한 또는 테넌트 접근 불가. 예: FORBIDDEN, PERMISSION_DENIED, TENANT_MISMATCH, TENANT_SUSPENDED
404리소스 없음. 예: NOT_FOUND, TENANT_NOT_FOUND, WMS_ITEM_NOT_FOUND
409중복 또는 상태 충돌. 예: CONFLICT, DUPLICATE_RESOURCE, WMS_STOCK_ALREADY_ALLOCATED
422업무 규칙 위반. 예: BUSINESS_RULE_VIOLATION, WMS_INSUFFICIENT_STOCK
429요청 제한. 예: RATE_LIMIT_EXCEEDED
500서버 내부 오류. 예: INTERNAL_SERVER_ERROR
503서비스 준비 안 됨 또는 외부 장애. 예: SERVICE_UNAVAILABLE, TENANT_NOT_READY
