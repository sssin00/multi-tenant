멀티테넌트 처리 규칙

모든 업무 데이터와 업무 API는 테넌트 경계를 기준으로 격리합니다. 초기 구현은 JWT의 tenantId, 선택적 X-Tenant-Id, 테넌트 상태 검증, tenant context 전파부터 시작합니다.

기본 원칙모든 업무 API는 tenant context가 있어야 처리합니다.
예외 API로그인, token refresh, health check, public config, tenant bootstrap API는 예외로 둘 수 있습니다.
기준값인증된 요청의 기준 테넌트는 JWT의 tenantId입니다.
Header 검증X-Tenant-Id가 있으면 JWT의 tenantId와 반드시 일치해야 합니다.
MismatchJWT와 header의 tenant가 다르면 403 TENANT_MISMATCH로 거부합니다.
Tenant 상태active 상태만 업무 API를 허용합니다. suspended, deleted, provisioning은 거부합니다.
Gateway 책임gateway-service가 tenant context를 확정하고 내부 서비스로 전파합니다.
내부 서비스 책임전파된 tenant context를 사용하되, 중요한 업무 처리 전 필요한 권한과 scope를 다시 검증합니다.
데이터 접근테넌트 소유 데이터 조회와 변경은 반드시 tenantId 기준으로 제한합니다.
로그/이벤트/감사로그, 도메인 이벤트, 감사 로그에는 tenantId와 requestId를 포함합니다.
Redis/S3Redis key와 S3 object key에는 tenant 식별자를 포함합니다.
관리자 구분system_admin과 tenant_admin은 서로 다른 관리자 범위로 분리합니다.
대리 접속테넌트 대리 접속은 별도 승인, 감사 로그, 권한 제한 규칙을 만들기 전까지 제공하지 않습니다.
테스트tenant 누락, tenant mismatch, 타 테넌트 데이터 접근, 비활성 tenant 접근을 테스트합니다.

Tenant Context

서비스 내부에서는 아래 형태의 tenant context를 공통 패키지로 전달하는 것을 기본으로 합니다.

type TenantContext = {
  tenantId: string;
  tenantCode?: string;
  userId?: string;
  requestId: string;
  source: 'jwt' | 'header' | 'subdomain' | 'custom_domain' | 'internal';
  dbStrategy?: 'shared' | 'schema' | 'database';
  enabledModules?: string[];
};
내부 전파 Header

X-Request-Id, X-Tenant-Id, X-User-Id, X-Internal-Auth

Key 예시

tenant:{tenantId}:config, permission:{tenantId}:{userId}, tenants/{tenantId}/uploads/{yyyy}/{mm}/{fileId}
