보안과 AWS 리소스 규칙

보안 기본 원칙, AWS 리소스 이름/태그/IAM/CDK 관리 기준을 정리합니다.

보안 규칙

보안 규칙은 인증/인가, 멀티테넌트, 로그, 환경 변수, 파일 업로드 정책을 하나로 묶는 기준입니다. 기본값은 안전하게 두고, public 접근은 명시적으로 허용된 경우에만 사용합니다.

업무 API모든 업무 API는 인증이 필요합니다. Public API는 명시적으로 선언된 endpoint만 허용합니다.
JWTJWT에는 roles, permissions, 개인정보 상세를 넣지 않습니다.
권한 판단권한 판단은 auth-iam-service 또는 permission checker를 통해 수행합니다.
TenanttenantId mismatch는 항상 403 TENANT_MISMATCH로 차단합니다.
Internal Auth내부 서비스 호출은 외부 사용자 token만 믿지 않고 내부 호출 인증을 사용합니다.
CORS운영 환경에서는 허용 origin을 명시하고 wildcard를 사용하지 않습니다. gateway-service는 GATEWAY_CORS_ALLOWED_ORIGINS 기준으로 origin을 허용합니다.
Security Headergateway-service는 content sniffing, frame embedding, referrer leakage, 과도한 browser permission을 제한하는 공통 보안 header를 적용합니다.
HTTPS외부 진입점은 ALB HTTPS listener와 ACM 인증서를 사용합니다. HTTP listener는 HTTPS로 리다이렉트합니다.
Admin API관리자 API는 일반 사용자 API와 surface를 분리합니다. 예: /api/admin
Rate Limit요청 제한은 gateway-service에서 적용합니다.
파일 업로드업로드 파일은 확장자, MIME type, 크기를 검증합니다.
S3 접근S3 object는 public으로 열지 않고 pre-signed URL을 사용합니다.
Secretsecret은 AWS Secrets Manager 또는 SSM Parameter Store에서 관리합니다.
민감정보password, token, secret, cookie, Authorization header는 로그에 남기지 않습니다.
에러 응답API 응답에는 stack trace, SQL, 내부 예외 메시지를 노출하지 않습니다.
Audit중요한 변경 작업은 감사 로그를 남깁니다.
WMS 1차에서 충분한 보안 범위

JWT 검증, tenant mismatch 차단, permission check 구조, 민감정보 로그 금지, 에러 응답 내부정보 노출 금지, gateway rate limit 기본 구조, S3 pre-signed URL 기준을 먼저 적용합니다.

AWS 리소스 규칙

AWS 리소스는 CDK로 관리하고, 이름/태그/환경/IAM 권한을 일관되게 유지합니다. 수동 생성 리소스는 예외로 두되 반드시 문서에 기록합니다.

이름 형식{project}-{env}-{resource} 형식을 사용합니다.
환경dev, staging, prod 환경을 분리합니다.
ECS Service{project}-{env}-{service} 형식을 사용합니다. 예: multi-tenant-dev-wms-service
SQS Queue{project}-{env}-{purpose}-queue 형식을 사용합니다. 예: multi-tenant-dev-audit-events-queue
EventBridge{project}-{env}-event-bus 형식을 사용합니다. 예: multi-tenant-dev-event-bus
S3 Bucket전역 unique가 필요하므로 {project}-{env}-{purpose}-{accountOrRegion} 형식을 사용합니다.
CloudWatch Logs서비스별 log group으로 분리합니다. 예: /multi-tenant/dev/wms-service
IAM Role서비스별 role을 분리하고 least privilege 원칙을 따릅니다.
VPCVPC, subnet, security group은 CDK로 관리합니다.
Secretsecret 값은 Secrets Manager 또는 SSM Parameter Store에서 참조합니다.
ACMALB에 연결하는 ACM 인증서는 ALB와 같은 region에 둡니다. 현재 기본 region은 ap-northeast-2입니다.
DNSRoute53 hosted zone 준비 전에는 외부 DNS에서 gateway domain을 ALB DNS 이름으로 연결합니다. Route53 도입 후에는 CDK로 A/AAAA Alias record를 관리합니다.
Removal Policy삭제 위험이 큰 리소스는 prod에서 보수적인 removal policy를 사용합니다.

배포 Workflow 규칙

gateway-service, auth-iam-service, tenant-service, admin-bff-service, audit-log-service, outbox-relay-service 배포 workflow는 모두 같은 CDK 스택을 갱신합니다. 한 서비스 배포가 다른 서비스를 암묵적으로 중지하지 않도록 비대상 서비스 값은 명시적으로 관리합니다.

자동 배포GitHub Actions push 자동 배포는 당분간 비활성화하고 workflow_dispatch 수동 실행만 허용합니다.
환경 매핑자동 배포를 다시 켜면 dev는 dev, staging은 staging, main은 prod로 배포합니다.
인증GitHub Actions는 OIDC로 AWS role을 assume하며, 환경별 secret AWS_ROLE_ARN을 사용합니다.
비대상 서비스서비스별 workflow는 비대상 서비스 desired count와 image tag를 필수 입력으로 요구합니다. 값이 비어 있으면 실패해야 하며, 다른 서비스를 0 또는 latest로 조용히 덮어쓰지 않습니다.
Desired CountgatewayDesiredCount, authDesiredCount, tenantDesiredCount, adminBffDesiredCount, auditDesiredCount, outboxDesiredCount는 0 이상의 정수만 허용합니다.
Secretdesired count가 0보다 큰 서비스는 필요한 Secrets Manager 또는 SSM Parameter Store ARN이 함께 제공되어야 합니다. outbox-relay-service는 worker가 켜져 있으면 OUTBOX_SOURCES에 포함된 source별 DB URL secret ARN이 필요합니다. 감사 이벤트 SQS queue URL은 기본적으로 CDK 관리형 queue를 사용하므로 외부 queue를 사용할 때만 override합니다.
Image Taggateway-service image tag는 {env}-{gitSha12} 형식을 사용하며, 다른 서비스도 동일한 형식을 우선합니다.
