Naming & Structure

네이밍과 구조 규칙

파일명, 폴더 구조, 서비스 경계, 개발 중 확정할 규칙을 정리합니다.

네이밍 규칙 요약

폴더/파일kebab-case. 예: wms-service, tenant-context.middleware.ts
클래스PascalCase. 예: InventoryService, JwtAuthGuard
타입/인터페이스PascalCase. I 접두어는 쓰지 않습니다. 예: TenantContext, EventPublisher
함수/변수camelCase. 예: resolveTenantContext, tenantId
상수/환경 변수UPPER_SNAKE_CASE. 예: TENANT_HEADER, DATABASE_URL
API URL/api/v1/{surface}/{domain}/{resource}, kebab-case, 복수형 리소스. 예: /api/v1/app/wms/inventory-adjustments
이벤트dot.case, 과거형. 예: wms.inventory.changed, tenant.created
AWS 리소스{project}-{env}-{resource}. 예: multi-tenant-dev-wms-service
브랜치/커밋브랜치: feature/wms-inventory. 커밋: feat(wms): add inventory adjustment

개발 중 확정할 규칙

배포/운영 규칙, Git/브랜치/커밋 규칙, DB 규칙은 지금 세부 항목까지 확정하지 않습니다. 첫 기능 개발, AWS 배포 실습, WMS 데이터 모델 설계 과정에서 실제 작업 흐름에 맞춰 결정합니다.

배포/운영 규칙

배포/운영 규칙은 ECS 배포 파이프라인, health check, rollback, alarm을 실제 구성할 때 확정합니다. 현재는 AWS ECS Fargate를 사용하고 CloudWatch 기반 관측성을 둔다는 방향만 유지합니다.

Git/브랜치/커밋 규칙

현재는 브랜치 type/short-description, 커밋 Conventional Commits 정도만 유지합니다. 기능 개발이 시작되면 커밋 단위, PR 단위, 브랜치 보호 기준을 확정합니다.

DB 규칙

DB 규칙은 WMS 데이터 모델을 설계할 때 확정합니다. 멀티테넌트 저장 전략, migration, index, transaction, soft delete, 공통 컬럼은 실제 모델이 보일 때 결정합니다. 

# 현재 유지할 최소 Git 기준
branch: feature/wms-inventory
commit: feat(wms): add inventory adjustment
commit: docs(rules): update logging rules

NestJS

입출력/모델

*.dto.ts, *.entity.ts, *.event.ts

인프라 코드

*.guard.ts, *.middleware.ts, *.interceptor.ts, *.policy.ts

폴더/파일 구조 규칙

일반적인 monorepo 구조를 따릅니다. 실행 가능한 앱은 apps, 공유 코드는 packages, 인프라는 infra, 문서는 docs에 둡니다.

apps/*실행 가능한 백엔드 서비스와 프론트엔드 앱을 둡니다.
packages/*여러 앱이 공유하는 공통 코드만 둡니다. 특정 서비스의 업무 로직은 넣지 않습니다.
infra/cdkAWS CDK 인프라 코드를 둡니다.
docs/design아키텍처, 기술스택, 공통 규칙 같은 설계 문서를 둡니다.
docs/development서비스별 개발과정, 구현 기록, 테스트 기록을 둡니다.
서비스 내부 구조

NestJS 서비스는 src/modules/{domain} 아래에 controller, service, repository, dto, events, tests를 둡니다.

파일 배치 원칙

파일은 사용하는 곳 가까이에 둡니다. 여러 서비스에서 재사용될 때만 packages로 이동합니다. 

apps/wms-service/
  src/
    modules/
      inventory/
        inventory.controller.ts
        inventory.service.ts
        inventory.repository.ts
        dto/
        events/
        tests/

서비스 간 의존성 규칙

서비스 경계를 흐리지 않기 위해 다른 서비스의 내부 코드를 직접 import하지 않습니다. 서비스 간 통신은 API 또는 이벤트로만 합니다.

허용apps/wms-service가 packages/events, packages/logger 같은 공통 패키지를 import
허용서비스 간 통신을 REST/gRPC API client 또는 이벤트로 처리
금지apps/wms-service가 apps/auth-iam-service/src 내부 파일을 직접 import
금지공통 패키지에 특정 서비스의 업무 규칙을 넣는 것
Gateway

공통 출입구 역할만 담당합니다. 도메인 업무 로직과 DB 직접 접근을 넣지 않습니다.

BFF

화면 데이터 조합만 담당합니다. 재고 차감, 출고 할당 같은 업무 규칙을 넣지 않습니다.

Domain Service

각 도메인의 업무 규칙과 데이터 소유권을 갖습니다.
