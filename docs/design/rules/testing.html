테스트 시나리오 규칙

E2E 자동화 코드 대신 테스트 시나리오 기반 AI 검증 방식을 정리합니다.

테스트 규칙

이 프로젝트의 1차 테스트는 자동화 E2E 테스트 도구를 중심으로 하지 않습니다. 대신 테스트 시나리오 문서를 기준으로 AI가 화면, API, 로그, 데이터 결과를 확인하는 방식으로 진행합니다.

기본 방식테스트는 코드 기반 E2E보다 테스트 시나리오 기반 검증을 우선합니다.
AI 역할AI는 시나리오의 사전 조건, 실행 단계, 기대 결과를 읽고 실제 화면/API/로그/데이터를 확인합니다.
자동화 범위1차에서는 Playwright 같은 E2E 테스트 코드를 작성하지 않습니다. 필요해질 때 별도 결정합니다.
필수 검증멀티테넌트 격리, API 응답 포맷, 에러 코드, WMS 핵심 업무 흐름, outbox 저장, 감사 로그 저장을 검증합니다.
성능 기준성능 테스트는 성능 테스트 기준의 p95/p99, 오류율, outbox/audit 지연, 자원 사용률, 정합성 조건을 함께 만족해야 통과입니다.
기록테스트 결과는 시나리오별로 성공/실패, 발견 이슈, 재현 단계, 수정 여부를 남깁니다.

테스트 시나리오 형식

테스트 시나리오는 AI와 사람이 모두 읽기 쉬운 형태로 작성합니다.

scenarioId시나리오 고유 ID입니다. 예: WMS-INV-001
title검증하려는 업무 흐름 이름입니다.
priorityP0, P1, P2로 중요도를 표시합니다.
preconditions테넌트, 사용자, 권한, 재고, 주문 상태 같은 사전 조건입니다.
stepsAI가 실행할 화면 조작 또는 API 호출 순서입니다.
expectedResult성공 응답, 에러 코드, 화면 상태, 로그, outbox, 감사 로그의 기대 결과입니다.
evidence검증 후 남길 증거입니다. 예: 응답 JSON, 화면 캡처, 로그 requestId, DB 확인 결과
 
{
  "scenarioId": "WMS-OUT-001",
  "title": "출고 할당 성공 시 outbox 이벤트가 저장된다",
  "priority": "P0",
  "preconditions": [
    "tenant_a가 active 상태이다",
    "user_wms_manager_a가 wms.outbound.allocate 권한을 가진다",
    "item_a_01의 가용 재고가 100개이다"
  ],
  "steps": [
    "출고 주문 out_001을 생성한다",
    "출고 할당 API를 호출한다",
    "응답과 outbox 저장 결과를 확인한다"
  ],
  "expectedResult": [
    "API 응답 success가 true이다",
    "출고 상태가 allocated가 된다",
    "outbox에 wms.outbound.allocated 이벤트가 저장된다",
    "이벤트에 tenantId와 requestId가 포함된다"
  ],
  "evidence": [
    "responseJson",
    "requestId",
    "outboxEventId"
  ]
}

WMS 1차 테스트 시나리오

1차에서는 아래 시나리오 묶음만 충분히 검증합니다.

멀티테넌트 격리

tenant 누락 실패, JWT/header tenant mismatch, tenant A 사용자의 tenant B 데이터 접근 차단, suspended tenant 접근 차단

API 응답/에러 포맷

성공 envelope, 목록 envelope, 에러 envelope, VALIDATION_FAILED, TENANT_MISMATCH, WMS_INSUFFICIENT_STOCK

WMS 핵심 업무

재고 조정, 일 단위 스냅샷 생성/조회, 입고 확정, 출고 할당, 포장 확정, 출고 확정

Outbox/Audit

업무 상태 변경 후 outbox 이벤트 저장, 재고 조정/스냅샷/포장/출고 확정 audit log 저장, 로그의 requestId 연결

성능 테스트 기준

판정 기준평균 응답 시간이 아니라 p95, p99, 오류율, 5xx, timeout, outbox/audit 지연, 정합성 오류 여부로 판단합니다.
1차 기준Local Docker는 조회 API p95 700ms, BFF 조합 조회 p95 900ms, WMS mutation p95 1200ms를 시작 기준으로 삼습니다.
dev AWS 기준단순 조회 p95 500ms, BFF 조합 조회 p95 800ms, 스냅샷 조회와 WMS mutation p95 1000ms, 5xx 0건을 목표로 합니다.
정합성 우선재고, 스냅샷, outbox, audit, tenant 격리에 오류가 있으면 latency 기준을 만족해도 실패입니다.

AI 테스트 수행 규칙

실행 기준AI는 테스트 시나리오에 적힌 단계만 기준으로 검증하고, 임의로 성공 판정을 내리지 않습니다.
증거 기준성공 판정에는 응답 JSON, 화면 상태, 로그, outbox, audit log 중 필요한 증거를 남깁니다.
실패 처리실패 시 실제 결과, 기대 결과, 재현 단계, 관련 requestId를 기록합니다.
외부 연동AWS/ERP 실제 운영 리소스는 테스트에서 직접 호출하지 않고 mock, local 환경, dev 환경으로 제한합니다.
회귀 테스트수정 후 같은 시나리오를 다시 실행해 통과 여부를 확인합니다.
