Application Development Decision

AWS ECS 기반 멀티 테넌트 WMS 앱 개발 기술스택

모든 인프라는 AWS 서비스를 사용하고, 실행 환경은 ECS Fargate를 기준으로 합니다. 초기에는 WMS를 빠르게 완성하되, 이후 MES, QMS, FMS, EMS를 같은 패턴으로 확장할 수 있도록 애플리케이션 구조와 AWS 서비스 선택을 정리합니다.

최종 추천 스택

Backend NestJS + TypeScript를 기준으로 서비스와 BFF를 구현합니다.
Runtime 모든 API, worker, relay 서비스는 ECS Fargate에서 컨테이너로 실행합니다.
Database WMS 정합성을 위해 RDS PostgreSQL을 메인 데이터베이스로 사용합니다.
Event 초기에는 EventBridge + SQS, 확장 시 MSK Kafka를 검토합니다.
Frontend 운영툴 성격에 맞게 React + Vite를 우선 추천합니다.
Multi-Tenant 초기에는 Shared DB + tenant_id, VIP는 별도 DB로 확장 가능한 구조를 둡니다.

기술스택 상세

Language TypeScript. 백엔드와 프론트엔드에서 같은 언어를 사용해 타입 일관성과 개발 속도를 확보합니다.
Backend Framework NestJS. Module, Provider, Guard, Interceptor, Pipe 구조가 Gateway, BFF, Tenant, Auth/IAM, WMS 서비스에 잘 맞습니다.
Frontend React + Vite. WMS 같은 내부 운영툴은 SEO보다 빠른 화면 개발, 명확한 상태 관리, 단순한 배포가 중요합니다.
Monorepo pnpm workspace + Turborepo. 서비스, 공통 타입, 공통 유틸, UI 패키지를 한 저장소에서 관리합니다.
Database Amazon RDS PostgreSQL. 재고, 입고, 출고 같은 트랜잭션 정합성이 중요한 업무에 적합합니다.
ORM Prisma. 타입 안정성과 마이그레이션 관리가 좋습니다. 복잡한 리포트, 대량 처리, 잠금 쿼리는 raw SQL을 병행합니다.
Cache Amazon ElastiCache Redis. gateway-service rate limit, 인증 토큰, 세션성 데이터, 짧은 TTL 캐시, 분산락 후보로 사용합니다.
Event Outbox + EventBridge + SQS로 시작합니다. 이벤트 양과 실시간 스트리밍 요구가 커지면 Amazon MSK Kafka를 추가합니다.
API REST first. 내부 서비스 간 고성능 통신이 필요한 시점에 gRPC를 도입합니다.
Auth JWT + Refresh Token + RBAC/ABAC. 모든 권한 판단은 tenant, organization, warehouse scope를 포함합니다.
File Amazon S3 + pre-signed URL. 엑셀 업로드, 라벨, PDF, 첨부파일을 저장합니다.
Observability CloudWatch Logs, CloudWatch Metrics, AWS X-Ray 또는 OpenTelemetry Collector를 사용합니다.
CI/CD GitHub Actions + ECR + ECS service update. 초기에는 rolling deploy, 운영 안정화 후 blue/green 배포를 검토합니다.

AWS 서비스 매핑

네트워크/진입점

ALB와 ACM을 기본 HTTPS 진입점으로 사용합니다. Route53 hosted zone이 준비되기 전에는 외부 DNS에서 ALB DNS 이름으로 연결하고, Route53은 도메인 관리가 준비된 뒤 CDK 리소스로 추가합니다.

Route53 later ACM ALB HTTPS CloudFront optional

컴퓨팅/배포

API 서비스와 worker 서비스 모두 ECS Fargate로 실행하고, 컨테이너 이미지는 ECR에 저장합니다.

ECS Fargate ECR CloudWatch Logs

데이터

업무 데이터는 RDS PostgreSQL, 캐시는 ElastiCache Redis, 파일은 S3를 사용합니다.

RDS PostgreSQL ElastiCache Redis S3

이벤트/비동기

초기 WMS MVP는 EventBridge와 SQS로 운영 부담을 낮추고, 스트리밍 요구가 명확해질 때 MSK를 도입합니다.

EventBridge SQS DLQ MSK later

EventBridge/SQS 먼저 쓰는 이유

초기 장점

  • MSK Kafka보다 운영 부담과 비용이 낮습니다.
  • ECS worker, Lambda, 외부 AWS 서비스와 연결하기 쉽습니다.
  • 알림, 감사, ERP 연동처럼 업무 이벤트 처리에 충분합니다.
  • DLQ와 retry 정책을 AWS 관리형으로 구성하기 쉽습니다.

전환 기준

  • 컨슈머 그룹 기반의 대량 스트리밍 처리가 필요할 때
  • 이벤트 replay와 장기 보존이 핵심 요구가 될 때
  • MES, 설비, IoT 데이터가 초당 대량으로 유입될 때
  • ClickHouse, Flink, Kafka Connect 같은 파이프라인이 필요할 때
설계 원칙

애플리케이션 코드에는 EventPublisher 인터페이스를 두고, 초기 구현체는 EventBridge로 만듭니다. 나중에 MSK Kafka 구현체를 추가해도 도메인 코드는 크게 흔들리지 않습니다.

멀티테넌트 데이터 전략

일반 테넌트 Shared DB + tenant_id. 초기 개발과 운영이 가장 단순하며 WMS MVP에 적합합니다.
중대형 테넌트 Schema per tenant. 데이터 격리와 운영 유연성이 필요할 때 선택합니다.
VIP 테넌트 Database per tenant 또는 별도 RDS Cluster. 성능, 보안, 계약 조건이 강한 고객에게 적용합니다.
공통 원칙 모든 요청, DB 쿼리, 이벤트, 로그, 감사 기록에는 tenant_id를 반드시 포함합니다.

결정 순서

  1. Monorepo 구조를 pnpm workspace + Turborepo로 확정합니다.
  2. 서비스 기본 템플릿을 NestJS로 만듭니다.
  3. 로컬 개발용 Docker Compose에 PostgreSQL, Redis, LocalStack 또는 AWS mock을 구성합니다.
  4. ECR, ECS Cluster, ALB, RDS, Redis, S3, SQS, EventBridge 리소스 기준을 문서화합니다.
  5. Tenant Context, 인증, 권한, 공통 로깅 패키지를 먼저 구현합니다.
  6. WMS service를 모듈형 구조로 만들고 기준정보, 재고, 입고, 출고 순서로 개발합니다.
  7. Outbox Relay와 EventBridge/SQS 발행 구조를 붙입니다.
  8. Audit, Notification, ERP 연동을 이벤트 consumer로 추가합니다.
  9. 검색, 통계, MES/QMS/FMS/EMS는 WMS 운영 데이터가 쌓인 뒤 확장합니다.

초기에는 미룰 항목

MSK Kafka 즉시 도입 장기적으로 유용하지만 초기 WMS MVP에는 비용과 운영 복잡도가 큽니다.
gRPC 전면 도입 REST first로 시작하고, 내부 병목이 확인된 서비스부터 도입합니다.
Service Mesh 서비스 수와 운영 요구가 커진 뒤 App Mesh 등을 검토합니다.
OpenSearch/ClickHouse 초기에는 PostgreSQL read model로 충분한지 먼저 확인합니다.
VIP DB 자동 프로비저닝 처음에는 전략 필드만 두고, 실제 자동화는 고객 요구가 생긴 뒤 진행합니다.
Saga Orchestrator 초기 업무 흐름은 단일 WMS 트랜잭션과 Outbox 이벤트로 단순하게 유지합니다.
