Smart Factory Multi-Tenant MSA Roadmap

WMS부터 시작하는 멀티 테넌트 스마트 팩토리 구축 순서

전체 MES, WMS, FMS, QMS, EMS를 한 번에 구현하지 않고, 멀티테넌트 플랫폼 기반과 WMS 핵심 도메인을 먼저 완성한 뒤 감사 로그와 이벤트 기반으로 서비스를 확장하는 단계별 실행 문서입니다.

핵심 원칙

  • WMS를 첫 메인 서비스로 완성
  • 초기 WMS는 모듈형 모놀리스로 시작
  • 서비스별 DB와 Outbox 패턴 유지
  • Admin BFF 이후 WMS, User BFF 순서로 진행

전체 진행 방향

1. 플랫폼 뼈대

Gateway, Auth/IAM, Tenant, Admin BFF와 공통 규약은 1차 구현 또는 배포 준비 상태입니다.

2. 감사/이벤트 기반

WMS 전에 audit-log-service와 outbox-relay-service의 저장/발행 기준을 먼저 잡습니다.

3. WMS 핵심

기준정보 조회, 재고 조회, 재고 조정, 입고/출고 1차 흐름을 구현합니다.

4. 화면과 확장

User BFF, web-wms, 운영 화면을 붙인 뒤 ERP, 알림, MES/QMS/FMS/EMS로 확장합니다.

현재 기준 상태

완료/1차 구현

gateway-service, auth-iam-service, tenant-service, admin-bff-service는 1차 구현과 로컬 검증 기준이 잡혀 있습니다.

배포 준비

CDK 스택과 수동 GitHub Actions는 gateway, auth, tenant, admin-bff까지 반영되어 있습니다.

다음 후보

audit-log-service와 outbox-relay-service의 최소 저장/발행 구조를 먼저 잡습니다.

그 다음

wms-service 1차 API를 구현한 뒤 user-bff-service와 web-wms를 연결합니다.

단계별 작업 순서

0 Direction

기준 결정

완료

프로젝트 전체의 기술, 저장소, 테넌트 전략, 서비스 경계를 먼저 결정합니다.

기술 스택NestJS, TypeScript, PostgreSQL, Redis, Docker, AWS ECS Fargate 기준으로 진행합니다.
테넌트 전략초기 Shared DB + tenantId를 사용하고 schema/database 분리는 후속 결정으로 둡니다.
저장소 전략pnpm workspace 기반 monorepo와 apps/*, packages/*, infra/cdk 구조를 사용합니다.
초기 서비스 경계Gateway, Auth/IAM, Tenant, Admin BFF, Audit Log, Outbox Relay, WMS, User BFF 순서로 경계를 잡습니다.
1 Foundation

개발 및 운영 기반

진행 중

서비스를 계속 추가해도 같은 방식으로 만들고 배포할 수 있는 기본 골격을 만듭니다.

로컬 인프라Docker Compose로 PostgreSQL, Redis, LocalStack을 구성했습니다. Redis host 접속 포트는 6380입니다.
공통 규약API 응답, 에러 코드, 로깅, request id, 환경 변수 표준화
CI/CD서비스별 수동 GitHub Actions와 CDK 배포 흐름을 준비했습니다. 자동 push 배포는 비활성입니다.
관측성JSON access log와 readiness check는 적용했습니다. metrics/tracing/alarm은 운영 보강 단계에서 확장합니다.
2 Platform

멀티테넌트 플랫폼

1차 구현

WMS보다 먼저 테넌트와 권한의 기준을 세웁니다. 이 단계가 이후 모든 서비스의 기반입니다.

GatewayJWT 검증, tenant mismatch 차단, Auth/BFF proxy route, rate limit, security/CORS를 1차 구현했습니다. tenant status check는 BFF 책임입니다.
Auth/IAM Servicelogin/refresh/logout, user/role/permission, permission summary/check, outbox 저장을 1차 구현했습니다.
Tenant Servicetenant status/modules, admin tenant 목록/생성/상세/수정/상태/module/domain 내부 API를 구현했습니다.
Admin BFFtenant, user, role, permission, audit 조회 surface를 1차 구현했고 Admin 화면 연결을 기다립니다.
3 Audit/Event

감사 로그와 이벤트 기반

다음 구현 후보

WMS 업무 API를 만들기 전에 변경 이력과 outbox 발행 기반을 먼저 안정화합니다.

Audit Log ServicePOST /api/internal/audit/logs와 GET /api/internal/audit/logs를 구현합니다.
Audit 저장소append-only 저장, 보존 기간, 장애 처리, Admin BFF 조회 계약을 확정합니다.
Outbox RelayAuth/IAM과 tenant-service의 pending outbox event를 EventBridge 또는 SQS로 발행합니다.
Retry/DLQretry count, failed 상태, DLQ, 수동 replay 기준을 정합니다.
4 WMS MVP

WMS 핵심 도메인

착수 전

초기 WMS는 서비스 하나 안에서 모듈형 구조로 시작하고, 기준정보 조회와 재고 조회부터 만듭니다.

WMS 기본 구조health, ready, request context, tenant isolation, permission check, error envelope를 기존 서비스와 맞춥니다.
기준정보 조회warehouse, location, item 목록/상세 조회 API를 먼저 구현합니다.
재고 조회inventory 목록, 필터, summary API를 tenantId와 warehouse scope 기준으로 구현합니다.
업무 변경 APIinventory adjustment, inbound confirm, outbound allocate/ship을 audit/outbox 기준에 맞춰 확장합니다.
5 User Surface

User BFF와 WMS 화면

WMS 1차 API 이후

User BFF는 WMS 1차 조회 API가 준비된 뒤 화면 데이터를 조합하고, web-wms는 BFF를 통해서만 업무 데이터를 조회합니다.

User BFF/api/app/me, /api/app/navigation, WMS dashboard, warehouses, inventory summary를 조합합니다.
web-wms로그인 후 tenant/user/menu context와 WMS 첫 화면을 연결합니다.
PDA/현장 화면입고, 재고, 출고 작업 화면은 WMS 1차 API 안정화 후 분리합니다.
API 계약WMS API와 User BFF 응답이 달라지면 API 문서와 테스트 시나리오를 함께 갱신합니다.
6 Operations

운영 보강

후속

운영 JWT, 권한 cache, 알림, 파일, 배치 작업은 WMS 1차 흐름이 잡힌 뒤 보강합니다.

JWT 운영운영 RS256, key rotation, JWKS 또는 public key 배포 기준 확정
Permission Cacherole/permission 변경 시 Redis cache invalidation 기준 확정
Notification Service이메일, SMS, Slack, Kakao 등 이벤트 기반 알림
File Service엑셀 업로드, 라벨/PDF, 첨부파일, S3 호환 스토리지
Scheduler Service배치, 재시도, 통계 생성, 외부 연동 재처리
7 Integration

외부 연동

후속

ERP와 장비 연동은 장애와 재처리가 중요하므로, 동기 API와 비동기 이벤트를 분리합니다.

ERP Integration품목, 거래처, 입출고 지시 수신, 출고 결과 전송
장비 IngestionPDA, 스캐너, IoT, MQTT, PLC 연동 진입점
Retry/DLQ연동 실패 재시도, Dead Letter Queue, 수동 replay
연동 로그요청/응답 payload, 실패 원인, 처리 상태 추적
8 Insight

검색 및 통계

후속

초기에는 PostgreSQL read model로 시작하고, 필요가 커질 때 전용 저장소를 도입합니다.

Search품목, 재고, 입출고 이력 검색이 커지면 OpenSearch 도입
Analytics재고 회전율, 출고 리드타임, 작업자 생산성은 ClickHouse 검토
Read Model대시보드와 조회용 테이블을 이벤트 기반으로 갱신
KPI창고별, 테넌트별, 기간별 지표 정의
9 Expansion

MES, QMS, FMS, EMS 확장

장기 확장

WMS가 안정화된 뒤 제조 실행, 품질, 설비, 에너지 순서로 서비스를 추가합니다.

MES생산 지시, 생산 실적, 자재 투입, WMS 재고 차감 연동
QMS입고 검수, 공정 품질, 출하 검사와 WMS 상태 연결
FMS설비 상태, 작업 지시, 유지보수 이력 관리
EMS에너지 모니터링, 설비 데이터 기반 최적화

추천 개발 순서

  1. 완료: Monorepo, 공통 규칙, 로컬 인프라, Docker Compose 기본 구성
  2. 완료: gateway-service 1차 구현과 proxy, rate limit. tenant status check는 BFF 책임으로 이관
  3. 완료: auth-iam-service 1차 구현과 user/role/permission, outbox 저장
  4. 완료: tenant-service 1차 구현과 tenant status/module/domain/admin 내부 API
  5. 완료: admin-bff-service 1차 구현과 tenant/user/role/permission/audit surface
  6. 다음: audit-log-service 저장소와 내부 저장/조회 API 구현
  7. 다음: outbox-relay-service polling, publish, retry, DLQ 기준 구현
  8. 다음: DB 규칙과 WMS 도메인 모델 확정
  9. 다음: wms-service 기본 구조와 기준정보 조회 API 구현
  10. 다음: WMS 재고 조회와 inventory summary API 구현
  11. 이후: WMS 재고 조정, 입고 확정, 출고 할당/출고 확정 구현
  12. 이후: user-bff-service 1차 구현
  13. 이후: web-wms 화면 연결
  14. 이후: 운영 JWT, permission cache, alarm, rollback 보강
  15. 이후: Notification, File, Scheduler, ERP 연동 구현
  16. 장기: 검색/통계 확장과 MES, QMS, FMS, EMS 순차 추가

초기 MVP 권장 아키텍처

ALB → API Gateway → Admin BFF → Tenant Auth/IAM Audit Outbox Relay WMS User BFF → WMS DB → Outbox → EventBridge/SQS → Audit / Notification / ERP
실행 기준

처음부터 모든 서비스를 MSA로 잘게 나누기보다, WMS를 도메인 모듈이 선명한 하나의 서비스로 완성한 뒤 트래픽, 조직 경계, 데이터 소유권이 명확해지는 부분부터 분리하는 것이 현실적입니다.
