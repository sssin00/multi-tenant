Smart Factory Multi-Tenant MSA Roadmap

WMS부터 시작하는 멀티 테넌트 스마트 팩토리 구축 순서

전체 MES, WMS, FMS, QMS, EMS를 한 번에 구현하지 않고, 멀티테넌트 플랫폼 기반과 WMS 핵심 도메인을 먼저 완성한 뒤 이벤트 기반으로 서비스를 확장하는 단계별 실행 문서입니다.

핵심 원칙

  • WMS를 첫 메인 서비스로 완성
  • 초기 WMS는 모듈형 모놀리스로 시작
  • 서비스별 DB와 Outbox 패턴 유지
  • Tenant Context를 모든 계층에 전파

전체 진행 방향

1. 플랫폼 뼈대

Gateway, BFF, Tenant, Auth/IAM, 공통 규약을 먼저 잡습니다.

2. WMS 핵심

기준정보, 재고, 입고, 출고, 작업 지시를 우선 구현합니다.

3. 이벤트 확장

Outbox, Kafka, Audit, Notification, ERP 연동을 붙입니다.

4. 제조 서비스 추가

WMS 안정화 후 MES, QMS, FMS, EMS 순서로 확장합니다.

단계별 작업 순서

0 Direction

기준 결정

프로젝트 전체의 기술, 저장소, 테넌트 전략, 서비스 경계를 먼저 결정합니다.

기술 스택NestJS 또는 Spring Boot, PostgreSQL, Redis, Kafka, Docker 기준 확정
테넌트 전략초기 Shared DB + tenant_id, VIP는 Database per tenant 확장 가능하게 설계
저장소 전략Monorepo 또는 Multi-repo 선택, 공통 패키지 관리 방식 결정
초기 서비스 경계Gateway, Admin BFF, User BFF, Auth/IAM, Tenant, WMS 정의
1 Foundation

개발 및 운영 기반

서비스를 계속 추가해도 같은 방식으로 만들고 배포할 수 있는 기본 골격을 만듭니다.

로컬 인프라Docker Compose로 PostgreSQL, Redis, Kafka, Kafka UI 구성
공통 규약API 응답, 에러 코드, 로깅, request id, 환경 변수 표준화
CI/CD빌드, 테스트, 린트, 이미지 생성 파이프라인 준비
관측성로그, 메트릭, 트레이싱에 tenant_id, user_id, request_id 포함
2 Platform

멀티테넌트 플랫폼

WMS보다 먼저 테넌트와 권한의 기준을 세웁니다. 이 단계가 이후 모든 서비스의 기반입니다.

Tenant Service테넌트 생성, 상태, 사용 모듈, 도메인, DB 라우팅 정책 관리
Auth/IAM Service로그인, 사용자, 역할, 권한, 조직, 창고 단위 접근 제어
Gateway인증 검증, tenant 식별, rate limit, 내부 라우팅
Tenant ContextHTTP/gRPC header, Kafka header, DB query, 로그에 tenant context 강제
3 WMS MVP

WMS 핵심 도메인

초기 WMS는 서비스 하나 안에서 모듈형 구조로 시작하고, 경계가 명확해진 뒤 분리합니다.

기준정보창고, 로케이션, 품목, 거래처, 단위, 바코드
재고재고 생성, 조회, 이동, 조정, 로트/시리얼 정책
입고입고 예정, 검수, 적치, 입고 완료
출고출고 요청, 재고 할당, 피킹, 패킹, 출고 완료
작업 지시입고 작업, 이동 작업, 피킹 작업, 작업 상태 관리
WMS DB모든 업무 테이블에 tenant_id와 감사 필드 적용
4 API Layer

관리자 및 사용자 BFF

BFF는 화면에 필요한 데이터를 조합하고, 업무 규칙은 WMS와 각 도메인 서비스에 둡니다.

Admin BFF테넌트 관리, 사용자/권한 관리, 모듈 설정, 감사 로그 조회
User BFFWMS 대시보드, 입고, 재고, 출고, 작업 화면 API
모바일 APIPDA, 바코드 스캐너, 현장 작업 화면을 고려한 API 분리
API 계약OpenAPI 또는 Proto 기준으로 요청/응답 계약 관리
5 Event

이벤트 기반 구조

업무 트랜잭션은 DB에 먼저 확정하고, Outbox Relay가 Kafka로 안정적으로 발행합니다.

Outbox TableWMS DB 내부에 이벤트 저장 테이블 추가
Outbox Relay미발행 이벤트를 Kafka로 전송하고 재시도 처리
Topic 설계wms.inventory.adjusted, wms.inbound.confirmed, wms.outbound.allocated, wms.outbound.shipped 등 업무 이벤트 정의
Schema Version이벤트 payload 버전과 하위 호환 정책 정의
6 Operations

운영 필수 서비스

운영 추적, 알림, 파일, 배치 작업은 WMS 운영 안정성에 직접 영향을 줍니다.

Audit Log Service누가, 언제, 어느 테넌트에서, 어떤 데이터를 변경했는지 기록
Notification Service이메일, SMS, Slack, Kakao 등 이벤트 기반 알림
File Service엑셀 업로드, 라벨/PDF, 첨부파일, S3 호환 스토리지
Scheduler Service배치, 재시도, 통계 생성, 외부 연동 재처리
7 Integration

외부 연동

ERP와 장비 연동은 장애와 재처리가 중요하므로, 동기 API와 비동기 이벤트를 분리합니다.

ERP Integration품목, 거래처, 입출고 지시 수신, 출고 결과 전송
장비 IngestionPDA, 스캐너, IoT, MQTT, PLC 연동 진입점
Retry/DLQ연동 실패 재시도, Dead Letter Queue, 수동 replay
연동 로그요청/응답 payload, 실패 원인, 처리 상태 추적
8 Insight

검색 및 통계

초기에는 PostgreSQL read model로 시작하고, 필요가 커질 때 전용 저장소를 도입합니다.

Search품목, 재고, 입출고 이력 검색이 커지면 OpenSearch 도입
Analytics재고 회전율, 출고 리드타임, 작업자 생산성은 ClickHouse 검토
Read Model대시보드와 조회용 테이블을 이벤트 기반으로 갱신
KPI창고별, 테넌트별, 기간별 지표 정의
9 Expansion

MES, QMS, FMS, EMS 확장

WMS가 안정화된 뒤 제조 실행, 품질, 설비, 에너지 순서로 서비스를 추가합니다.

MES생산 지시, 생산 실적, 자재 투입, WMS 재고 차감 연동
QMS입고 검수, 공정 품질, 출하 검사와 WMS 상태 연결
FMS설비 상태, 작업 지시, 유지보수 이력 관리
EMS에너지 모니터링, 설비 데이터 기반 최적화

초기 MVP 권장 아키텍처

ALB → API Gateway → Admin/User BFF → Tenant Auth/IAM WMS → WMS DB → Outbox → Kafka → Audit / Notification / ERP
실행 기준

처음부터 모든 서비스를 MSA로 잘게 나누기보다, WMS를 도메인 모듈이 선명한 하나의 서비스로 완성한 뒤 트래픽, 조직 경계, 데이터 소유권이 명확해지는 부분부터 분리하는 것이 현실적입니다.
