Project Context

프로젝트 구조와 작업 맥락 정리

지금까지 결정한 멀티테넌트 스마트 팩토리/WMS first 방향, AWS ECS 운영 방식, pnpm workspace 기반 monorepo 구조, 다른 컴퓨터에서 이어서 작업하는 방법을 한 문서에 정리합니다.

프로젝트 방향

목표

멀티테넌트 기반 스마트 팩토리 서비스를 만들고, 첫 메인 서비스는 WMS로 시작합니다.

확장 순서

WMS 안정화 후 MES, QMS, FMS, EMS를 같은 패턴으로 순차 추가합니다.

구현 전략

초기 WMS는 모듈형 모놀리스로 시작하고, 경계가 명확해진 뒤 서비스 분리를 검토합니다.

확정 기술스택

Backend NestJS + TypeScript
Frontend React + Vite
Runtime AWS ECS Fargate
Database Amazon RDS PostgreSQL
Cache Amazon ElastiCache Redis
Event Outbox + EventBridge + SQS first, MSK Kafka later
Infra as Code AWS CDK TypeScript
Monorepo pnpm workspace + Turborepo

전체 AWS ECS 기반 아키텍처는 draw.io 파일로 관리합니다. 처음 공유한 스마트 팩토리 구성도를 참고하되, 현재 프로젝트의 결정사항에 맞춰 WMS first, ECS Fargate, EventBridge/SQS, RDS PostgreSQL 중심으로 다시 정리했습니다.

파일 위치 docs/design/smart-factory-aws-ecs.drawio

설계도에서 표현한 것

VPC, Public Subnets, Private App Subnets, Managed Services, Private Data Subnets, 공통/확장 Consumer 서비스, VIP 데이터 분리, 제조 도메인 확장 영역을 표현했습니다.

설계도에서 줄인 것

모든 서비스 간 세부 호출선을 그리면 가독성이 떨어지므로, 핵심 트래픽 흐름만 선으로 표시하고 상세 호출 관계는 서비스별 문서로 분리하기로 했습니다.

draw.io 수정 기준

요소 위로 선이 지나가지 않도록 유지하고, 선은 핵심 흐름에만 사용합니다. 상세 의존성, DB 접근, AWS 권한, 이벤트 구독 관계는 별도 문서나 서비스별 README에 정리합니다.

폴더 역할

apps

실제로 실행되는 백엔드 서비스와 프론트엔드 앱입니다. 서비스 간 내부 구현을 직접 import하지 않고 API 또는 이벤트로 통신합니다.

packages

공통 타입, 설정, 로거, 테넌트 컨텍스트, 이벤트, AWS 클라이언트, UI 컴포넌트를 둡니다.

infra/cdk

현재는 VPC, ALB, ECS/ECR, Cloud Map, Redis, 감사 SQS/DLQ, IAM, Security Group을 CDK 코드로 관리합니다. RDS/Aurora, S3, EventBridge bus, Route53은 후속 또는 외부 리소스입니다.

docker

로컬 개발용 PostgreSQL, Redis, LocalStack과 서비스별 Dockerfile을 관리합니다.

AWS 작업 방식

  1. AWS 리소스는 콘솔 수동 생성보다 infra/cdk의 CDK 코드로 관리합니다.
  2. 애플리케이션은 ECS Fargate에서 실행하고 이미지는 ECR에 배포합니다.
  3. 서비스 코드는 필요한 경우에만 packages/aws를 통해 S3, SQS, EventBridge, Secrets Manager를 사용합니다.
  4. AWS 리소스 이름, Queue URL, Bucket 이름, Event Bus 이름은 환경 변수로 주입합니다.
  5. ECS Task Role에 권한을 주고, 앱 코드에 AWS access key를 직접 저장하지 않습니다.
중요 원칙

infra/cdk는 AWS 리소스를 만드는 코드이고, apps/*는 AWS 위에서 실행되는 앱입니다. 앱 안에 인프라 생성 로직을 섞지 않습니다.

CDK와 Terraform 선택 기준

AWS CDK TypeScript 같은 일반 프로그래밍 언어로 AWS 인프라를 정의하고 CloudFormation으로 배포합니다.
Terraform HCL로 여러 provider의 인프라를 선언적으로 관리하고, Terraform state를 통해 리소스 상태를 추적합니다.
현재 추천 AWS only, ECS 중심, TypeScript monorepo 구조이므로 초기에는 AWS CDK를 추천합니다.
Terraform 검토 시점 멀티 클라우드, 인프라팀 표준, Terraform module 생태계, Terraform Cloud/OpenTofu 운영 체계가 필요해질 때 검토합니다.
결론

현재 프로젝트는 infra/cdk를 기준으로 진행합니다. 나중에 Terraform이 필요해지면 infra/terraform을 별도로 추가해 비교 검토합니다.

코드 작성 진행 순서

  1. gateway-service는 health/readiness, request context, JWT guard, proxy routing, Redis rate limit까지 1차 구현을 완료했습니다. tenant status check는 BFF로 이관했습니다.
  2. auth-iam-service는 로그인, token refresh, 사용자, 역할, 권한, internal permission summary/check API를 1차 구현했습니다.
  3. tenant-service는 health/readiness, tenant status/modules 조회, tenant 생성과 code 자동 생성, outbox 저장, BFF tenant status check 연결을 1차 구현했습니다.
  4. admin-bff-service는 tenant, 사용자, role, permission, module, audit 조회 API와 CDK 배포 준비까지 1차 구현했습니다.
  5. audit-log-service는 append-only 저장소, 내부 기록/조회 API, SQS 이벤트 consumer, Admin/User BFF 연동을 1차 구현했습니다.
  6. outbox-relay-service는 auth-iam, tenant, wms source polling과 mock/EventBridge/SQS publisher, retry/failed 상태 전환을 구현했습니다.
  7. wms-service는 기준정보, 재고 원장/스냅샷, 재고 조정, 입고 확정, 출고 할당, 포장, 출하 내부 API와 outbox event 저장을 구현했습니다.
  8. user-bff-service는 app shell과 navigation 기본 API, tenant/permission guard, audit-log best-effort 기록, gateway /api/app/** route를 구현했습니다.
  9. 다음 단계는 user-bff-service WMS 화면 조합 API와 web-wms 첫 화면 계약을 만드는 것입니다.

BFF 제작 단계

BFF는 domain service의 업무 규칙을 소유하지 않고 화면 데이터 조합과 tenant readiness를 담당합니다. Admin BFF와 User BFF는 모두 gateway 뒤에 있으며, domain/platform service 직접 노출을 막는 경계 역할을 합니다.

Admin BFF admin-bff-service. 외부 /api/admin/** surface를 소유하고 tenant-service, auth-iam-service, audit-log-service, 필요한 WMS 내부 API로 운영 화면 데이터를 조합합니다.
User BFF user-bff-service. /api/app/me, /api/app/navigation 기본 API는 구현됐고, 다음으로 WMS dashboard/warehouse/material/inventory-summary 데이터를 조합합니다.
BFF 역할 화면에 필요한 데이터 조합, 여러 서비스 호출 aggregation, 프론트용 응답 형태 변환, 메뉴/버튼 상태 구성입니다.
BFF에 넣지 않을 것 재고 차감, 입고 완료, 출고 할당 정책, 권한의 최종 판단, DB 직접 접근 같은 도메인 업무 로직입니다.

gateway-service 기능 범위

gateway-service는 ALB 뒤에서 외부 요청을 받아 Auth/IAM과 BFF로 넘기는 공통 출입구입니다. 이 서비스는 업무 로직을 처리하지 않고, 공통 보안, 테넌트 식별, 요청 컨텍스트, 라우팅을 담당합니다.

구현된 공통 기능

health check, request id 발급/전파, tenant id 추출, access log, JWT guard 기본 구조, public/private route 구분, Auth/BFF proxy routing, Redis rate limit, CORS 정책, 보안 헤더, 요청 크기 제한, 공통 에러 포맷을 구현했습니다.

후속 확장 기능

관측성 연동, 운영 trace 전파, route별 세부 정책, subdomain/custom domain 기반 tenant 식별을 추가합니다.

라우팅 /api/auth/** → auth-iam-service, /api/admin/** → admin-bff-service, /api/app/** → user-bff-service로 전달합니다. gateway-service는 tenant-service, wms-service, audit-log-service를 직접 호출하지 않습니다.
Tenant 식별 현재는 X-Tenant-Id header와 JWT tenantId claim을 사용하고, 이후 subdomain/custom domain을 추가합니다.
Request Context request_id, tenant_id, user_id를 내부 요청과 로그에 전파합니다. JWT에는 role/permission을 넣지 않고 BFF와 domain service가 Auth/IAM permission API를 호출합니다.
넣지 않을 기능 재고 차감, 입고 완료, 품목 생성 규칙, ERP 연동, 복잡한 권한 정책 계산, DB 직접 접근은 Gateway에 넣지 않습니다.
